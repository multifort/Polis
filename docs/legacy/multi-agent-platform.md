# 多智能体平台 — 架构设计基线

> 本文是设计讨论的沉淀，**只描述架构（是什么/为什么/职责/数据/流程/待定/风险），
> 不涉及实现节奏与 MVP 拆解**（按约定，整体理清前不谈落地）。
> 这是一份"活文档"，随讨论推进更新。最后更新：2026-06-09。

---

## 0. 目标与背景

在现有"宁翼智能（chat + agent 双模式，agent 由 hermes CLI 驱动）"之上，演进为一个
**可定义角色 / 可编排协同 / 可自动规划** 的多智能体平台。用户通过它把系统性任务
（如"做一份竞品调研报告"）交给多个定制 AI 角色协作完成。

设想的三层（用户原话归纳）：

- **第一层 角色定义**：onboarding 问答 → 生成一个 AI 角色 = 系统提示词（能力）+ 技能（skills）。
- **第二层 多角色协同**：多个角色围绕同一任务协作，彼此可共享数据；通信与调度由协同层负责，角色只执行。
- **第三层 任务规划**：用大模型把系统性任务拆解（任务-规范-计划等维度），交给第二层调度第一层执行。

执行链路自上而下：**规划层拆任务 → 编排层分派 → 角色层执行**。

## 1. 三条核心设计原则

这三条贯穿全局，是后面所有组件设计的根。

### 原则一：角色（配置） 与 执行器（运行时） 解耦

- **角色 = 一份廉价的配置数据** `{系统提示词 + 技能集 + 工作目录 + 可用能力}`，可以无限多。
- **需要容器化、需要扩容的是执行器（hermes 运行时）**，做成一个 **Worker 池**（横向多副本）。
- **并发度 = Worker 副本数**，绕开 hermes 单进程不能并发的限制（不需要它并发，多开即可）。
- 推论：**不是"每个角色一个常驻容器"**（那样空闲角色白占资源），而是"任意角色配置丢给任意空闲 Worker 执行"。

### 原则二：Worker 无状态，状态全部外部化、运行时注入

- 每次角色执行 = 启动 → 读外部状态 → 干活 → 写回 → 进程销毁；可随时杀/重试/换 Pod。
- **运行时注入给无状态 Worker 的三样东西**（完全对称）：
  1. **上下文切片**（这次该知道的全局上下文）
  2. **启用的 skills**（这个角色声明要用的技能）
  3. **模型配置 + 凭证**（按能力解析出的模型 + 用户的临时密钥）
- 好处：重试、并行、换 Worker 都不需要"会话黏在某个 Pod 上"，可靠性高。

### 原则三：目标每步在场（目标一致性）

- 任务的"目标 + 验收标准"不可变，**注入到每一个角色的上下文切片里** = "北极星"。
- 每步产出在并入全局上下文前，可由门禁（校验/审核角色）对照验收标准放行。

> hermes 在此体系中的定位：**hermes 就是执行器运行时**；"角色"是喂给 hermes 的参数
> `{system prompt, skill 子集, cwd}`。不替换 hermes，也不重写 agent loop。

## 2. 整体分层与组件

```
控制面(宁翼 server 升级)：角色注册表 | 模板库 | 规划器 | 人审 UI
        │
编排层：Temporal（DAG 执行、重试、人审暂停、运行中 signal 干预、缺失 skill 子工作流）
        │
   ┌────┴───────────────────────────┐
Context Manager(上下文管家)        Skill 库
 持有全局上下文/切片/收信封/出处     注册·版本·信任·检索 + 自动生成 + 验证沙箱
        │                               │
执行层：hermes Worker 池(要工具) | 纯 LLM(纯推理) | 媒体生成器(图/视频)
        │
Model Router(能力路由)：系统模型目录 + 每用户凭证保险库 → 按需注入 → 各家模型后端
        │
数据面：每任务一个共享卷(任意模态产物) + DB(产物描述符台账, 结构化 + JSONB)
```

### 组件职责


| 组件                  | 职责                                              |
| ------------------- | ----------------------------------------------- |
| **角色注册表**           | 存角色配置（提示词+技能+可用能力+执行器档位）                        |
| **模板库**             | 预先写好的 Plan（DAG 模板），版本化管理                        |
| **规划器**             | 目标 → Plan(DAG)；混合式：检索模板→LLM 填充/微调，否则 LLM 全自动拆解  |
| **人审 UI**           | 计划确认、审核断点、运行中干预、成本可见                            |
| **编排层(Temporal)**   | 按 DAG 调度、并行/串行、重试、人审暂停(signal)、缺失处理子流程          |
| **Context Manager** | 持有全局上下文；给每个角色切片；收"结果信封"；维护有界台账与出处               |
| **Skill 库**         | skill 的注册/版本/信任/检索；缺失时检索或自动生成 + 验证沙箱            |
| **执行层**             | 三档执行器：hermes Worker（工具）/ 纯 LLM（推理）/ 媒体生成器（图/视频） |
| **Model Router**    | 系统模型目录 + 用户凭证库；按"能力+预算"解析模型，运行时注入凭证             |
| **数据面**             | 共享卷存任意模态产物；DB 存结构化描述符台账                         |

## 3. 关键数据模型（草案）

### 3.1 角色（Role）

```
{ id, name, systemPrompt,
  skills: [ "name@version", ... ],     // 扁平或带依赖，由角色定义决定，不一刀切
  availableCapabilities: [ ... ],        // 受控词表中的能力
  executor: hermes | llm | media,        // 执行器档位
  source: builtin-template | user-custom }
```

### 3.2 计划（Plan / DAG）— 模板与 LLM 产出统一成同一 schema

```
{ goal, acceptanceCriteria,
  nodes: [
    { id, role, deps: [ids],
      executor: hermes | llm | media,
      requiredCapabilities: [ ... ],     // 受控词表；用于匹配 skill / 模型 / 发现缺口
      inputHint, expectedOutput }
  ] }
```

- **模板** = 预先写好的 Plan（可带占位槽）。
- **LLM 拆解** = 输出同一 schema。下游执行器不关心计划来自模板还是 LLM。
- 串行/并行由 `deps` 依赖边决定（动态，由规划结果确定）。

### 3.3 产物描述符（Artifact Descriptor）— 多模态统一

```
{ id, taskId, nodeId,
  modality: text | image | video | dataset | ...,
  uri,                 // 指向产物存储（共享卷/对象存储），不是字节本身
  mime, caption,        // 文字摘要：注入上下文用这个，不注入字节
  provenance,           // 出处（源 URL + 抓取时间等）
  meta: JSONB }          // 模态专属字段（分辨率/时长…），统一结构化外壳 + 灵活内核
```

> 不在"结构化/非结构化"里二选一：**描述符永远结构化（行 + JSONB），内容任意模态存外部**。

### 3.4 结果信封（Result Envelope）— 每个角色干完产出

```
{ role, nodeId, status,
  artifacts: [ {path/uri, kind} ],
  facts: [ {claim, sourceUrl, retrievedAt, confidence} ],   // 关键事实 + 出处
  summary,              // 一句话：做了什么、结论是什么（进有界台账）
  needsHuman?: bool }
```

### 3.5 注入给角色的上下文切片（Context Slice）

```
{ goal + acceptanceCriteria,   // 北极星，不可变
  planPosition,                // DAG 位置 + 上游给了什么
  contextLedger,               // 累积事实(带出处)+各步摘要，体量受控
  workspacePaths,              // 可读文件清单
  roleInstruction }            // 本角色具体职责
```

### 3.6 Skill 包（与 hermes 现状一致）

- 形态：`skills/<name>/SKILL.md`（+ 可选 `references/`）。**是 Markdown 操作手册，不是代码**。
- frontmatter 现成字段：`name / version(semver) / description("When to Use" 触发条件) / metadata.hermes.tags / category`。
- 库要补的社区维度：`publisher / trust(信任档) / review_status / permissions(声明会用的工具/权限)`。
- 启用：共享卷只读挂载整个 skill 目录；角色声明 `skills:[name@version]`；hermes 用 `-s` 启用子集。

### 3.7 模型目录 与 用户凭证

```
模型目录(系统托管，无密钥)：
  { modelId, provider, capabilities: [受控词表], adapterType, contextWindow, pricing }
用户凭证库(BYO，加密隔离)：
  { userId, modelId, credential(加密), quota/budget }
```

## 4. 几条关键流程

### 4.1 规划 → 人审 → 执行

```
用户目标
  → 规划器：检索模板库找最匹配 → 命中则 LLM 填槽/微调；不命中则 LLM 全自动拆解
  → 产出 Plan(DAG)，每节点标 requiredCapabilities
  → 【人审断点】用户确认/修改 DAG + 看到预估模型成本 + 设预算上限
  → Temporal 执行：按 deps 串/并行派给 Worker
       每步：Worker 读上下文切片 → 干活 → 写工作区 + 产出结果信封
             → Context Manager 更新台账（事实+出处+摘要）
  → 运行中可发 signal 暂停/改计划/重派（人随时干预）
```

### 4.2 缺失 skill 处理闭环（前期"先记待办"，不自动触发生成）

```
规划器标 requiredCapabilities → 语义匹配 skill 库 description
  命中 → 启用
  未命中 = 能力缺口
    → 生成【Skill 需求规格】{ 要解决的能力, 触发条件, 期望输入/输出, 约束/权限 }
        （一物两用：检索的查询 + 生成的任务书）
    → ① 检索社区/库（有则按来源定信任档拉回）
       ② 自动生成（"Skill 作者角色"按规格写 SKILL.md）—— 因为是 Markdown，生成可行
    → 验证沙箱（先用断言：样例输入 + 期望输出，在受限工具权限下真跑一遍）
    → 人审 → 通过才升信任档入库
  ⚠ 难点不是"生成"是"验证"；自动生成的 skill 默认落最低信任档，跑通验证+人审才升档。
  ⚠ 前期策略：缺口先记成待办，由人决定要不要补（不自动拖慢任务）。
```

### 4.3 模型按需注入 + 凭证隔离（BYO-Key，用户付费）

```
节点声明需要的能力(如 image-generation)
  → Model Router 按 { 能力满足 + 用户预算/偏好 } 在"用户已连接的模型"里选一个
     （够用就选便宜的，成本感知）
  → 取该用户凭证 → 以"任务级短时凭证"注入 Worker（仅本次调用存在）
  → Worker 调模型 → 用完即焚（凭证永不落盘/进日志/进上下文台账）
  → 记录用量（用户可见，用于他自己的计费）
  缺凭证 = 能力缺口 → 提示用户去连模型/选替代，不静默失败
```

## 5. 已确定的决策

- **执行/角色**：角色=配置非容器；Worker 池横向扩容；hermes 即运行时；执行器三档（hermes/纯LLM/媒体）。
- **状态**：Worker 无状态；状态两层外部化（共享卷存产物 + DB 存结构化台账）；目标每步在场。
- **编排**：用 **Temporal**；编排者拓扑（角色不直接对话，通信调度由编排层做）；串/并由 DAG 决定。
- **数据共享**：共享任务工作目录（文件级），不走消息总线；上下文进 DB（描述符结构化 + JSONB，字节存外部）。
- **规划**：模板 + LLM 自动拆解都要，统一成同一 Plan schema，混合式规划；人审在"计划确认 + 审核"且可中途干预。
- **数据源（示例任务）**：互联网检索（强制带出处，是"数据准确"最脆的一环）。
- **Skill**：独立 skill 库（社区取可信 + 考虑自动生成）；skill=版本化 Markdown 手册非代码；
扁平/依赖由角色定义决定；信任分级（官方/已验证发布者/社区/私有，低档限权/需人审）；
前期只组合，缺口先记待办；验证先用断言。
- **能力词表**：受控词表打底 + 语义匹配兜底（规划/skill/模型/路由全靠它对齐）。
- **模型层**：BYO-Key、用户付费；系统托管模型配置、Worker 按能力按需加载；成本感知 + 成本在人审透明；
"自动集成"= 标准接口配置即用 + 非标准接口写薄适配器（不是任意 API 零成本接入）。
- **部署目标**：服务器/云，k8s 容器集群承载（先在 Mac 验证——具体节奏本文不展开）。

## 6. 待定项（需继续讨论）

1. **Skill 依赖机制**：何时是依赖树、如何解析/版本约束。
2. **谁能往 skill 库发布**：私有库 / 受信任发布者 / 开放社区 —— 决定审核流水线现在还是后置。
3. **验证沙箱的断言来源**：谁准备样例+期望输出；是否需要 LLM-as-judge 兜底。
4. **凭证保险库的具体技术**：KMS/vault 选型；Mac 阶段的加密存储方案。
5. **媒体产物存储**：共享卷 → 对象存储（S3/MinIO）的切换时机与选型。
6. **发布/通知 connector**：渠道（平台 API/企业微信/邮件）做成配置；具体 connector 后定。
7. **du**图像/视频/向量化各自的适配器接口标准。
8. **成本/预算模型**：每任务预算、超预算行为（降级/暂停/询问）。
9. **角色自由生成的边界**：前期限定"选已有 skill + 生成系统提示词"；"生成全新 skill"为后续高级能力且需人审。

## 7. 重点风险（设计阶段就要正视，不能后补）

1. **互联网数据准确性**：网页过时/矛盾/SEO 垃圾/诱发幻觉。对策：每条事实带出处+时间、关键结论多源交叉验证、
  矛盾标记而非硬合并、审核角色明确核对引用与时效。
2. **共享 Worker 池的凭证隔离**：A 的任务绝不能拿到 B 的密钥。对策：加密存储 + 任务级短时下发 + 用完即焚 +
  永不落盘/日志/上下文。做不好就是重大数据事故。
3. **自动生成 skill 的质量与安全**：真正的难点是验证而非生成。对策：验证沙箱 + 默认最低信任档 + 受限权限 +
  人审升档；否则库会被低质量/有注入风险的 skill 污染。
4. **递归自造 skill 的漂移**：平台用自己的 agent 扩展自己的 skill，缺乏闸门会失控（库膨胀、质量下降）。
5. **BYO-Key 的 onboarding 摩擦**：用户要连一堆模型 Key。对策：onboarding 做成"连接模型"步骤，
  能力按已连模型动态点亮，未连能力在创建角色时灰掉并提示。
6. **"自动集成模型"的预期边界**：其真实含义是"系统托管配置 + 按需注入凭证 + 标准接口配置即用"，
  不是"任意模型 API 零代码接入"——要管理好预期。
