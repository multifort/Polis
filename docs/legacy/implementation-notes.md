# 各部分实现说明（配套 interaction-diagram.svg）

> 配合 `interaction-diagram.svg` 看。每个组件给：**职责 / 实现方式 / 关键点**。
> 这是设计层面的实现指引，不含落地节奏。最后更新：2026-06-09。
>
> 全局两条底色：① **本地先 docker-compose + volume，生产上 k8s + Vault/KMS + 对象存储**（同接口）；
> ② **执行器去 hermes 化**——默认 lite-agent，skill 是 Markdown 与运行时无关。

---

## 控制面

### 角色注册表
- **职责**：存角色配置。
- **实现**：PostgreSQL 一张表，记录 `{name, 系统提示词, skills[name@version], 可用能力, executor 档位, 来源}`。
- **关键点**：角色是**纯数据**，不是容器；新增角色 = 插一行，零运行时成本。

### 模板库
- **职责**：预写好的 Plan(DAG) 模板，供规划器复用。
- **实现**：模板 = 与 LLM 产出同构的 Plan schema（带占位槽），版本化存 DB；按目标做语义检索取最匹配。
- **关键点**：模板和 LLM 拆解**统一成同一 Plan schema**，下游不区分来源。

### 规划器
- **职责**：把用户目标变成结构化 Plan(DAG)。
- **实现**：混合式——先检索模板命中则 LLM 填槽/微调，不命中则 LLM 全自动拆解；每个节点标 `requiredCapabilities`（受控词表）。
- **关键点**：产出交人审确认；**不做 skill 依赖解析**（skill 扁平自包含）。

### 人审 UI
- **职责**：计划确认、审核断点、运行中干预、成本透明。
- **实现**：复用现有 Web + 移动端；展示 Plan(DAG) + 预估模型成本；用户操作经 **Temporal signal** 回写（暂停/改计划/重派）。
- **关键点**：人在"规划确认 + 审核"两处介入，可中途干预。

---

## 编排层

### Temporal
- **职责**：按 DAG 调度、串/并、重试、人审暂停、signal 干预。
- **实现**：Temporal SDK；**一个任务 = 一个 Workflow，一个角色执行 = 一个 Activity**；依赖边决定串/并；人审断点用 Workflow 的"等待 signal"实现。
- **关键点**：持久化编排，断点续跑、重试、长流程人审暂停都由它兜底，不自己手写。

---

## 协同支撑

### Context Manager（上下文管家）
- **职责**：持有全局上下文；给每个角色切片；收结果信封；维护有界台账 + 出处。
- **实现**：独立服务 + DB。每步执行前组装"上下文切片"（目标+计划位置+台账+工作区清单）注入 Worker；执行后收"结果信封"（产物+事实+出处+摘要）写台账。
- **关键点**：**台账有界**（注入摘要不注入字节），**出处必带**（应对互联网数据准确性）。

### Skill 库
- **职责**：skill 的注册/版本/信任/检索 + 自动生成/验证。
- **实现**：**共享卷存 `SKILL.md + references/`，DB 存元数据**（name/version/desc/tags/trust/permissions）；发布走"提交→自动检查→**强制人审**→定信任档→上架"。
- **关键点**：skill = 版本化 Markdown 手册（非代码）；description 的 "When to Use" 同时驱动人检索和机器选用；**可深看 NoDeskAI 的 `genehub`/`skill-keeper`(TS) 直接复用**。

### 验证沙箱
- **职责**：判定（尤其自动生成的）skill 是否可用、安全。
- **实现**：**断言（客观：格式/字段/无错）+ LLM-as-judge（主观：是否满足意图）**；在**受限工具权限**下跑样例输入。
- **关键点**：自动生成的 skill 默认最低信任档，跑通验证 + **人审为终审**才升档。

## 执行层（可插拔 Executor · 无状态）

> 角色按需声明用哪档；运行时注入三样（上下文切片 / 启用技能 / 模型配置+短时凭证）。

### lite-agent（**默认**）
- **职责**：要用工具的角色的轻量执行器。
- **实现**：**自研一个最小 agent 循环**——注入系统提示词 + 启用的 SKILL.md 手册 + 用模型原生 tool-calling + 一组可控的小工具集（检索、文件读写等），循环到完成。
- **关键点**：比 spawn 整个 hermes 轻得多；skill 是 Markdown 直接注入，无需 hermes。代价是自研循环/工具执行/安全要自己保证。

### 纯 LLM / 媒体生成器 / hermes
- **纯 LLM**：一次模型调用、无工具（审核/校正这类）。
- **媒体生成器**：调图像/视频模型服务（经 Model Router）。
- **hermes**：仅在确需其特定能力时 spawn，可选备用（偏重）。

---

## 模型接入

### Model Router
- **职责**：把"需要的能力"解析成"具体模型 + 调用方式"。
- **实现**：模型目录（系统托管、无密钥），每模型一份**声明式 HTTP 连接器**配置 `{endpoint, authType, requestTemplate, responseParsePath, streamMode}`；按 `能力 + 用户预算/偏好` 选模型。
- **关键点**：连接器覆盖标准同步 + SSE；**长尾（异步视频/签名鉴权）走薄适配器逃生口**。

### Credential Broker
- **职责**：BYO-Key 的安全存储 + 按需注入。
- **实现**：统一接口 `getScopedCredential / revoke`；**信封加密**（DEK+KMS）；后端可插拔——生产 Vault/云 KMS（k8s 经 Vault Agent/CSI 注入 tmpfs、短 TTL），本地加密文件/sqlite。
- **关键点**：Worker 只拿**任务级短时句柄**；凭证**永不落盘/日志/上下文**；每次取用审计。共享池里**严禁跨用户串密钥**。

### 模型后端
- **职责**：实际的各家模型（用户付费）。
- **实现**：BYO-Key；统一经声明式 HTTP 连接器调用，长尾用薄适配器。

---

## 数据面

### 共享卷（任务工作目录）
- **职责**：存任意模态的大块产物。
- **实现**：每任务一个目录；本地 hostPath/volume，生产 RWX PVC（NFS/CephFS/EFS）；媒体大文件后续转对象存储（S3/MinIO，接口不变）。
- **关键点**：角色间不发消息，靠"A 写文件 / B 读文件"+ 编排层控制谁何时挂载。

### DB（结构化）
- **职责**：结构化状态与台账。
- **实现**：PostgreSQL；存 `角色 / 计划 / 产物描述符台账(出处, JSONB 容纳模态专属字段) / 凭证密文`。
- **关键点**：**描述符永远结构化（行+JSONB），内容字节存共享卷/对象存储**——统一了结构化与多模态。

---

## 跨层工具：互联网检索（Role A 用）
- **实现**：用隔离浏览器做受控抓取（**可复用 NoDeskAI 的 `browser-pilot`**：隔离 Chrome + 指纹/网络隔离 + REST/CLI）。
- **关键点**：每条抓取的事实强制带 **源 URL + 抓取时间**，进结果信封 → 台账，供审核角色核对引用与时效。
