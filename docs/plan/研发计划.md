# Polis 研发计划（V1）

> 本计划把 `docs/design/02~07` 的任务分解(WBS)卷成可执行的里程碑与节奏。
> 配套任务清单见 [研发任务清单.md](研发任务清单.md)。过程纪律见 `docs/constraints/15`，门禁见 `14`。
> 版本：V1.0 / 2026-06-16（估时为初估，进入各里程碑前按 DoR 复评）。

---

## 1. 目标与范围

**V1 唯一目标：验证「大脑能不能转」**——用一个真实场景（采购分析）把
*意图→编配→规划→路由→执行→记忆→审批→可观测* 的最小闭环跑通，并用数据决定是否进 V2。

- 范围（做）：薄 Org + 1~2 Agent + MCP 工具 + 模板优先 Planner + Plan 校验 + Temporal
  + 任务记忆(pgvector) + LiteLLM + 凭证短时注入 + Langfuse + 审批收件箱 + 最小 Web 前端。
- 不做（后置 V2/V3）：完整能力路由历史信号、三层记忆衰减治理、编配器全自动、guardrails 全量、
  Marketplace/治理/多租户 RLS/K8s（见 `07 §4`）。

## 3. 关键路径与并行轨

```text
后端主轨(关键路径):  M0 → M1 → M2 → M3 → M4 → M7
                                 ├→ M5 ┐
                                 └→ M6 ┘→(汇入)M4/M7
前端轨(并行):        M0 →(契约桩)→ MW1 → MW2(待 M2/M3/M6 接口就绪)
基础设施/横切:        M0 起 docker-compose；M1 起 org_id 隔离+审计骨架(T8.1)，全程随功能补隔离测试
```
- **并行建议**：1 人推后端关键路径(M1→M3→M4)，1 人推 M5+M6（记忆/网关/观测可与 M3 并行），
  1 人推前端轨（MW1 用 OpenAPI 桩先行，MW2 待接口就绪）。2~3 人可在 M2 后明显并行。
- M4 是汇流点：依赖 M3(节点)、M5(写回检索)、M6(模型/凭证)，安排在三者各自最小可用之后。

## 5. V1 验收门（决定是否进 V2 · 来自 03 §9）

| 指标 | 口径 | 阈值 |
|---|---|---|
| DAG 可用率 | 产出 DAG 经校验+人评可执行的比例 | ≥ 70% |
| 路由命中率 | Router 选的 Agent 与人评最优一致 | ≥ 75% |
| 人审通过率 | 端到端产出一次通过 | ≥ 50% |
| 单任务成本/时延 | LiteLLM 统计 | 在预算内 |

**未达标 → 回炉调 Planner/Router/词表/记忆，不是加模块**（F3/F9）。达标才规划 V2。

## 6. 节奏与纪律（对接 constraints/15）

- **任务驱动**：每条任务进开发前过 DoR；完成过 DoD（见 CLAUDE.md §3）。1~3 天粒度，超则拆分。
- **迭代**：按里程碑评审，用**数据/产出**验收，不用纸面论证。
- **实证先行(F9)**：M3/M4 的不确定硬核(规划质量/路由)先做最小原型 + eval，再决定扩不扩。
- **决策留痕(F6)**：里程碑中产生的选型/权衡写 ADR。
- **IP 捕获(F10)**：识别到 `08` 的 P1–P8 候选点即时登记 IP 台账。

## 7. 主要风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| Planner 产出不可靠 | DAG 可用率不达门 | 模板优先兜底 + Plan 校验拦截 + 有界重规划；先原型+eval(F9) |
| 路由冷启动选不准 | 路由命中率低 | 冷启动降权历史/亲和、能力+成本主导；积累 run_manifest 后切完整公式 |
| 注入/越权 | 安全事故 | 三道防线(Guardrails+最小权限+危险 gate)；M4 必带注入样例测试 |
| BYO 凭证串租 | 严重安全 | 任务级短时句柄+用完即焚+审计；M5 带跨用户不串 Key 测试 |
| 范围蔓延 | 拖慢 V1 | 严守"核心 vs 增强"表；V1 验收门前不引入增强项 |

## 8. V2 / V3 展望（简）

- **V2（能协同）**：完整能力路由(历史) + 三层记忆 + 编配器(生成+验证) + 有界重规划 + guardrails 全量 + 共享记忆并发；对象存储换 MinIO。
- **V3（能治理）**：Agent Registry/Marketplace + 预算/多租户 RLS + Policy Engine(OPA) + K8s 弹性 +（按需）微服务拆分。
