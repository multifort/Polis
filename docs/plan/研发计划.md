# Polis 研发计划（V1）

> 本计划把 `docs/design/02~07` 的任务分解(WBS)卷成可执行的里程碑与节奏。
> 配套任务清单见 [研发任务清单.md](研发任务清单.md)。过程纪律见 `docs/constraints/15`，门禁见 `14`。
> 版本：V1.0 / 2026-06-16（估时为初估，进入各里程碑前按 DoR 复评）。

---

## 1. 目标与范围

**V1 唯一目标：验证「大脑能不能转」**——用一个真实场景（采购分析）把
*意图→编配→规划→路由→执行→记忆→审批→可观测* 的最小闭环跑通，并用数据决定是否进 V2。

- 范围（做）：身份/账号(09，登录+多 Org+权限+RLS) + 薄 Org + 1~2 Agent + MCP 工具 + 模板优先 Planner
  + Plan 校验 + Temporal + 任务记忆(pgvector) + LiteLLM + 凭证短时注入 + Langfuse + 审批收件箱 + 最小 Web 前端。
- 不做（后置 V2/V3）：完整能力路由历史信号、三层记忆衰减治理、编配器全自动、guardrails 全量、
  Marketplace/治理/多租户 RLS/K8s（见 `07 §4`）。

```text
后端主轨(关键路径):  M0 → M1 → M2 → M3 → M4 → M7
                                 ├→ M5 ┐
                                 └→ M6 ┘→(汇入)M4/M7
```

## 5. V1 验收门（决定是否进 V2 · 来自 03 §9）

| 指标 | 口径 | 阈值 |
|---|---|---|
| DAG 可用率 | 产出 DAG 经校验+人评可执行的比例 | ≥ 70% |
| 路由命中率 | Router 选的 Agent 与人评最优一致 | ≥ 75% |
| 人审通过率 | 端到端产出一次通过 | ≥ 50% |
| 单任务成本/时延 | LiteLLM 统计 | 在预算内 |

**未达标 → 回炉调 Planner/Router/词表/记忆，不是加模块**（F3/F9）。达标才规划 V2。

## 6. 节奏与纪律（对接 constraints/15）

- **任务驱动**：每条任务进开发前过 DoR；完成过 DoD（见 CLAUDE.md §3）。1~3 天粒度，超则拆分。
- **迭代**：按里程碑评审，用**数据/产出**验收，不用纸面论证。
- **实证先行(F9)**：M3/M4 的不确定硬核(规划质量/路由)先做最小原型 + eval，再决定扩不扩。
- **决策留痕(F6)**：里程碑中产生的选型/权衡写 ADR。
- **IP 捕获(F10)**：识别到 `08` 的 P1–P8 候选点即时登记 IP 台账。

## 7. 主要风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| Planner 产出不可靠 | DAG 可用率不达门 | 模板优先兜底 + Plan 校验拦截 + 有界重规划；先原型+eval(F9) |
| 路由冷启动选不准 | 路由命中率低 | 冷启动降权历史/亲和、能力+成本主导；积累 run_manifest 后切完整公式 |
| 注入/越权 | 安全事故 | 三道防线(Guardrails+最小权限+危险 gate)；M4 必带注入样例测试 |
| BYO 凭证串租 | 严重安全 | 任务级短时句柄+用完即焚+审计；M5 带跨用户不串 Key 测试 |
| 范围蔓延 | 拖慢 V1 | 严守"核心 vs 增强"表；V1 验收门前不引入增强项 |

## 8. 历史展望（已由后续设计取代）

- **V2（能协同）**：完整能力路由(历史) + 三层记忆 + 编配器(生成+验证) + 有界重规划 + guardrails 全量 + 共享记忆并发；对象存储换 MinIO。
- **原 V3 展望（已废止）**：曾设想引入 Agent Registry/Marketplace、OPA、K8s 和微服务拆分。当前第三轮
  以稳定企业运行内核为目标，边界和路线以本文第 10 节及 `docs/design/v3/kernel/` 为准；上述基础设施
  不属于当前 V3。

# 9. V2 研发计划（能协同 · 1→10）

> V1 验收门已通过（DAG/路由 100% · 人审 91.7% · 成本时延在预算），进入 V2。
> 设计全集见 `docs/design/v2/`（00 总纲 + 01–05 + ADR-0009~0014）；本节是计划级总览，任务级见 [研发任务清单.md](研发任务清单.md) 的 V2 章节。

## 9.1 目标与范围
**V2 目标：从「能转」到「能协同」**——让目标**驱动结构本身**（检索/生成对的计划与团队）、跑得稳、错能修、长能跑、并发可控，并让资产**自增长**（越用越聪明）。

- 范围（做）：生成式内核（检索→生成→校验→门控→沉淀）+ 三层记忆(黑板/晋升/预算) + 协同框架(控制面/分级纠错/长跑) + 资产仓库(三层/可见性 public+private) + 产品面(任务实体/附件/执行记录/导出/看板/MinIO)。
- 不做（当时的后置项）：跨组织共享/Marketplace、Policy Engine(OPA)、K8s、微服务拆分。该历史表述
  不代表当前 V3 承诺；当前范围见本文第 10 节。
- 非阻塞并行轨（按需）：Guardrails-AI、真实外部 MCP、多模型/主模型/按 Agent 选型(第二步)、reranker。

## 9.2 里程碑总览（4 阶段 · 17 里程碑）

# 10. V3 研发计划（稳定企业运行内核）

> 状态：开发中；规范版本 V3.4。权威设计见
> [`docs/design/v3/kernel/`](../design/v3/kernel/README.md)，实施任务与验收见
> [`07-实施任务与测试验收.md`](../design/v3/kernel/07-实施任务与测试验收.md)。若本节与内核规格冲突，
> 以内核规格、ADR-0015～0018 和 `CLAUDE.md` 为准。

## 10.1 目标与边界

V3 不重新设计用户页面，也不推翻现有 Task/Plan/Temporal/Agent 能力。它通过可版本化定义、Scope、责任、
类型化 Command、事件、评价和恢复协议，把现有系统渐进演化为可承载多行业业务的企业运行内核。

- **范围内**：Definition/Bundle、Scope、责任绑定、WorkItem、三类 Command、状态与策略、Outbox、Temporal
  适配、评价闭环、迁移兼容、对账、审计和租户隔离。
- **范围外**：新产品信息架构、首个行业定制、Marketplace、OPA、K8s、微服务拆分及旧表物理删除。
- **连续性要求**：K0～K6 期间现有页面、API、数据和深链接持续可用；新产品交互仅在 K5 Gate 后进入 PX。

|---|---|---|---|

## 10.3 执行纪律

1. 严格按 K0 → K6 顺序推进，上一阶段 Gate 未通过不得开启下一阶段功能开关；
2. 每个任务保持 1～3 天粒度，并满足 `10-无歧义自主开发验收.md` 第 5 节工作包合同；
3. migration 先加后删、功能开关生产默认关闭、同一阶段只有一个业务真相源；
4. 沙箱阻止本地 socket 时按 `CLAUDE.md` 授权以相同命令提升权限重跑并记录真实结果；
5. 不影响当前 Gate 的工程妥协登记 `docs/tech-debt.md`；影响正确性、协议唯一性或安全边界的问题不得
   降级为技术债继续开发。
