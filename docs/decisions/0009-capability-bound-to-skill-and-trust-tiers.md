# ADR-0009：能力绑定 Skill + 生成物信任分层

- 状态：proposed
- 日期：2026-06-22
- 关联：[V2 · 生成式内核](../design/v2/01-生成式内核（编配器·规划器·能力模型·信任）.md)；演进 [ADR-0006](0006-m2-deterministic-provisioning.md)

## 背景
V2 让编配器/规划器从"只能选预设"升级为"选不到就 LLM 生成（角色/计划/技能）"。两个根问题待定：
1. **能力词表归属谁**——绑 Agent 还是 Skill？
2. **生成物如何可信**——既要 CLAUDE.md §4.6"AI 生成默认不可信、须人审"，又要避免"每步人审导致系统不可用"。

## 选项
1. 能力绑 **Agent**（V1 现状：`agent_capability` 直接声明）—— 简单，但可能"声称有能力却无 Skill 实现"的空头承诺，背书无依据。
2. 能力绑 **Skill**，Agent 能力由其 Skills 并集派生 —— 背书链闭合（能力信用来自实现它的 Skill 过审+eval）；Agent 组合=拼已审积木，可自动化。
3. 信任：每步人审（安全但慢/不可用）/ 全自动（快但不安全）/ **分层**（按资产性质分路径）。

## 决定
- **能力由 Skill 实现（`skill.capability`），Agent 能力派生**（`agent_capability` 降级为派生缓存/兼容旧预设）。能力词表开放（允许新增）但**只有被审过的 Skill + eval 背书的能力才 `active`**，规划器只用 active 能力。
- **信任分三路径**：
  - 任务**热路径**（生成计划/组队→跑→出结果）→ **自动评测(Evaluator)背书**，无人卡；
  - **新 Skill 入库** → **人工审核**（代码/安全/沙箱）；
  - **场景模板入库** → **用户满意度**（用户主动保存）。
- 推论：**自动化链跑到"拼装已有 Skill"为止；只有"需要造新 Skill"才停下等人审**（卡点由模型长出，非人为设）。

## 后果
- 正面：人审收敛到唯一的"新代码"资产(Skill)；Agent/角色可自动生成+eval；满足"信任周期尽量自动"；能力词表既开放又有背书。
- 负面/代价：高度依赖 **Evaluator 的可靠性**（升级为核心组件）；`agent_capability` 写入逻辑改为派生；能力新增需语义去重防同义爆炸。
- 影响范围：内核（规划/编配/能力）、Skill 仓库、Evaluator、路由。
