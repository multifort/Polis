# 14 工程化约束（仓库 / 质量门禁 / 治理）

> 覆盖 E1–E3, E5–E7, E9–E17（**已排除 E4 CI/CD、E8 容器部署**）。
> ⚠ 因暂不做 CI/CD：**门禁初期以本地 pre-commit + pre-push 钩子执行**，引入 CI 后再迁移为流水线。
> 每项：**规则 / 工具 / 门禁形态**。

---

## E1 仓库与分支策略 ★
- 规则：Polis 独立仓库；建议**单仓多包**(`backend/` `frontend/` `infra/` `docs/`)；
  **trunk-based**：短分支 + 快速合并；分支命名 `feat/…` `fix/…` `chore/…`；`main` 受保护、禁直推。
- 工具：git 分支保护(本地约定，CI 期再加 server 端)。**门禁：约定+评审**。

## E2 提交规范 ★
- 规则：**Conventional Commits**(`feat: / fix: / docs: / refactor: / test: / chore:`)；
  一次提交一件事；提交信息说清 what+why；AI 协作提交带 `Co-Authored-By`。
- 工具：**commitlint + commitizen**(pre-commit 校验)。**门禁：可(commitlint)**。

## E3 代码评审规范 ★
- 规则：所有改动经 PR；**至少 1 人评审**通过才合并；PR 模板含(动机/改动/测试/风险/关联任务)；
  评审清单引用 11/12 的"评审项"(分层依赖、org 过滤、禁 any、只经网关调模型…)。
- 工具：PR 模板 + 评审清单。**门禁：评审(人工)**。

## E5 版本与发布
- 规则：**semver**；维护 CHANGELOG(按 Conventional Commits 自动生成可选)；
  发布打 tag + release note；后端/前端各自版本线。
- 工具：标准 git tag(+ 可选 semantic-release，CI 期)。**门禁：流程**。

## E6 依赖管理 ★
- 规则：**锁定 lockfile**(uv/poetry、pnpm)；新增依赖需评审理由；定期更新 + **CVE 审计**；
  禁未锁版本、禁随意引重复同类库。
- 工具：`pip-audit`/`npm audit`/`pnpm audit`(pre-push)。**门禁：可(audit)**。

## E7 环境管理 ★
- 规则：环境分 **dev/staging/prod**；配置全走环境变量(`pydantic-settings`/`NEXT_PUBLIC_*`)；
  `.env.example` 提供模板，真实 `.env` 不入库；密钥经 secret 管理(C10)，**严禁入库/日志**。
- 工具：dotenv + gitleaks(secret 扫描)。**门禁：可(secret 扫描)**。

## E9 数据库迁移规范 ★
- 规则：**只走 Alembic**；迁移**向后兼容**(先加列/表、灰度、再删)，禁破坏性直改；每个迁移可回滚(downgrade)；
  迁移与代码同 PR；上线前在 staging 跑过。
- 工具：`alembic check` + 迁移评审清单。**门禁：可(alembic check)**。

## E10 测试策略与覆盖率门槛 ★
- 规则：分层——单元(domain/service) > 集成(repo/testcontainers、api) > e2e(关键流)；
  **核心模块(规划/路由/执行/凭证/记忆)覆盖率 ≥ 70%**(可调)；新功能必带测试。
- 工具：pytest+coverage / vitest+coverage / playwright(pre-push 跑关键集)。**门禁：可(覆盖率阈值)**。

## E11 质量门禁 ★
- 规则：**lint + 类型 + 测试 + secret 扫描 不过则不可合并**；初期用 **pre-commit/pre-push 钩子**本地强制，
  绕过(`--no-verify`)需评审说明。引入 CI 后迁移为必过检查。
- 工具：pre-commit 框架(统一管理 ruff/mypy/eslint/prettier/commitlint/gitleaks)。**门禁：可(本地钩子)**。

## E12 文档规范(ADR / OpenAPI) ★
- 规则：架构/选型决策写 **ADR**(`docs/decisions/NNNN-*.md`，状态 proposed/accepted/superseded)；
  后端 API 以 **OpenAPI 自动生成为准**，前端类型从它生成(B8)；README/运行手册随代码更新(doc-as-code)。
- 工具：ADR 模板 + FastAPI OpenAPI 导出。**门禁：评审(架构变更须有 ADR)**。

## E13 监控与告警
- 规则：结构化日志 + **Langfuse trace**(LLM/Agent 全链路) + 基础指标(请求量/错误率/时延/成本)；
  关键失败(凭证、规划、外部依赖)有告警通道；定义初步 SLO(后置细化)。
- 工具：Langfuse + 日志聚合(后置选型)。**门禁：否(建设项)**。

## E14 安全工程 ★
- 规则：**SAST(bandit)** + **依赖 CVE(pip-audit/npm audit)** + **secret 扫描(gitleaks)** 纳入门禁；
  (可选)SBOM 生成；注入防御见 04/C16；最小权限见 C11。
- 工具：bandit / pip-audit / gitleaks(pre-commit/pre-push)。**门禁：可**。

## E15 性能基准 / 压测
- 规则：关键路径(规划/执行/检索)留**基准用例**；上线前对热点接口做压测(locust/k6)，记录基线；
  回归显著退化需说明。初期可手动，规模期常态化。
- 工具：locust/k6(按需)。**门禁：否(里程碑项)**。

## E16 错误监控
- 规则：前后端接入**错误监控**(Sentry 类)，捕获未处理异常 + 关联 trace/correlation-id；
  错误分级 + 责任归属。初期可用日志兜底，尽早接入。
- 工具：Sentry(自托管/SaaS)。**门禁：否(建设项)**。

## E17 数据治理与多租户隔离测试 ★
- 规则：所有业务表 `org_id` 隔离(C11)；**专门的隔离回归测试**(构造 A/B 两租户，断言互不可见)纳入测试集；
  PII 字段标注 + 脱敏 + 留存策略；敏感读写审计(audit_log)。
- 工具：pytest 隔离用例 + (规模期)PG RLS。**门禁：可(隔离测试)**。
