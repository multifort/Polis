# Polis 产品定位与待决问题（赛道 · 买家 · 数据安全）

> 状态：**讨论纪要 / 假设待验证**（2026-06-22）｜性质：产品战略，非技术决策（暂不落 ADR，待客户发现验证后再固化）
> 缘起：V2 开发前，厘清"产品定位、出发点、第一个买单人、数据安全模型"。当时无明确数据可及性、无明确买单人，目标是先把定位想清楚再去找特定的人卖。

---

## 1. 核心疑问点（讨论的出发点）
1. 就当前产品设计，**具体应用的细分赛道在哪、如何定位**？
2. 大部分企业管理已依赖信息化/数字化（业务系统），AI 要进**企业管理核心场景**，如何**不与既有信息化冲突**又**发挥 AI 优势**？
3. 在没有数据可及性、没有买单人的前提下，**先卖给谁（哪个 persona）**？
4. 这类买家数据隐私/安全要求高，当前直连第三方模型有泄露风险，**能否用专业分析的小模型替代**？

---

## 2. 定位结论（假设，倾向性强）

### 2.1 进"分析-决策层"，不进"记录-交易层"
企业软件三层：
| 层 | 代表 | 谁强 |
|---|---|---|
| 记录-交易层 (SoR) | ERP/CRM/MES/财务 | **信息化强**（事务/合规/审计）|
| 互动层 (SoE) | OA/门户/IM | 各半 |
| **分析-决策-协同层 (SoI/Agency)** | 基本空白，靠人脑+Excel+会议 | **AI 主场** |

**Polis = 第三层"治理化 AI 虚拟职能团队/参谋层"**，不去重做 ERP。理由：交易层与信息化正面冲突且打不过；第三层是信息化没覆盖好、又最耗人的判断密集型脑力活。

### 2.2 与信息化"不冲突"的协作模型
Polis 站在信息化**之上**，把信息化当**数据源 + 执行末端（工具，经 MCP/API）**，自己只当**大脑/参谋**：
- 读多写少；产出"结论/草稿/建议/工单"而非记账；权威交易仍由原系统 + 人确认（审批 gate）。
- 信息化越完善，Polis 越有料 → **互补放大，不是替代**；对已重金投信息化的企业，是"让既有数据真正产生决策价值"的增量层（保护既有投资）。
- 治理能力（审批/出处/审计/成本透明/org 隔离）= AI 进核心场景的**入场券**，不是负担。

### 2.3 差异化
- vs 单体 Copilot/ChatGPT：多角色分工 + 可复用场景模板 + 治理审计（受治理的多步职能工作，非一次性问答）。
- vs RPA：处理非结构化判断（RPA 只自动化结构化点击）。
- vs BI：给分析+结论+下一步动作（BI 只给看板）。
- vs 垂直 SaaS：平台按场景开出角色化团队，靠模板飞轮横向复制。

### 2.4 定位一句话（候选）
> Polis = 架在企业既有信息化之上的"治理化 AI 虚拟职能团队"——按业务场景，把散落各系统的数据做成分析、结论与决策草案，全程可审批、可追溯、成本透明。

## 3. 第一个买家假设：买家=用户=数据拥有者=受益人"四合一"

### 3.1 结论：先卖给"靠脑力/分析/报告交付吃饭的专业服务机构"（乙方）
管理咨询、行业研究、尽职调查、市场调研/数据分析、会计税务审计、（部分）律所——**分析报告本身就是他们的产品**。

为什么（六条）：
1. **痛且高频**：分析师人力是主成本。
2. **买单人明确、能独立拍板**（合伙人/交付负责人）。
3. **数据自带、零信息化集成**：项目数据以**文档**随项目进来（正是 P2 附件上传的用途）——**不用碰客户 ERP，第 2 个疑问在此买家身上直接不存在**。
4. **天然审阅文化**：本就有专家复核，契合审批 gate，信任门槛低。
5. **ROI 极清晰**：原来 N 人 M 天的报告 → 大幅压缩，直接换毛利。
6. **隐喻 1:1**：咨询项目组 = 项目经理+研究员+分析师+报告员 = Polis 的角色化虚拟公司。

### 3.2 Polis 已建特性与该买家高度契合
| 已设计/在做 | 对该买家价值 |
|---|---|
| 多角色团队 + 场景模板复用 | 把资深方法论/SOP 产品化沉淀 |
| 附件上传(P2) + 导出 md/pdf(P3) | 材料进、交付物出 |
| 审批/出处/审计 | 质量控制 + 对客户可辩护的出处 |
| 成本透明 | 单项目盈利核算 |
| 采购分析(已跑通) | 即"供应商尽调/比价报告"一种可卖交付物 |

### 3.3 暂时排除
- **企业内部职能部门**（采购总监/CFO/风控）：盘子大，但第一单"数据/买单/痛点"三道门分散、要碰信息化、信任周期长 → **第二步**（用服务机构跑出模板与证据后，land-and-expand 回卖给"自己也做同样分析"的企业内部团队）。
- **个人/Prosumer**：太小、不需要治理、用不上核心差异化。

## 4. 数据隐私与模型部署

### 4.1 关键认知：隐私的解药是"部署方式（数据不出域）"，不是"模型变小"
- **小模型 ≠ 安全；自托管才安全**。小模型跑在第三方云上数据照样出域；大模型部署在客户内网/VPC 则数据不离边界。
- 当前 DeepSeek API 风险点：**数据流到第三方服务器**。注意 **BYO-Key 只解决"用谁的额度"，不解决"数据出不出域"**——对尽调/审计基本一票否决。

### 4.2 Polis 已为"换自托管模型"架好（配置级，非重构）
- LiteLLM 网关**模型无关**（M6-C）：把 chat 指向自托管 OpenAI 兼容端点即可。
- **已有先例**：embedding 用**本地 TEI(bge)**（M6-A）——"本地模型挂 OpenAI 兼容端点"已跑通，chat 同套路。
- → **"私有化部署"是一个部署档，不是重构**，本身是卖点。

### 4.3 产品化为"部署分层"
| 档 | 数据边界 | 适合 | 代价 |
|---|---|---|---|
| SaaS（云模型） | 出域第三方 | 非敏感 | 最快/最便宜/质量最高 |
| 私有端点（云上单租户 + 无留存） | 仍在云、隔离不留存 | 中敏感 | 中 |
| **本地/VPC 私有化（自托管开源模型）** | **不出客户边界** | 尽调/审计/涉密 | GPU + 运维 |

→ 对目标买家，**第三档是入场券**，做成标准交付选项。

### 4.4 "专业小模型替代"的权衡（结论）
1. **质量会掉，且此赛道最吃质量**（F3 已证质量对模型能力高度敏感）。
2. **"小"是误区**：要私有化应选**可自托管的强开源模型（32B–72B 级）**，隐私靠自托管、能力别牺牲太多。
3. **Polis 架构能部分补偿模型变弱**：多 Agent 拆小步 + 确定性校验 + 有界自修复 + 检索接地 → 不需巨无霸一口气做完。
4. **现在不自研/微调专用小模型**（无训练数据、无 MLOps）；它是**后期护城河**，不是起点。起点 = 模型可插拔 + 提供自托管档。
5. **架构级数据最小化**（无论用哪个模型）：记忆黑板/按需拉取 + 脱敏(redaction) skill，减少暴露面，可写进合规话术。

## 5. 待验证清单（下一步该做的）
1. **三变量交集**：哪个细分场景**数据可及 + 有明确买单人 + 痛且高频**（在专业服务机构里找）。
2. **可触达性（当前决定变量）**：这个月能约到 5~8 个哪类机构（咨询/审计/调研/尽调）的交付负责人访谈。
3. **上云接受度**：目标买家对"数据上云"是**一票否决**，还是"合同+隔离可接受"——决定是否要**提前插一个"自托管模型档"验证任务**。
4. **验证方法**：① 客户访谈（"一份报告几人几天、卡在哪"）；② **零集成 demo**（拿其真实项目脱敏文档当附件喂 Polis，跑出 80 分草稿，不碰其任何系统）。

## 6. 对 V2 的影响
- **可能新增（按需，非阻塞）**：在 V2 之外/并行插一个**"自托管开源模型档"验证任务**——把 LiteLLM 网关指到一个自托管开源 chat 模型（沿用 bge 本地化套路），对比采购分析的质量/成本，作为"私有化部署"能力的证据。**仅当目标买家"数据不出内网"时优先做。**
- **脱敏/redaction skill**：可作为仓库里一个早期 Skill 候选。
- 其余 V2 计划（阶段 0/A/B/C）不受影响，按 `docs/design/v2/00` 推进。
