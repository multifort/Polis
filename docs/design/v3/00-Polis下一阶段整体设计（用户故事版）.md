# Polis 下一阶段整体设计（用户故事版）

> 状态：讨论稿 v0.1
> 日期：2026-07-14

> 配套：[产品版](./01-Polis下一阶段整体设计（产品版）.md)

## 1. 一句话故事

**Polis 帮助不同经营阶段的人，以“创生、共建或嵌入”的方式建立一支可持续工作的虚拟企业或虚拟职能团队。人定义方向、关系、承诺和边界，Polis 负责组织角色、规划工作、驱动 Agent 执行、检查结果，并在真实经营反馈中持续调整。**

它不是单纯的任务拆解器，也不是让用户手工维护 Agent 的管理后台。

用户最终购买的是一种经营结果：

- 独立运营者在不提前招聘完整团队的情况下，获得团队级交付能力；
- 中小企业把已经形成的方法和项目流程交给人机团队共同运行；
- 成熟企业在不重建组织和现有系统的前提下，增加受治理的虚拟职能团队。

## 2. 三类用户，三种成立条件

### 2.1 创生模式：独立运营者与专业创业者

典型用户不是“只有一个点子、还不知道客户是谁”的泛创业者，而是：

> 已经拥有专业能力，知道大致服务谁、解决什么问题，并有潜在客户或早期收入，但暂时没有完整团队的独立 B2B 服务创业者。

例如独立咨询顾问、行业研究创业者、招投标与解决方案顾问、采购与供应链顾问、小型营销策略工作室、软件实施顾问。

他们愿意付费，因为 Polis 可以让一个人获得市场研究、项目调度、专业生产、质量检查和知识沉淀能力，从“亲自完成全部工作”走向“领导一家人机协作的小公司”。

在这个场景里，系统可以从经营意图生成候选业务线、经营项目、角色和 Agent，但生成结果是**经营假设**，需要通过客户、项目和交付结果持续验证。

### 2.2 共建模式：已经经营的中小企业或专业服务机构

典型用户是创始人、合伙人、业务线负责人或交付负责人。他们已经有客户、人员、服务和历史项目，不需要 Polis 重新发明一家公司。

他们希望 Polis：

- 理解现有业务线和方法；
- 从历史项目中识别可复用模式；
- 补充虚拟研究、分析、生产和质检角色；
- 帮助项目负责人同时管理更多项目；
- 在不打断现有经营的情况下逐步提高自治程度。

在这个场景里，企业模型来自“导入事实 + 系统识别 + 负责人确认”，而不是从一句理念生成。

### 2.3 嵌入模式：成熟企业

典型买家是某个业务部门负责人，并需要 IT、安全、法务或数据负责人共同参与。

他们不购买“创建虚拟企业”，而是购买：

> 在现有组织、数据、流程和权限体系内，建立一支边界明确、可以审计的虚拟职能团队。

例如采购分析团队、售前方案团队、客户风险分析团队、行业研究团队。现有 ERP、CRM、项目系统和组织架构仍是权威来源，Polis 只在授权范围内读取、判断、执行和回写。

## 3. 首个付费用户假设

下一阶段优先验证的用户建议是：

> **已有专业能力与潜在客户、但缺少团队的独立 B2B 专业服务创业者。**

### 3.1 他现在如何工作

他一个人承担：

- 找客户和理解需求；
- 准备方案和报价；
- 研究、分析和写作；
- 管理计划和进度；
- 检查质量；
- 向客户交付；
- 保存资料和复盘经验。

问题不是他不会做，而是所有工作都依赖他本人，无法并行、无法稳定复制，也不敢在收入不确定时提前招聘完整团队。

### 3.2 他为什么购买 Polis

他期待获得的不是更多 AI 工具，而是：

- 能够承接原本超出个人容量的项目；
- 把大量资料整理和初稿生产交给可靠的虚拟角色；
- 有一个持续掌握上下文的项目调度者；
- 每次交付都经过明确的事实核查和质量检查；
- 把个人经验沉淀成公司可以复用的服务、方法和 Agent；
- 清楚知道每个项目用了多少时间、模型成本和人工精力。

### 3.3 暂不作为首个用户

- 只有模糊创业想法、没有专业能力和客户验证的人；
- 只想偶尔完成一个通用任务的个人用户；
- 需要一次性重构整个组织和 IT 系统的大型企业；
- 交易、记账、生产控制等必须由现有核心业务系统完成的场景。

## 4. 主用户的完整故事

### 故事一：从专业能力创建一家虚拟公司

一位独立采购与供应链顾问已经服务过几个客户。他知道自己擅长供应商评估、成本分析和采购策略，但不知道如何把个人能力变成一家公司。

他选择“创建虚拟公司”，告诉 Polis：

- 自己的专业能力和经验；
- 希望服务的客户；
- 可以交付的结果；
- 当前客户线索和历史材料；
- 每月预算、可投入时间和不可自动执行的事项。

Polis 不直接宣称已经替他设计好了公司，而是给出一份“筹建方案”：

- 候选业务线及判断依据；
- 建议的首个服务产品；
- 需要验证的客户与收费假设；
- 初始经营项目；
- 所需角色和 Agent；
- 人必须承担的关键职责；
- 风险、缺失信息和建议的自治等级。

顾问修改并确认后，公司进入试运营。候选业务线仍标记为“验证中”，直到出现真实客户、项目和可接受交付。

### 故事二：系统主动推动内部经营工作

公司确认“供应商风险评估”是一个待验证业务线。Polis 自动提出内部经营项目：

- 分析目标市场和竞品；
- 定义标准服务包和边界；
- 制作一份脱敏样例报告；
- 建立需求访谈提纲；
- 准备报价与项目范围模板；
- 选择十个潜在客户进行验证。

这些项目不代表市场事实，而是验证经营假设的行动。创始人决定优先级和预算，虚拟团队完成研究、材料和跟进准备，并把需要本人沟通的事项送入工作台。

### 故事三：从客户沟通形成项目

一个客户表达了供应商交付风险分析需求。项目不会因为业务线存在就自动成立。

创始人把会议记录、邮件和客户材料交给 Polis。系统：

- 提炼客户目标、约束和决策人；
- 区分已经确认与尚未确认的信息；
- 生成下一轮澄清问题；
- 草拟工作范围、里程碑、交付物和报价依据；
- 标记潜在范围风险。

在创始人与客户确认范围、时间和验收方式后，项目才从“机会/草案”变为“已立项”。商业承诺始终由人作出。

### 故事四：围绕项目组建人机团队

项目需要的不只是几个带 title 的 Agent。Polis 建议一支团队：

- 创始人担任业务负责人，负责客户关系和最终承诺；
- 项目调度角色持续掌握范围、进度、依赖和客户反馈；
- 研究、数据分析和报告角色完成专业工作；
- 独立审核角色检查事实、逻辑、格式和验收标准。

每个角色都有目标、权限、输入、输出、协作对象、质量标准和升级条件。项目调度角色发现范围冲突或信息不足时，不会继续机械拆解任务，而是暂停相关工作并要求澄清。

### 故事五：Agent 经过试岗再上岗

Polis 可以生成或组装一个“供应商研究员”候选 Agent，但它不能因为配置完成就被视为胜任。

系统使用企业历史案例或标准测试任务进行试岗：

- 能否正确使用企业方法；
- 是否引用可靠出处；
- 是否遵守输出格式；
- 是否知道何时升级给人；
- 是否在权限范围内调用工具；
- 成本、速度和稳定性是否达标。

通过后，Agent 先在限定项目、限定权限下工作。持续表现决定其升级、降级、返训或停用。

### 故事六：交付不是任务列表的终点

项目执行中，创始人主要看到：

- 今天需要他决定什么；
- 项目是否偏离范围、预算或时间；
- 哪些结论缺少证据；
- 哪些交付物已经可以审核；
- 客户反馈导致了什么变更。

系统自行处理普通任务的拆解、分派、执行、检查和有限返工。最终报告通过创始人审核后交付客户。客户反馈进入项目复盘，并更新：

- 业务线是否值得继续；
- 服务产品如何修改；
- 哪些方法可以复用；
- 哪些角色或 Agent 表现可靠；
- 下一次报价、计划和风险判断如何改进。

虚拟企业由此不是一次生成完成，而是在真实经营中逐步形成。

## 5. 用户故事清单

### 5.1 企业与经营方向

- 作为独立创业者，我希望从自己的专业能力、目标客户和经营约束得到一份可解释的公司筹建方案，从而不必从空白组织图开始。
- 作为创始人，我希望每条生成的业务线都带有依据、假设和验证状态，从而不会把模型猜测误当成经营事实。
- 作为已有企业负责人，我希望导入现有服务、项目和组织材料，从而让 Polis 理解现状而不是重建公司。
- 作为成熟企业负责人，我希望明确哪些对象来自现有权威系统、哪些可以由 Polis 修改，从而避免组织和数据冲突。

### 5.2 业务线与经营项目

- 作为创始人，我希望系统把模糊的经营方向转成可执行的验证项目，从而知道下一步应验证客户、产品还是交付能力。
- 作为业务线负责人，我希望查看目标、机会、项目、收入或成果指标以及风险，从而管理经营结果而不是管理一堆 Agent。
- 作为负责人，我希望系统根据周期、机会和指标异常提出项目草案，但重大项目必须经我确认。

### 5.3 客户机会与项目形成

- 作为项目负责人，我希望系统从会议、邮件和材料中区分客户已经确认的事实与待澄清问题，从而减少错误承诺。
- 作为创始人，我希望在客户确认范围、交付物和验收标准后再正式立项。
- 作为项目负责人，我希望每次范围变化都有来源、影响和决策记录，从而控制 ToB 项目失控风险。

### 5.4 角色、协同与调度

- 作为创始人，我希望角色说明“对什么结果负责、拥有什么权限、何时必须找人”，而不只是一个职位名称。
- 作为项目负责人，我希望有一个持续掌握项目上下文的调度角色，协调专业角色、依赖、客户反馈和返工。
- 作为审核人，我希望产出角色和审核角色相互独立，从而避免 Agent 自产自验。
- 作为成员，我希望知道某项决定由人、角色还是 Agent 负责，从而避免责任模糊。

### 5.5 Agent 任职与能力

- 作为负责人，我希望 Agent 在上岗前使用真实案例或标准样本试岗，而不是只看一段自动生成的描述。
- 作为负责人，我希望给不同 Agent 设置不同的数据、工具、预算和决策权限。
- 作为负责人，我希望看到 Agent 在真实项目中的质量、返工率、成本和升级记录，从而决定是否继续使用。
- 作为成熟企业管理员，我希望 Agent 遵守现有权限和安全策略，并且所有外部动作可追溯。

### 5.6 执行、交付与学习

- 作为创始人，我希望普通任务由系统自动规划、执行、检查和有限返工，只把关键决策和异常交给我。
- 作为项目负责人，我希望围绕里程碑和交付物查看进展，而不是只看底层 DAG 状态。
- 作为客户交付负责人，我希望每项结论带出处、版本和审核记录，从而能够向客户解释。
- 作为业务线负责人，我希望项目结果自动更新业务线假设、方法、角色和 Agent 评价，从而让公司越做越可靠。

## 6. 用户看到的产品，而不是技术后台

主用户默认看到：

1. **经营驾驶舱**：目标、业务线验证、机会、项目、风险和需要本人决定的事项；
2. **业务线**：服务、客户、假设、指标、经营项目与交付项目；
3. **机会与项目**：从客户沟通、范围确认到交付和复盘；
4. **工作台**：今天要决定、审核或亲自处理的工作；
5. **组织**：业务负责人、项目调度者、专业角色、审核角色及其任职情况；
6. **企业资产**：方法、案例、交付模板、客户许可材料和可复用能力。

Agent、模型、Skill、DAG、Token 和运行日志仍然存在，但默认收进角色、运行详情和高级治理页面。用户首先管理经营，而不是管理技术组件。

## 7. 用户价值验收

首个产品是否成立，不以“能否生成组织图”判断，而以真实经营结果判断：

- 用户是否用 Polis 完成了一个真实客户项目；
- 从需求确认到合格首稿的时间是否缩短；
- 每个项目所需人工小时是否下降；
- 同一人能够并行承担的项目数是否增加；
- 首稿通过率、返工率和客户接受率是否改善；
- 企业方法和历史经验是否在下一项目被真正复用；
- 用户是否愿意按月持续付费，而不是只体验一次生成过程。

---

## 8. 故事边界

- Polis 不替创始人凭空发现正确商业模式；它生成假设并推动验证。
- Polis 不替人建立客户信任或作出未经授权的商业承诺。
- Polis 不把项目拆解等同于项目管理；沟通、范围、依赖、变更和验收都是项目的一部分。
- Polis 不把 title 等同于角色；角色必须承载责任、权限和协同协议。
- Polis 不把一次配置等同于 Agent 能力；能力必须通过试岗和真实结果证明。
- Polis 不要求成熟企业重建已有组织和系统；成熟企业采用映射与嵌入模式。

---

## 9. 待客户验证的问题

1. 首个细分用户究竟是咨询、研究、采购供应链、营销策略还是另一类专业服务创业者？
2. 他们最愿意先交给虚拟团队的是获客准备、项目调度、研究分析、报告生产还是质量检查？
3. 什么样的交付物最容易客观验收并形成复购？
4. 用户愿意上传哪些历史材料来训练企业上下文，哪些材料不能进入系统？
5. 用户愿意授予 Polis 多大自治权，哪些动作必须逐次批准？
6. 用户愿意为何种结果付费：席位、项目容量、运行量、交付物还是节省的人力？

这些问题得到真实客户证据前，“首个用户”仍是假设，不应直接固化成不可逆的行业架构。
