# Polis 第三轮：稳定内核开发规格

> 状态：**implementation-ready v1.4**
> 日期：2026-07-22
> 适用范围：Polis 后端内核、领域包适配层及其测试；**不定义用户界面和具体行业产品**。
> 决策依据：[ADR-0015](../../../decisions/0015-stable-kernel-and-definition-bundle.md)、
> [ADR-0016](../../../decisions/0016-explicit-command-families-and-runtime-protocol-closure.md)、
> [ADR-0017](../../../decisions/0017-v3-deterministic-metadata-and-runtime-safety.md)、
> [ADR-0018](../../../decisions/0018-v3-governance-scope-and-kernel-model-ownership.md)

## 1. 这套文档解决什么问题

Polis 现有系统已经具备 Agent、Skill、Plan、TaskRun、Memory、Approval、Evaluator 和
Temporal 编排能力，但这些能力主要围绕“通用任务拆解与 Agent 管理”组织。第三轮不再先设计一个
看起来像企业的页面，而是先建立可长期复用的稳定内核：

```text
领域规则被版本化定义
→ 工作实例固定引用一组定义
→ 角色在明确范围内承担责任
→ 命令经过策略和状态机
→ 执行产生结果
→ 结果经过评价
→ 评价转化为下一条命令
→ 事件触发后续工作
```

内核不知道“采购公司”“咨询项目”或“客户成功经理”的业务含义。企业、业务线、项目等对象由
`DomainPackageVersion` 声明为不同的 `Scope` 类型，由上层产品或行业适配器解释。内核只保证这些
对象引用的定义、权限、状态、执行和审计规则可以一致运行。

## 2. 文档权威顺序

发生冲突时按以下顺序处理：

1. `CLAUDE.md`、`docs/constraints/` 中的工程与安全红线；
2. 已接受的 ADR；本轮新增决策以 ADR-0015、ADR-0016、ADR-0017、ADR-0018 为准；
3. 本目录的稳定内核开发规格；
4. `docs/design/v3/00-*`、`01-*` 等用户故事和产品讨论稿；
5. 更早的 V1/V2 设计。

本目录中，`00` 定义不可突破的边界，`01` 至 `09` 是它的展开，`10` 定义能否自主开发的 Gate，`11` 关闭
声明式语言与运行安全协议，`12` 固定三个 Definition V1 的完整 JSON 合同。任何实现若需要改变 `00` 的
不变量，必须先新增或修订 ADR，不能只在代码中变更。

## 4. 实施基线与非目标

本轮实施基线是：

- 新增稳定的定义版本、DefinitionBundle、Scope、ScopeRelation、WorkItem、范围任职与工作槽位绑定；
- 显式拆分 Definition、Scope、Work 三类 Command 信封与处理管线，不提供通用聚合信封；
- 所有工作状态变化收口到 Command Handler；
- 建立 DomainEvent、Transactional Outbox、幂等收件和对账；
- 将现有 Plan、TaskRun、ResultEnvelope、ArtifactDescriptor、Approval、RunManifest 的 ORM 声明迁入
  `kernel.models`，保持原表名、数据和旧模块兼容导入，再逐步接入 WorkItem；
- 保持现有 `/api/tasks` 和 Temporal 工作流可运行，采用“先加后删”的迁移；
- 以测试夹具证明闭环，不以某一行业页面证明闭环。

本轮明确不做：

- 不决定首个行业、客户、收费模式；
- 不设计驾驶舱、企业页面或业务线页面；
- 不让模型依据一句经营理念自动创建并发布真实业务结构；
- 不把每种状态机、表单或行业字段做成独立顶级元模型；
- 不引入微服务、Kafka、OPA 或新的工作流引擎；
- 不在迁移期删除 `task`、`plan`、`task_run` 等既有表或 API。

内核完成不等于立即替换当前产品页面。现有工作台、工作、计划、场景、技能、花名册和设置必须在
K0–K6 期间持续可用；内核稳定后再按 09 文档的 PX 阶段重新梳理产品交互，并经灰度与可用性验证后
决定是否替换旧页面。

## 5. 开发开始条件

一个内核开发任务只有同时满足下列条件才可进入实现：

- 指向本目录中的具体章节和规范条目；
- 输入、输出、错误码、事务边界和事件已经明确；
- 涉及的表有 RLS 与 A/B 租户隔离测试设计；
- 涉及 Temporal 的行为有幂等策略和 replay 测试设计；
- 迁移为 additive，且说明旧 API 在混合版本期间如何工作；
- 验收可由自动化测试判定，不依赖“看起来合理”。

此外，整个 V3 设计只有在 [10 无歧义自主开发验收](10-无歧义自主开发验收.md) 的 G1–G6 全部
通过且未决业务/架构选择为 0 时才可标记 `implementation-ready`。单个任务仍须满足该章第 5 节的
工作包输入合同。
