# 05 数据模型与迁移

## 1. 数据设计原则

- PostgreSQL 16/现有版本能力范围内实现；
- UUID 主键、TIMESTAMPTZ、snake_case；
- 组织运行态表全部 `org_id NOT NULL`，应用过滤 + RLS；
- 定义资产沿用 `owner_org_id + visibility` 的公开/私有可见性；
- 重要引用用外键，扩展配置才用 JSONB；
- JSONB 入库前必须通过 Pydantic/JSON Schema，不把校验责任交给调用者；
- append-only 事实不提供普通 UPDATE/DELETE Repository；
- migration 只走 Alembic，先加后删；
- 首轮不做 event sourcing，不引 Kafka。

## 2. 新增表

以下是逻辑 DDL 规格。实现以 SQLAlchemy model + Alembic migration 为准，约束名遵循现有 naming
convention。

### 2.1 `domain_package_version`

| 列 | 类型 | 约束 |
|---|---|---|
| id | UUID | PK |
| owner_org_id | UUID | nullable FK org |
| key | TEXT | not null |
| version | TEXT | not null |
| visibility | TEXT | check public/private |
| status | TEXT | check draft/published/deprecated |
| schema_version | INTEGER | not null default 1 |
| revision | INTEGER | not null default 1 |
| definition | JSONB | not null |
| checksum | TEXT | not null |
| created_by | UUID | nullable FK app_user |
| created_at/updated_at | TIMESTAMPTZ | not null |
| published_at | TIMESTAMPTZ | nullable |

索引：

- public 资产 `UNIQUE(key, version) WHERE owner_org_id IS NULL`；
- private 资产 `UNIQUE(owner_org_id, key, version) WHERE owner_org_id IS NOT NULL`；
- `(owner_org_id, status)`；
- checksum 普通索引。

`work_definition_version` 和 `role_definition_version` 使用同一列结构及索引，不采用单表继承，避免
不同定义类型失去明确 FK。

### 2.10 `domain_event`

`(org_id, aggregate_type, aggregate_id, aggregate_version, event_sequence)` 索引和
`(org_id, correlation_id, occurred_at)` 索引。append-only、RLS。

### 2.17 `execution_reservation`

- id/org_id、work_item_id、execution_run_id、resource_kind(org_budget/actor_capacity)、resource_key；

- start_work 按 11 §13 在同事务、稳定 advisory lock 内创建，Run 终态/cancel/对账释放；DB 对 held reservation

### 2.18 `approval_decision`

append-only 保存 id/org_id、approval_id、approval_version、family_command_id、

### 2.19 `service_identity` 与 `org_kernel_setting`

`service_identity`：id/org_id/key/status、allowed_command_families、created_at；`(org_id,key)` 唯一，service
actor_ref 必须指向此表。

`org_kernel_setting`：org_id PK、`kernel_mode=legacy/shadow/kernel`、
`governance_state=uninitialized/active`、`governance_scope_id UUID nullable`、config_version、changed_by/at。
uninitialized 时 pointer 必须为空，active 时必须非空并以 `(org_id,governance_scope_id)` 复合 FK 指向同 org、
`scope_type=org_governance`、`status=active` 的 Scope；该跨行类型/status 不变量由 DEFERRABLE constraint trigger
在事务提交前验证。模式变化
必须走受审计管理用例；运行时不读环境变量推断单个 org 模式。

`scope` 增加 partial unique：同 org 在全部历史中至多一个 `scope_type='org_governance'`。`archive_scope`
对治理 Scope 固定返回 `GOVERNANCE_SCOPE_PROTECTED`，不得通过归档后重建来替换政策事实源。数据库
trigger 拒绝该类型的 parent/external_ref、非 `kernel.governance` domain version，以及不符合 OrgPolicyV1 的
attributes。应用层先返回稳定错误，数据库约束用于阻止旁路。

`kernel_shadow_comparison`：id/org_id、source_type/source_id、legacy_checksum/kernel_checksum、
status(match/allowed_mismatch/mismatch)、reason_code、allow_owner/allow_expires_at、compared_at；唯一
`(org_id,source_type,source_id,legacy_checksum,kernel_checksum)`，RLS。存在未过期允许项之外的 mismatch 时
禁止 org 切到 kernel。

## 3. 扩展现有表

### 3.1 `plan`

新增 nullable：

- `work_item_id UUID FK work_item`；
- `definition_bundle_id UUID FK definition_bundle`；
- `checksum TEXT`；
- `schema_version INTEGER DEFAULT 1`；
- `created_by_command_id UUID`。
- `source_work_version INTEGER`；
- `source_input_revision INTEGER`；
- `source_input_checksum TEXT`；
- `registered_work_version INTEGER`；
- `accepted_work_version INTEGER`；
- `planning_snapshot_id UUID FK work_snapshot`；
- `status_reason TEXT`，支持 `superseded/input_changed`。

新链路中均为必填；暂时 nullable 兼容旧数据。PlanNode schema 新增必填 `role_slot_key`，旧计划读取时
允许为空并走 legacy router。

### 3.2 `task_run`

逻辑上作为 ExecutionRun 使用，新增：

- `work_item_id`；
- `definition_bundle_id`；
- `run_sequence INTEGER`；
- `run_manifest_checksum`；
- `logical_attempt INTEGER DEFAULT 1`；
- `version INTEGER NOT NULL DEFAULT 1`，Run 子聚合诊断/状态更新乐观锁；
- `outcome_detail JSONB`；
- `last_heartbeat_at`；
- `failure_class/failure_code`；class 使用 protocol manifest 固定 check，code 为 adapter 受控码；
- `execution_mode TEXT`，固定 `legacy/kernel`，创建后不可变；
- 扩展状态 `starting/waiting/succeeded/partial/timed_out/cancelled`。

兼容期保留 `done/needs_review`，Repository 统一映射：

```text
done → succeeded
needs_review → waiting
pending → queued
```

新代码只写新枚举；旧 API 响应适配回旧枚举。增加唯一
`(org_id,work_item_id,run_sequence) WHERE work_item_id IS NOT NULL`。

另增加 partial unique：

```sql
CREATE UNIQUE INDEX uq_task_run_one_active_work
ON task_run (org_id, work_item_id)
WHERE work_item_id IS NOT NULL
  AND status IN ('queued','starting','running','waiting');
```

`task_run` 增加 `UNIQUE(org_id,work_item_id,id)`；`work_item(org_id,id,active_run_id)` 以三列 FK 指向
`task_run(org_id,work_item_id,id)`，直接保证 active_run 属于同 org、同 WorkItem。

### 3.3 `result_envelope`

新增 `work_item_id`、`run_id`（可继续复用 task_id，但新代码写二者一致）、`result_schema_ref`、
`schema_version`、`logical_attempt`、`idempotency_key`、`checksum`、`superseded_by_id`。

`(org_id,idempotency_key)` 唯一。正文可继续在 content 兼容，但新大结果应写 Artifact，content 只留
受限摘要。

### 3.4 `artifact_descriptor`

新增 `work_item_id`、`run_id`、`kind`、`size_bytes`、`checksum`、`schema_ref`、`created_by_kind/ref`、
`status=staging/ready/quarantined/deleted`、`object_key`、`object_etag/object_version_id`、`verified_at`。
`(org_id,object_key)` 唯一；只有 ready
可被 Result/Snapshot/Evaluation 引用，应用事务与 `DEFERRABLE INITIALLY DEFERRED` constraint trigger 共同校验。
trigger 对新增/变更的 artifact FK 在事务结束前验证同 org 且 status=ready。停止使用
`meta.owner_task_id` 新写归属；兼容读取保留到 K6。两阶段提交和 orphan GC 按 11 §12。

staging 时 `size_bytes/checksum/verified_at` 可空；转 ready 时三者必须非空且 checksum 为 64 位小写 hex；
storage adapter 在事务前完成 HEAD/stream checksum，DB trigger 只验证这些持久化证据字段和引用状态，不在
数据库事务内调用 MinIO。quarantined 保存 observed checksum/size
与稳定 reason_code，但不可被业务引用。

### 3.5 `approval`

新增 03 章规定的 command_family、domain_package_version_id/work_definition_version_id/
role_definition_version_id/scope_id/work_item_id（按 family/kind 恰填一列且显式 FK）、command_type/fingerprint、
approval_purpose、version、requested_by、required_role_slots、expires_at、decision_reason、
consumed_by_command_id、command_receipt_id、resume_mode。status 固定
pending/approved/rejected/expired/revoked/consumed。保留 kind/ref_id/payload 供旧接口。

新增 `approval_schema_version default 1`：历史行保持 v1 且显式目标/version 可空，只能由旧适配器读取/决定；
所有新内核行写 v2，数据库 check 对 v2 强制显式目标、purpose、version、receipt/fingerprint。不得猜测历史
`kind/ref_id` 对应哪个 Definition 表；能通过旧外键明确映射的记录可批量升级，否则保留 v1 至 K6 清理 ADR。

### 3.6 `run_manifest`

新增 `definition_bundle_id/checksum`、`work_role_bindings`、`work_snapshot_ids`、
`evaluator_versions`、`workflow_version`、`kernel_contract_version`、`policy_provenance`、
`interpreter_versions`、`manifest_checksum`。现有 JSONB 字段继续使用，但新结构通过

### 3.7 跨组织复合外键

每个被组织运行态对象引用的表必须有 `UNIQUE(org_id,id)`。Scope、WorkItem、Run、ScopeRoleAssignment、WorkRoleBinding、
Snapshot、Approval、Result、Evaluation、Event/Outbox 的关键引用使用 `(org_id,foreign_id)` 复合 FK；不允许
只靠 Repository 注释保证同 org。对公开定义版本的引用保留普通 FK，因为其所有权语义不是运行态 org。

## 4. RLS

每个新增组织运行态表 migration 必须：

```sql
ALTER TABLE <table> ENABLE ROW LEVEL SECURITY;
ALTER TABLE <table> FORCE ROW LEVEL SECURITY;
CREATE POLICY org_isolation ON <table>
USING (
  org_id = NULLIF(current_setting('app.current_org', true), '')::uuid
)
WITH CHECK (
  org_id = NULLIF(current_setting('app.current_org', true), '')::uuid
);
```

定义资产采用：

- SELECT：`owner_org_id IS NULL AND visibility='public' OR owner_org_id=current_org`；
- INSERT/UPDATE/DELETE：`owner_org_id=current_org`；
- 平台 seed/迁移由运维角色执行；
- public 资产不能由 `polis_app` 更新。

注意现有策略可能只定义 USING；第三轮 migration 应给新表同时定义 WITH CHECK，并另开安全任务评估
旧表补齐，不能顺手破坏旧链路。

三个 DefinitionVersion 表还必须通过数据库 trigger 防止已发布内容被普通 UPDATE 原地改变。trigger
比较 OLD/NEW：当 OLD.status 为 `published/deprecated` 时，禁止改变 key、version、owner、visibility、
definition、checksum 和 schema_version；只允许 `published → deprecated` 及审计时间字段变化。
Service 层仍先返回 `DEFINITION_IMMUTABLE`，DB trigger 是绕过 service 时的兜底。

### 4.1 既有核心 ORM 声明迁移规则

K0 只迁移代码所有权，不迁移数据：

1. 把 `Plan/TaskRun`、`RunManifest/Approval`、`ResultEnvelope/ArtifactDescriptor` 的原 SQLAlchemy class 声明
   移入 `polis.modules.kernel.models`；`__tablename__`、列、索引、约束、sequence 与现有 Alembic revision 不变；
2. `planner.models`、`observability.models`、`memory.models` 从 kernel import 并 re-export 原 class 名称；禁止
   第二次继承 `Base` 或调用 `Table(...)`；
3. `polis.db.models` 显式 import `kernel_models`，旧模块仅注册仍由其拥有的模型；同一 metadata table key
   必须只有一个 Table object；
4. kernel models 对旧表只使用字符串 FK/relationship 目标或 kernel-owned class，不 import 旧模块；
5. K0 migration 必须为空：`alembic check` 不得产生 rename/drop/create，数据库 schema fingerprint 前后相同；
6. characterization tests 证明旧 import path 与新 path 得到同一 class object，旧 repository/API 行为不变。

## 6. 数据回填

历史 Task 不能凭空推断正确 WorkDefinition。回填采用明确默认定义：

1. seed `core.legacy_task` Domain/Work/Role definitions；
2. 为每个 org 编译 legacy bundle；
3. 每个 Task 创建一个 `scope_type=legacy_workspace` 的默认 Scope（org 一个）；
4. 创建 WorkItem，inputs 保存 task_id/goal/inputs 引用，状态从 Task 派生；
5. 关联已有 Plan/TaskRun/Result；
6. 无法解释的状态标记 `legacy_unknown`，不自动完成/失败；
7. 回填每批带 migration_run_id，可重入、可核对数量；
8. 回填不生成虚假的用户 Command/Event，只写 `work.migrated` 系统事件并保留 source refs。

## 7. 回滚

- 每阶段 migration 的 downgrade 只在未产生生产数据时执行；
- 已产生新数据时应用回滚通过 feature flag 停用新写入，不物理 drop；
- K3 以后 downgrade 前必须验证无新 WorkItem-only 数据；
- Outbox/Command 表的 downgrade 绝不能在有 pending/processing 消息时执行；
- workflow 回滚需先通过 Temporal replay；
- 回滚步骤和数据核对 SQL 写入每个 migration docstring。

## 8. 数据完整性验收 SQL/断言

自动化集成测试必须证明：

- WorkItem 与 Scope/bundle 的 org 一致；
- WorkItem 的 Scope domain package 与 bundle domain package 一致；
- bundle 的所有定义版本存在且 checksum 匹配；
- active_run_id 指向同 org、同 WorkItem 的 Run；queued/running 与 execution_gate/pause waiting 对应非终态
  Run，evaluating/quality-review waiting 可指向 succeeded/partial/failed/timed_out Run，其他 Work 状态必须为空；
- 每个 org 的 kernel mode 与新 WorkItem/Run 模式符合 09 的创建规则；
- WorkItem version 与最新 WorkTransition.work_version 一致；
- 每个实际修改聚合的成功命令存在对应 transition/audit、event 和 receipt；02 明确的四类 no_state_change
  仅有 succeeded receipt；
- published outbox 对应 event 或明确的 system message；
- consumed Approval 的 command fingerprint 匹配；
- current Result 的 evaluation 引用同一 run/attempt；
- 跨 org FK 即使 UUID 已知也不能通过 service 写入；
- DB 复合 FK 直接拒绝关键运行对象的跨 org 连接；
- nullable snapshot/evaluation 去重在 NULL 场景仍生效；
- outbox system message 即使无 event_id 也由 dedupe_key 去重；
- scheduled command 同 schedule/version 最多派发一个 receipt；
- Approval 每个 version 恰有零或一条 decision，所有状态变化来自对应 family Command；
- held reservation 汇总不超过 actor/org 上限，terminal Run 无未释放 reservation；
- succeeded Result/EffectReceipt 只引用 ready Artifact；staging 不可进入 Context；
- uncertain 外部 effect 不被自动重试；
- shadow materialize 失败时 legacy Task 仍成功且 pending/failed link 可被修复；
- 公开 definition 对其他 org 只读，private 不可见。
