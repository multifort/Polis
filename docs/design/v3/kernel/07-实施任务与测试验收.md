# 07 实施任务与测试验收

## 1. 实施原则

本路线以“每个任务 1–3 天、可单独验收”为粒度。不得先把所有表建完再一次性接业务；每阶段都必须
形成一个可运行的纵切。阶段 K1–K2 不依赖具体行业或前端。

平台部署级功能开关固定为：

```text
kernel_definitions_enabled
kernel_work_items_enabled
kernel_legacy_task_shadow_enabled
kernel_execution_enabled
kernel_evaluation_loop_enabled
kernel_reconciliation_enabled
```

开关集中在 `Settings`，生产默认关闭尚未验收的阶段。组织业务模式不使用这些开关推断，必须读取
`org_kernel_setting.kernel_mode`。禁止散落环境变量读取。

K0 是无数据库变化的代码所有权前置阶段；K1–K6 是功能增量阶段。所有阶段同时受
[09 渐进演化与产品连续性](09-渐进演化与产品连续性.md) 约束：当前页面关键
操作回归不通过时，不得因为内核测试通过而开启下一阶段。

## 2. K0：核心 ORM 所有权与兼容基线

### K0-T1 旧行为特征测试

产出：Plan/TaskRun/RunManifest/Approval/ResultEnvelope/ArtifactDescriptor 的 import、metadata、Repository、
API 和 Alembic schema fingerprint 特征测试。

验收：测试必须先在移动前通过；记录六个 class 的表名、关键约束和旧 import path，`alembic check` 无差异。

### K0-T2 ORM 声明迁入 kernel

产出：六个 class 唯一声明于 `kernel.models`；三个旧 models 模块兼容 re-export；`polis.db.models` 单次注册；
架构依赖测试。

验收：新旧 import path 使用 `is` 得到同一 class；`Base.metadata.tables` 不重复；kernel 不 import
planner/runtime/memory/observability；旧测试全集通过；数据库产生 0 条 DDL diff。

K0 Gate：代码所有权已收口但数据库、API、Temporal 和用户行为完全不变。失败时不得进入 K1。

## 3. K1：定义、Bundle 与 WorkItem 骨架

### K1-T1 定义 Pydantic Schema

产出：

- 三个 Definition V1 discriminated schemas；
- SchemaProfileV1、ConditionExprV1、MappingExprV1、StateMachine、PolicyBinding、RoleSlot、ExecutionProfile；
- canonical JSON 与 checksum 工具；
- semver/key/schema validation。
- 将 `jsonschema` 声明为后端直接依赖并锁定版本；不得依赖现有间接依赖偶然存在。

验收：

- 合法 fixture 往返序列化 checksum 稳定；
- RFC 8785 官方/项目测试向量 checksum 一致；PathV1 missing/null/type/date-time 语义逐项通过；
- 任意代码、远程 `$ref`、未知 effect 被拒绝；
- 定义错误包含 JSON path 和稳定 code；
- domain 层无框架依赖。

### K1-T2 定义与 Bundle 数据表

产出：models、Alembic、RLS、Repository、`polis.db.models` 注册。

验收：

- upgrade/downgrade 在空库通过；
- public/private 唯一索引正确处理 NULL owner；
- org A 看不到 org B private，二者可读 public 但不可修改；
- published 定义通过 service 和 DB 防护不可变。

### K1-T3 Definition Compiler

产出：引用解析、兼容性检查、静态状态机/触发器校验、bundle 去重。

验收：

- 相同输入重复编译返回同 bundle；
- role slot 缺版本、key 不匹配、未发布/不可见版本失败；
- checksum 任一输入变化即变化；
- deprecated 版本不能编译新 bundle，旧 bundle 仍可读。
- child dependency key 缺失/多余、key 不匹配和递归环确定性失败，错误含 dependency path；
- dependency 深度/节点上限在边界值通过，超一确定性失败；
- DefinitionValidator/Compiler 不持 session、不 commit；架构测试证明 publish/compile 只能经 CommandService。

### K1-T4 Scope、治理政策、Relation、责任绑定与 WorkItem

产出：Scope/ScopeRelation、ScopeRoleAssignment、WorkRoleBinding、WorkItem、service identity、mode 数据表、
RLS、`kernel.governance`/`kernel.governance_owner` seed、OrgPolicyV1、唯一治理 Scope、governance bootstrap、
create/query service，不实现业务 transition。

验收：

- inputs 按固定 bundle schema 校验；
- scope parent 类型和 org 一致；
- Scope assignment authority 只能缩小，Work binding 必须有有效责任链；
- WorkItem bundle 创建后不可改；
- A/B 租户隔离覆盖所有新表。
- uninitialized 仅 org owner/admin 可完成治理 owner bootstrap；pointer/Scope/assignment 同 org 且激活原子；
  active 后同一身份也必须经任职授权；第二治理 Scope 与非法 policy 由 service/DB 双重拒绝。

K1 Gate：能够发布通用 fixture、编译包含子工作依赖的 bundle，创建 Scope/Relation/范围任职/WorkItem/
工作 binding，并完整读回版本关系。

## 4. K2：命令、状态、事件和 Outbox

### K2-T1 纯状态机

产出：transition lookup、guard evaluator、effect 生成、双层状态验证。

验收：

- 所有 fixture transition 正反路径；
- 不可达/歧义 transition 在发布时失败；
- guard 不访问白名单外字段；
- 终态不可转出；
- 单元测试分支覆盖率 ≥ 90%。

### K2-T2 Policy 与 Approval Domain

产出：Policy 合并、权限交集、command fingerprint、Approval version/decision 表、三个 family 的显式审批命令。

验收：

- deny 优先级最高；
- 内容/version 任一变化使 Approval 无效；
- 自批、过期、重复消费被拒绝；
- 职责分离和 scope 继承正确。
- decide/expire/revoke 只经所属 family，锁顺序与原命令 resume 并发无死锁、无双消费。

### K2-T3 Command Handler

产出：三类显式信封与三个 handler、durable receipt、家族锁顺序、expected version、transition/event 原子提交。

验收：

- 100 个相同幂等请求只有一次 version 增长；
- 两个不同命令争用同 version 只有一个成功；
- 失败命令不写 transition/event；
- 成功命令必有 receipt + transition + event；
- API/service/worker 使用命令所属家族的唯一 handler；
- command discriminator 在三个家族间无重复，代码中不存在 Generic/aggregate target 写入口；
- T1 后崩溃、T2 前/中崩溃均可由 lease 恢复且不产生部分业务效果。

### K2-T4 Outbox 与 Trigger

产出：outbox worker、lease/retry/dead letter、trigger receipt、durable scheduler、聚合顺序与循环保护。

验收：

- worker 崩溃后 lease 可回收；
- 重复 Event 只生成一个 Command；
- causation depth 超限被 suppress 并审计；
- 单条 poison message 不阻塞后续消息；
- 不持 DB 锁执行网络调用。
- 事件 gap 不越序消费，补齐后恢复；schedule dispatch 与 receipt 创建同事务且只派发一次，receipt 失败
  不把 schedule 改回 pending。

### K2-T5 Kernel API

产出：定义、bundle、scope relation/assignment、work binding/command、approval resume、timeline API 和统一错误。

验收：

- OpenAPI response model 完整；
- Idempotency-Key/If-Match 缺失有稳定错误；
- 错误符合统一 envelope；
- 跨租户访问返回 404/403 且无对象信息泄漏；
- timeline 不返回敏感 payload。

K2 Gate：08 中 Fixture A 从创建到 `plan_ready` 的非执行路径全部通过，Definition、Scope、Work 写入已
分别收口到三个类型化 handler。

## 5. K3：旧 Task/Plan 适配

### K3-T1 Legacy 定义和 link

产出：`core.legacy_task` seed、link 表、可重入回填。

验收：回填前后 Task 数量一致；重复执行不重复；未知状态不伪造完成；每条有 migrated event。

### K3-T2 Task API shadow/cutover adapter

产出：feature flag、旧响应映射、适配器。

验收：现有 task/plan/attachment 回归测试不改断言仍通过；shadow materialize 失败时旧请求仍成功、pending
link 可修复且不反写 legacy；新 WorkItem 状态可对账；关闭 flag 行为不变。

### K3-T3 禁止旁路写

产出：架构测试或静态扫描，列出允许的 migration/legacy adapter 例外。

验收：

- API/Temporal/Evaluator 不直接 assign WorkItem state；
- K3 后旧状态更新仅存在 kernel adapter/repository；
- 发现 `update_*_status` 新调用即测试失败。

K3 Gate：旧产品可继续使用；shadow 中旧 Task 仍是真相且 WorkItem 可对账，kernel 中 WorkItem 是唯一真相；
混合版本可回滚。

## 6. K4：Temporal 执行接入

### K4-T1 WorkSnapshot、Plan/Run/Manifest V2 与资源保留

产出：additive migration、role_slot 路由、manifest checksum。

验收：

- required slot 未任职无法出有效 Plan；
- planning/input revision 固定，输入变化使 Plan/Approval 失效；
- RunManifest 固定所有执行、policy、contract 与 interpreter 版本；
- 旧 Plan 仍可 legacy read；
- 同 WorkItem 不能创建两个 active Run。
- 并发 start 通过 ExecutionReservation 不超卖 actor capacity/组织预算，terminal/cancel 后幂等释放。

### K4-T2 Outbox → Temporal Adapter

产出：幂等启动、状态回写 Command、cancel/signal。

验收：

- Temporal 离线时 API 仍返回 queued，不丢请求；
- 恢复后自动启动；
- WorkflowAlreadyStarted 不产生重复 Run；
- started/terminal 回写重复安全。

### K4-T3 Kernel Workflow 或版本化旧 Workflow

产出：引用式输入、Activity 幂等、ExternalEffectReceipt、heartbeat、两类人审等待、bounded correction。

验收：

- replay 旧 history；
- 串行、并行、retry、human gate、cancel、timeout；
- provider 支持幂等键时 Activity 重试不重复副作用；不支持且结果歧义时进入 uncertain 并停止自动重试；
- Workflow 内无 DB/网络/now/random。

### K4-T4 对账器

产出：04 章故障扫描与修复 Command。

验收：为故障矩阵逐项构造不一致；一轮或两轮内修复；重复对账无额外 version 增长。

K4 Gate：杀死 API/Worker/Temporal Worker 的演练中不丢工作、不重复外部效果、最终可收敛。

## 7. K5：结果、评价、审批闭环

### K5-T1 Artifact/Result V2

验收：work/run/node 归属明确；schema 校验；MinIO 前缀隔离；staging→verify→ready 与崩溃恢复通过；Result
只引用 ready；orphan GC 不删仍被引用对象；相同 activity key 返回原 Result。

### K5-T3 四结果 Trigger

验收：指纹、TTL、职责分离、一次消费、自动/手动 resume；现有 approval tests 继续通过。

## 10. 强制验收用例

### AC-01 定义固定

已发布 bundle 创建 WorkItem 后，发布 1.1.0；旧 WorkItem 仍读取 1.0.0 且 checksum 不变。

### AC-02 幂等

同 key 并发发送 20 次 `submit_for_review`，仅一个 transition/event，所有成功响应 command result 一致。

### AC-03 乐观锁

两个 actor 基于 version 3 发不同命令，只允许一个成功；另一条返回最新 version。

### AC-04 精确审批

金额/收件人/目标状态任一变化，旧 Approval 不可使用。

### AC-05 责任路由

组织有能力相同 Agent A/B，但只有 A 任职 reviewer，Plan 必须选 A；A suspend 后不允许偷偷选 B。

### AC-06 评价解耦

写 Evaluation 后、Trigger 前 WorkItem 不变化；Trigger Command 后才变化。

### AC-07 Outbox 恢复

提交 start_work 后关闭 Temporal；Run 保持 queued。恢复后启动一次且 active Run 唯一。

### AC-08 Activity 重试

支持幂等键的外部副作用成功但 Activity 响应丢失，重试使用同键并返回原结果；不支持幂等键的歧义超时
进入 uncertain，断言不会自动第二次调用。

### AC-09 对账

Temporal terminal/DB running 的人工构造状态被对账器通过 command 修复，并有 audit/event。

### AC-10 租户

org B 使用 A 的 WorkItem、bundle、ScopeRoleAssignment、WorkRoleBinding、Approval UUID 均不可读取或关联。

### AC-11 Trigger 环

构造闭环定义时发布失败；绕过静态检测的间接环在运行深度上限处停止。

### AC-12 旧链路

未开启 feature flag 时现有测试行为完全不变；开启后旧响应兼容且生成新 timeline。

### AC-13 当前产品页面

公司开办、工作台、Task 创建/出图/运行/历史/导出、Plan 详情、人审、场景、Skill、花名册和设置
在 legacy 与 kernel 两种模式下关键 E2E 均通过；旧深链接继续可访问。

### AC-14 Command 家族

三个 schema union 的 command_type 无交集；Definition 命令不能携带 work target，Scope 命令不能进入
Work handler，任何 family mismatch 在加锁前返回稳定错误。

### AC-15 Scope 关系与继承

创建、重复创建、反向无向关系、非法类型/cardinality、跨 org 关系均有确定结果；nearest/merge/none
继承按 fixture 解析，暂停范围任职后新 binding/Run 被拒绝。

### AC-16 责任与委派

accountable A 将允许委派的节点交给 Agent B，Manifest 同时记录 A 的责任 binding 与 B 的执行身份；撤销
委派后新 Run 不选 B；职责分离冲突拒绝。

### AC-17 输入与计划快照

相同输入 no-op 不增 version；实际变化增 input_revision，使旧 Plan 和 Approval 失效；旧 plan_ready 迟到
返回 stale 且不覆盖新计划。

### AC-18 审批续接恢复

分别在 receipt awaiting、Approval approved、resume outbox published 后杀进程；manual/automatic 均最终只
消费一次 Approval、只增长一次聚合 version，version 变化时确定性 APPROVAL_STALE。

### AC-19 持久调度

调度器多实例、lease 过期、misfire fire_once/skip、cancel/dispatch 竞争均最多产生一个业务 receipt，
重启后不丢计划。

### AC-20 子工作版本固定

父 bundle 固定 child 1.0 后发布 child 2.0；触发仍创建 1.0。依赖缺失/环在编译期失败，运行时不得查询
latest。

### AC-21 迁移模式与数据库约束

legacy/shadow/kernel 模式下创建的 Task/Work/Run mode 可审计；模式切换只影响新实例；active Run 唯一、
nullable 去重、outbox dedupe 和跨 org 复合 FK 在数据库层直接拒绝冲突。

### AC-22 声明语言确定性

同一 definition 在两个进程得到相同 JCS/checksum；JSONPath、未知 Schema keyword、隐式类型转换、缺失的
required mapping 均以稳定错误拒绝；null 与 missing 断言符合 11 §4。

### AC-23 Approval/Schedule 状态所有权

直接 Repository 状态写在架构测试失败；三个 family 的 decide/expire/revoke 并发只产生一个 Approval
version/decision；schedule dispatch 原子创建一个 receipt，receipt rejected 后 schedule 仍 dispatched。

### AC-24 Run 失败与两类人审

failed/timed_out 均进入 evaluating 后由 Evaluation 决定终态；execution_gate 保持 Workflow waiting 并 signal，
quality_review 时 Workflow 已 terminal 且不 signal；unexpected cancel 进入评价。

### AC-25 Artifact 崩溃恢复

分别在 staging commit、上传完成、checksum verify、ready+Result commit 边界杀进程；最终只有一个 ready
Artifact/Result，未验证对象不可读，orphan 可安全回收。

### AC-26 治理启动与 Definition 旁路

非 owner/admin 不能 bootstrap；治理 owner 激活后 governance active 且不可普通回退；静态/架构测试证明
publish/compile 没有 DefinitionService/Repository 旁路。

### AC-27 容量保留

对 max_active=1 的 actor 并发启动 20 个 WorkItem，只有一个 held reservation/queued Run；terminal/cancel/
lease recovery 后容量精确释放，重复释放 no-op。

### AC-28 Shadow 非阻塞

注入 WorkItem 物化失败，legacy Task API 仍按旧合同成功；pending/failed link 与 outbox 可观测并最终修复，
修复前后均不反向更新 legacy 状态。

### AC-29 治理政策唯一来源

同 org 并发创建两个 org_governance Scope 只有一个成功；激活事务同时写 pointer/state；修改 policy 后
Scope version/checksum 唯一增长，旧 Run/Approval/Reservation 仍保留旧 snapshot；缺失/跨 org/归档 pointer
确定性返回 GOVERNANCE_SCOPE_MISSING，不能使用默认值继续执行。

### AC-30 核心 ORM 唯一所有权

六个核心 class 只在 kernel.models 声明；旧路径 re-export 与新路径对象相同；metadata table key 唯一；
Alembic 0 DDL diff；静态依赖测试证明 kernel 不反向 import planner/runtime/memory/observability。

## 11. 非功能门槛

首版目标，不作为过早微服务化理由：

- 三类 Command Handler 的单次处理（不含外部调用）P95 < 200 ms；
- WorkItem GET P95 < 150 ms（本地/测试数据库基线）；
- 1 万 pending outbox 下领取查询有索引且 EXPLAIN 不全表扫描；
- 相同 WorkItem 50 并发写不死锁，只有合法版本成功；
- 对账扫描分页，不一次加载全组织；
- timeline cursor 分页，默认 50、最大 200；
- 事件 payload ≤ 64 KiB；
- 核心 kernel domain/application 覆盖率 ≥ 80%，状态机/policy ≥ 90%。

## 12. 每个 PR 的 DoD

- 对应 Kx-Ty 和本文验收；
- migration 可升级，空数据可 downgrade；
- 新业务表有 RLS 和 A/B 隔离测试；
- 新 API 有 response_model、错误码和 OpenAPI；
- 当前产品合同 E2E 通过，页面、状态文案和旧深链接无阻断回归；
- 新 Temporal 变化有 replay；
- 新状态效果只经 Command Handler；
- 测试、ruff、mypy、secret/security 门禁通过；
- 文档/ADR/技术债同步；
- 没有无负责人的未完成占位项、跳过测试或兼容旁路。

## 13. 自主开发时的决策规则

后续 Codex 只有在 [10 无歧义自主开发验收](10-无歧义自主开发验收.md) G1–G6 通过后才按本文自主实现。
每个工作包必须满足该章第 5 节；发现两个实现都符合规范时标记 `blocked-by-spec` 并先补文档。

可自行决定：

- 文件内私有函数拆分、命名和测试 fixture 组织；
- 等价索引顺序和 Repository 查询实现；
- 在不改变协议的前提下复用现有 helper；
- 分批 PR 的具体边界。

必须停止并请求决策：

- 要增加第四个顶级元模型；
- 要修改十六条内核不变量；
- 要合并三类 Command 信封或增加第四个 Command 家族；
- 要让 published 定义或 WorkItem bundle 可原地更新；
- 要直接删除/重命名旧表或破坏旧 API；
- 要引入 Kafka、OPA、微服务或替换 Temporal；
- 要改变危险动作/审批的安全默认；
- 发现现有生产数据无法按 K3 回填且不同处理会改变业务事实。
