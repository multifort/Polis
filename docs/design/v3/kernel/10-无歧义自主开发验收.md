# 10 无歧义自主开发验收

## 1. 目标与判定

本章定义 V3 设计是否可以进入自主开发。通过条件不是“文档足够详细”，而是：

> 开发者或 Codex 在不补充业务规则、不选择架构分支、不猜测异常语义的前提下，能够从任一验收
> 用例追溯到唯一的 API、Command schema、处理管线、状态转换、数据约束、事件和恢复动作。

只有本章全部设计 Gate 通过，README 状态才能从 `design-candidate` 改为 `implementation-ready`。实现中
若发现两个都符合文档的行为，必须先修订规格；不得自行选择后继续开发。

## 2. 规范语言

- **必须/不得**：强制要求，测试需覆盖；
- **应当**：默认要求，只能由已接受 ADR 例外；
- **可以**：不影响协议的实现自由；
- 带“建议、考虑、可选、待定、视情况”的句子不得出现在 normative 表、算法、schema 和验收条件中；
- 示例与实现必须一致；示例若省略字段，必须标注 `// presentation-only`，不得作为可解析 fixture。

冲突顺序为：`CLAUDE.md/constraints > accepted ADR > 00 > 11/protocol-manifest > 01–10 > 产品讨论稿 > 旧设计`。同级章节
冲突视为验收失败，不由实现者解释。

## 3. 唯一命令路由表

| family | 信封 | 目标与锁 | version 字段 | 定义固定字段 | handler | 事件前缀 |
|---|---|---|---|---|---|---|
| definition | `DefinitionCommandEnvelope` | adapter 预分配 `definition_version_id`；锁对应版本行 | `expected_definition_revision` | 无 bundle；发布/编译读取显式版本 IDs | `DefinitionCommandService` | `definition.*` |
| scope | `ScopeCommandEnvelope` | adapter 预分配 `scope_id`；锁 scope | `expected_scope_version` | `domain_package_version_id` | `ScopeCommandService` | `scope.*` |
| work | `WorkCommandEnvelope` | adapter 预分配 `work_item_id`；锁 work | `expected_work_version` | `definition_bundle_id` | `WorkCommandService` | `work.*` |

`command_type` 必须在且只在一个家族目录出现。CI 收集三个 discriminated union 的 discriminator，
若重复或未登记则失败。代码中禁止定义 `GenericCommandEnvelope` 或接收 `aggregate_type + aggregate_id`
的公共写入口。

表中“目标与锁”描述普通业务命令；family 内 Approval/Schedule 子聚合命令严格使用 11 §8/§9 的例外锁顺序，
但不改变信封 family、显式目标或 handler。

## 5. 自主开发工作包输入合同

每个 1–3 天实现任务必须包含：

1. 唯一的规范章节与 command_type 列表；
2. 请求/响应 Pydantic 类型及字段必填性；
3. Repository 读写集、锁顺序和单事务边界；
4. 前置状态、guards、policy、approval 和版本冲突规则；
5. 成功后的字段差异、聚合 version、Transition、Event、Outbox；
6. 所有稳定错误码及 HTTP/application 映射；
7. 重试、重复、崩溃、迟到和并发竞争结果；
8. RLS、复合 FK、唯一约束和不可变约束；
9. 旧 API/旧数据行为、feature flag、回退路径；
10. Given/When/Then 自动验收和 fixture。

缺少任一项时任务状态为 `blocked-by-spec`，不是 `ready`。

## 6. 允许实现者决定的内容

自主开发只可自行决定下列不改变外部协议的事项：

- 同一模块内私有函数名和文件拆分；
- SQLAlchemy 查询写法，只要锁、隔离与结果等价；
- 指标/日志的额外字段，只要不含敏感信息；
- 测试辅助函数和 fixture builder 的组织；
- 达到或优于规格硬上限的批次大小默认值，但生产默认值必须进入 Settings。

下列事项必须停止并补 ADR/规格：新增顶级定义根、改变 Command 家族、改变真相源、放宽策略、改变
状态机语义、破坏 API/event schema、删除兼容路径、引入外部 broker/策略/工作流引擎。

## 7. 机器可验证 Gate

### G1 结构完整

- README 链接 00–11、protocol manifest 与 ADR-0015/0016/0017/0018 均有效；
- 三个 Command union 的 discriminator 无重复；
- 每个 command_type 都有 result/event 或明确 `no_state_change` 声明；
- 每个状态枚举在 schema、DDL、API 和测试 fixture 一致。

### G2 无歧义文本

对 normative 段落扫描 `TODO|TBD|待定|二选一|可按需要|视情况|建议使用|或由实现决定` 必须为零；
允许在“背景/被否决选项/未来范围”中出现，但必须有非规范标识。

### G3 协议与数据库

- 所有 org 运行态 FK 对关键聚合使用 `(org_id, id)` 复合 FK；
- nullable 去重使用 `NULLS NOT DISTINCT` 或规范中给出的生成键；
- 所有活动 Run、receipt、schedule、approval 竞争有数据库唯一/CAS 保护；
- Event/Outbox 与聚合状态在同一事务提交。
- Definition/Approval/Schedule 不存在 Repository/application 写旁路；
- org policy 只从唯一 governance Scope 解析，revision/checksum 与 Scope version/JCS 一致；
- 六个既有核心 ORM class 只由 kernel.models 声明，旧路径只 re-export，Alembic 0 意外 DDL；
- protocol manifest 与 Pydantic discriminator、数据库 check 和 fixture 枚举完全一致。

### G4 行为闭环

设计冻结时，08 章所有 fixture 必须给出成功、拒绝、重复和恢复断言，07 章 AC-01 至 AC-30 必须映射
到明确任务与测试文件。实现验收时再要求这些测试全部通过；依赖 sleep、人工浏览或直接改业务表触发
后续步骤的测试不算通过。

### G5 连续性

设计冻结时必须明确同一兼容合同测试如何在 `legacy` 与 `kernel` 运行、`shadow` 差异报告结构、无破坏
回退和持久化模式审计；实现完成后对应测试与演练必须通过。

### G6 声明与运行安全

- SchemaProfileV1、PathV1、ConditionExprV1、MappingExprV1 和 JCS 有跨进程测试向量；
- failed/timed_out/unexpected cancel、execution_gate/quality_review 各有唯一转换与对账断言；
- 外部副作用 uncertain、Artifact 两阶段恢复和 ExecutionReservation 并发均有故障注入；
- governance bootstrap、Definition 写入口和 shadow 非阻塞由架构/集成测试证明。

冻结时必须在 PR/设计验收记录填写：

AC-01..AC-30 映射：PASS/FAIL
Fixture A..R 可执行：PASS/FAIL
歧义词扫描：PASS/FAIL
跨章节链接与冲突检查：PASS/FAIL
legacy/shadow/kernel 连续性：PASS/FAIL
声明式协议/运行安全：PASS/FAIL

设计通过只表示可以无歧义实现，不表示代码已经验收。代码完成仍必须满足 CLAUDE.md 的 DoD、测试、
OpenAPI、Alembic、Temporal replay、安全与人工评审。
