# 11 声明式协议与运行安全闭合

## 1. 地位与边界

本章是 V3 的规范性协议，不增加第四个定义聚合根、第四类 Command 或外部基础设施。它关闭
Schema、条件、映射、编译依赖、审批、调度、外部副作用、制品提交、容量和启动治理的解释空间。
三个 Definition 及其内嵌 Profile/Effect/Trigger 的完整字段合同见 12 章。
若 01–10 章的旧示例与本章冲突，以本章和
[协议清单](protocol-manifest.yaml) 为准，并必须在同一设计变更中修正旧示例。

## 2. 规范 JSON 与摘要

所有 definition、bundle、Command payload、condition/mapping 和 fingerprint 都使用下列唯一算法：

1. 先按对应 Pydantic/SchemaProfileV1 模型验证，拒绝未知字段；
2. 字符串保持原 Unicode code point，不做隐式 trim、大小写或 Unicode normalization；
3. 数字必须是 JSON number，拒绝 NaN、Infinity、负零和超出 IEEE-754 安全整数范围的整数；金额必须用
   明确的整数最小单位字段，例如 `budget_cents`；
4. 按 RFC 8785 JSON Canonicalization Scheme 输出 UTF-8 字节；
5. checksum 为这些字节的 SHA-256 小写十六进制，共 64 字符。

同一协议对象不得存在第二套“键排序 JSON”实现。Python 实现必须以固定测试向量验证 RFC 8785；
checksum 算法改变时必须提升 kernel contract major。

## 3. SchemaProfileV1

Definition 中所有业务 Schema 使用 JSON Schema Draft 2020-12 的以下闭集。未列出的 keyword 一律
`DEFINITION_INVALID/SCHEMA_KEYWORD_UNSUPPORTED`。

允许：

```text
$schema, $id, $defs, $ref, title, description,
type, enum, const, required, properties, additionalProperties,
items, minItems, maxItems, uniqueItems,
minimum, maximum, exclusiveMinimum, exclusiveMaximum, multipleOf,
minLength, maxLength, minProperties, maxProperties,
allOf, anyOf, oneOf, not, format
```

限制：

- `$schema` 缺省或必须等于 `https://json-schema.org/draft/2020-12/schema`；
- `$ref` 只允许以 `#/$defs/` 开头的本定义内引用；禁止远程、相对文件、`$dynamicRef` 与递归引用；
- `format` 只支持 `uuid`、`date-time`、`date`、`email`、`uri`，并作为 assertion 校验；
- `date-time` 必须是 RFC 3339 且带时区；入库规范化为 UTC `Z`；
- object 必须显式给出 `additionalProperties`；发布时不得依赖库默认值；
- JSON Schema `default` 首版禁止；Definition 自身的结构默认值由对应 Pydantic V1 模型在 checksum 前显式
  展开，业务输入/结果永不因 Schema annotation 被静默补值；
- schema 深度不超过 32，单 schema 节点不超过 2,000，`$defs` 不超过 128，规范 JSON 不超过 256 KiB；
- 首版禁止 `default`、`pattern`、`patternProperties`、`contains`、`dependentSchemas`、`propertyNames`、
  `unevaluatedProperties`、`unevaluatedItems`、`if/then/else`、`content*` 和自定义 keyword。

## 4. PathV1 与值语义

所有 condition 和 mapping 路径只使用 RFC 6901 JSON Pointer，名称为 `PathV1`。根路径是空字符串
`""`；分隔符是 `/`；token 中 `~` 写作 `~0`、`/` 写作 `~1`。禁止 `$`、点号路径、通配符、切片、
过滤器和负数组下标。

求值规则固定为：

- 路径不存在返回内部值 `MISSING`；它与 JSON `null` 不相等；
- `exists` 对存在且值为 null 也返回 true；`not_exists` 只匹配 `MISSING`；
- `eq/ne/in/not_in` 使用 JSON 类型严格深比较，不做字符串/数字/布尔转换；object 比较忽略键顺序；
- `lt/lte/gt/gte` 两侧必须同为 number，或同为已验证并规范化的 RFC 3339 date-time 字符串；否则
  返回 `CONDITION_TYPE_MISMATCH`，不得当作 false；
- `in/not_in` 的右值必须是数组，按严格深比较判断成员；
- `all/any` 按声明顺序短路，空 `all=true`，空 `any=false`；`not` 恰有一个子表达式；
- condition 访问 `MISSING` 而 op 不是 `exists/not_exists/input_exists/artifact_exists` 时，结果为 false；
- evaluator 不读取系统当前时间；涉及时间的比较必须读取 envelope 的 `requested_at` 或事件 `occurred_at`。

允许的上下文根固定为 `/command`、`/work`、`/scope`、`/event`、`/bundle`、`/actor`、`/capacity`、
`/approval`。每条 rule 类型再以 Pydantic 明确可读子路径；未在白名单的路径发布时拒绝。

## 5. ConditionExprV1

Condition 使用带 discriminator 的 JSON AST，不接受表达式字符串：

```text
CompareExpr: {op:eq|ne|lt|lte|gt|gte, path:PathV1, value:JsonValue}
SetExpr:     {op:in|not_in, path:PathV1, value:JsonArray}
ExistsExpr:  {op:exists|not_exists|input_exists|artifact_exists, path:PathV1}
LogicalExpr: {op:all|any, conditions:[ConditionExprV1...]}
NotExpr:     {op:not, condition:ConditionExprV1}
SlotExpr:    {op:role_slot_filled, role_slot_key:string}
EvalExpr:    {op:evaluation_outcome_is, value:pass|rework|human_review|fail}
EventExpr:   {op:event_field_matches, path:PathV1, value:JsonValue}
```

所有对象 `extra=forbid`。AST 深度不超过 16，总节点不超过 256。策略、guard 和 Trigger 共用同一解释器和
同一测试向量；`event_field_matches` 只允许 Trigger，`evaluation_outcome_is` 只允许 Work guard/Trigger。

## 6. MappingExprV1

`payload_mapping`、`input_mapping` 只接受以下 AST；不得使用“字段名对应路径字符串”的隐式语法：

```text
Const:    {op:const, value:JsonValue}
From:     {op:from, path:PathV1, required:true|false}
Coalesce: {op:coalesce, values:[MappingExprV1, ...]}
Object:   {op:object, fields:{target_field:MappingExprV1, ...}}
Array:    {op:array, items:[MappingExprV1, ...]}
```

求值从不可变 Event/Work/Result 安全上下文读取。`from(required=true)` 遇到 MISSING 返回
`MAPPING_SOURCE_MISSING`；`required=false` 返回 MISSING，所在 object 字段被省略、array 中则拒绝，避免
数组位置漂移。`coalesce` 返回第一个既非 MISSING 也非 null 的值，全空则返回 MISSING。输出先做规范 JSON，
再用目标 Command payload schema 或 child Work input_schema 校验；失败进入 Trigger/Schedule dead letter，
不得提交部分 payload。

MappingExprV1 的 AST 深度、节点和规范 JSON 大小上限固定为 12 §6；ConditionExprV1 同时增加 64 KiB
规范 JSON 大小上限。超限在 Definition publish 前拒绝，不进入运行时解释器。

## 7. 子工作依赖与确定性编译

WorkDefinition 的 `child_dependencies` 是内嵌值对象：

```json
{
  "dependency_key": "remediation_v1",
  "work_definition_key": "core.remediation",
  "allowed_scope_types": ["work_group"]
}
```

`compile_definition_bundle` payload 固定为：

```json
{
  "domain_package_version_id": "uuid",
  "work_definition_version_id": "uuid",
  "role_versions_by_slot": {"owner": "uuid", "reviewer": "uuid"},
  "child_dependencies_by_key": {
    "remediation_v1": {
      "domain_package_version_id": "uuid",
      "work_definition_version_id": "uuid",
      "role_versions_by_slot": {"owner": "uuid"},
      "child_dependencies_by_key": {}
    }
  }
}
```

规则：

- payload key 集必须与 WorkDefinition 的 slots/dependencies 完全相等，不允许缺失或多余；
- dependency 深度最大 16、闭包节点最大 128；超限返回 `BUNDLE_DEPENDENCY_LIMIT_EXCEEDED`；
- compiler 先用 payload 和普通读取收集完整 definition 闭包、执行 DFS；随后把闭包中所有 version row 按
  `definition_kind(domain,work,role) + UUID` 全局排序并一次性 `FOR SHARE` 锁定，重新读取/校验 published、
  可见性、checksum 与 key，再自底向上编译/插入 bundle；并发相同 checksum 由唯一键返回既有 bundle；
- DFS 以 `(domain_package_version_id, work_definition_version_id)` 为节点检测环；发现灰色节点返回
  `BUNDLE_DEPENDENCY_CYCLE`，并给出 dependency key 路径；
- child work key、scope type、role key 和可见性必须匹配声明；
- `definition_bundle_dependency` 保存真实 FK，compiled JSON 只保存 dependency key、bundle ID 与 checksum；
- 运行时只按父 bundle 的 dependency row 解析，不接收 child bundle ID，不查询 latest，不替换 deprecated
  bundle；数据损坏时确定性 dead letter。

## 8. Approval 是受控子聚合

Approval 不是第四类 Command family。状态变化分别由原 family 的显式命令完成：

```text
decide_definition_approval / expire_definition_approval / revoke_definition_approval
decide_scope_approval      / expire_scope_approval      / revoke_scope_approval
decide_work_approval       / expire_work_approval       / revoke_work_approval
```

每个 payload 固定含 `approval_id`、`expected_approval_version`；decide 另含
`decision=approve|reject` 与可空 `reason_note`，expire/revoke 含受控 `reason_code`。信封仍携带原 family 的
显式目标和定义固定字段。Approval 有独立 `version`，每次实际状态变化 +1；决定写 append-only
`approval_decision`。Approval Event 使用 `aggregate_type=approval`、`aggregate_id=approval_id`、
`aggregate_version=approval.version`，事件名仍是 `{family}.approval_decided/expired/revoked/consumed`。

锁顺序：

1. decide/expire/revoke 只 `FOR UPDATE` Approval；目标只做普通一致性读取，不锁目标；
2. approved decision 提交时写 resume outbox；reject/expire/revoke 同事务终结原 receipt；
3. 原命令续接先锁 family 目标，再锁 Approval；重新校验目标 version、指纹、身份、任职、策略和 guard；
4. 成功时目标、Approval consumed、receipt succeeded、event/outbox 同事务；失效则 Approval revoked、
   receipt rejected(`APPROVAL_STALE`) 同事务。

decision 语义固定：`reject` 在审批者仍有权限时允许关闭 pending Approval，即使目标已变化；`approve` 若
目标 version/fingerprint 已变化，则同事务把 Approval→revoked、原 receipt→rejected，决定命令 receipt
记 succeeded/response_code=`APPROVAL_STALE` 并写 revoked event，HTTP 返回 409 与当前 Approval；它不得先
写 approved。`expected_approval_version` 不匹配则决定命令 rejected(`APPROVAL_VERSION_CONFLICT`)且不改任何行。

`approval_purpose` 固定为 `command_policy/execution_gate/quality_review`：command policy 与 quality review
等待数据库续接；execution gate 属于仍在运行的 Temporal human gate，可在审批决定后由 outbox signal。
expire handler 只以 PostgreSQL `transaction_timestamp()` 判断 `expires_at <= now`；客户端 requested_at 和
worker 本地时钟不参与到期判定。相同规则用于 schedule due/misfire 与 reservation lease。

Definition family 的数据库目标不使用多态 UUID：`domain_package_version_id/work_definition_version_id/
role_definition_version_id` 三列恰填一列。Pydantic 可使用 `definition_kind + definition_version_id` 作为
输入 union，但 Repository 必须映射到对应显式 FK。

## 9. ScheduledCommand 是受控子聚合

Schedule 只属于 Scope 或 Work family，创建于产生 `schedule_command` effect 的同一业务事务：

- Scope schedule 明确保存 `scope_id + domain_package_version_id`；
- Work schedule 明确保存 `work_item_id + definition_bundle_id`；
- schedule 有独立 `version` 和 `status=pending/dispatched/cancelled/failed`；取消使用原 family 的
  `cancel_scope_schedule/cancel_work_schedule`，不增长目标 version；
- dispatcher 锁 schedule，以 CAS `pending→dispatched`，在同一 PostgreSQL 事务插入目标
  CommandReceipt、写 `{family}.schedule_dispatched` Event 和 outbox；`dispatched_command_id` 不可变；
- `dispatched` 表示意图已交给 durable receipt，不表示业务命令成功。业务结果从 receipt 查询，receipt
  失败不得把 schedule 改回 pending 或再次派发；
- dispatcher 只有在 stored template checksum/schema 已损坏、无法创建合法 receipt 的不可恢复情形，才在
  单事务 CAS `pending→failed` 并写 `{family}.schedule_failed`；可重试数据库/进程错误必须回滚保持 pending；
- misfire `skip` 由 dispatcher 在到期领取事务 CAS `pending→cancelled`，reason_code=`MISFIRE_SKIPPED`，并写
  `{family}.schedule_cancelled`；不创建 receipt。任何 schedule 变化都通过 family handler 或 dispatcher 的
  上述唯一协议完成。

Schedule Event 使用 `aggregate_type=scheduled_command` 和 schedule.version；dispatcher 永不锁目标，取消
先锁目标再锁 schedule，因此不存在 schedule→target 的反向加锁。

## 10. Execution、评价与 human gate

Run outcome 的唯一映射：

| Run outcome | Work 处理 |
|---|---|
| succeeded/partial/failed/timed_out | `record_run_outcome` 令 Work `evaluating`，保留 active_run_id，并请求 Evaluation |
| cancelled（Work 已先提交 cancel intent） | 只归档 Run/诊断；Work 保持 cancelled、active_run_id=null |
| unexpected external cancel | Run 记 `failure_class=unexpected_cancel`；Work 进入 evaluating，由 Evaluation 决定 fail/human_review |

`record_evaluation` 仍保持 evaluating。Trigger 随后发 complete/rework/request_human_review/fail；只有这些
命令同时确定 lifecycle 与 execution 终态。失败和超时不得绕过 Evaluation。

两类人审不得混淆：

- `execution_gate`：Temporal Run 仍为 waiting，Work waiting，Approval 通过后 outbox signal；
- `quality_review`：Temporal Run 已终态且 Workflow 已结束，Work evaluating 或 quality-review waiting，审批
  通过后续接精确 complete intent；不得向已结束 Workflow 发 signal。

对账规则：DB Run terminal 而 Temporal running 时，若存在同 Run 的 pending execution_gate，则不取消也不
反转 Run，标 critical invariant violation 等待人工修复；其他情况请求幂等 cancel。quality_review 时
Temporal 必须 terminal。

## 11. 外部副作用协议

`ExternalEffectReceipt` 是运行态协议对象，不是定义根。唯一业务键为
`(org_id, execution_run_id, node_id, logical_attempt, operation)`，至少包含 provider/tool、idempotency_key、
request_checksum、status、provider_ref、response_artifact_id、version、lease、时间与安全错误码。状态：

```text
prepared → in_flight → succeeded
                     → failed
                     → uncertain
```

协议：

1. Activity 先创建/读取 prepared receipt；相同键不同 request checksum 拒绝；
2. provider 支持幂等键时传递该键，超时可用同键重试；
3. provider 不支持幂等键时，只有在能证明请求未离开本进程时可重试；连接中断、超时或进程在调用后崩溃
   一律记 `uncertain`，停止自动重试并进入对账/人工处置；
4. 成功响应先按 §12 提交 Artifact，再将 effect receipt 与 response_artifact_id 标为 succeeded；
5. `failed` 只用于无歧义的拒绝/未执行结果；不得用本地 receipt 宣称外部系统一定 at-most-once。

worker 必须在调用前提交 `in_flight + lease`。lease 过期时：provider_supports_idempotency=true 可由同键重领
重试；false 则 reconciler CAS 为 uncertain，不执行 provider call。prepared 只有在确认尚未 dispatch 时才可
重领。人工处置通过受审计 `reconcile_work` 记录 provider_ref/证据后决定 succeeded 或 failed，不得直接改表。

因此 K-I04 的精确承诺是：数据库业务效果至多一次；外部系统只有在 provider 接受幂等键时可承诺
至多一次，否则承诺“不盲目重试且不确定性可见”。

## 12. Artifact 两阶段提交

`ArtifactDescriptor.status=staging/ready/quarantined/deleted`。Result、WorkSnapshot、Evaluation 和 Context
只能引用 ready Artifact。

1. 在 DB 创建 staging descriptor 与确定对象 key
   `{org_id}/{work_item_id}/{artifact_id}/{checksum-or-upload-id}`；
2. 上传对象；以服务端 checksum/size 或下载校验确认内容；
3. 单一 DB 事务将 descriptor 改为 ready，并创建引用它的 Result/EffectReceipt 更新；
4. 上传前/中失败保留 staging 供重试；校验不匹配转 quarantined；
5. 上传成功但 DB 提交失败时，同 artifact id/key 重试步骤 3；超过保留期的无引用 staging/orphan 由 GC
   先 HEAD/checksum 后删除，并写审计；
6. 任何路径都不得创建引用不存在、未验证或非 ready 对象的成功 Result。

对象写必须 create-only（S3 `If-None-Match: *` 或等价条件写），descriptor 保存 ETag/version_id。若 key 已存在，
只有 HEAD/校验 checksum 与 size 完全相同才视为幂等成功；不同则 descriptor quarantined 并返回
`ARTIFACT_CHECKSUM_MISMATCH`，禁止覆盖原对象。读取高风险证据时可按策略复验 checksum。

MinIO 与 PostgreSQL 不宣称原子事务；上述状态机和幂等 key 是唯一恢复协议。

## 13. 责任、容量与预算保留

RoleSlot 的 `responsibility_kind` 是唯一责任语义。WorkRoleBinding 不接受 `binding_purpose`；它保存
`role_slot_key/responsible_assignment_id`，可冗余 `responsibility_kind_snapshot`，但必须等于 bundle slot。
`executor_kind/ref` 仅表示委派执行者，不能改变责任。

`ExecutionReservation` 是运行态资源保留记录：org/work/run、`resource_kind=org_budget/actor_capacity`、
resource_key；actor_capacity 行必填 binding/actor/capacity_units 且 budget_cents=0，org_budget 行每个 Run
恰有一条、binding/actor 为空且保存该 Run 全部 budget_cents。状态为
`held/released/expired`，并有 version、lease/expires_at。`start_work` 在同一事务中先锁 WorkItem，再取得
组织预算与各 actor 的 PostgreSQL transaction advisory lock，校验 held reservation 汇总并创建新保留与
queued Run；唯一 `(org_id,execution_run_id,resource_kind,resource_key)`。Run 终态、cancel 或启动租约过期时，通过 Work Command/对账
释放；重复释放为显式 no-op。并发 start 不得超过 RoleDefinition/组织硬上限。

advisory key 固定为 SHA-256 字节串前 8 字节按 big-endian signed bigint：组织预算输入
`polis:v3:budget:{org_id}`，actor 输入 `polis:v3:capacity:{org_id}:{actor_kind}:{actor_ref}`。先取 budget key，
actor key 按 bigint 升序取得；hash 冲突只会额外串行，不影响正确性。组织上限只从治理 Scope 的
`OrgPolicyV1` 读取，启动时固定 scope version/policy checksum；角色上限来自 RoleDefinition。Reservation
保存两类 limit/checksum snapshot 供审计。

每个 Run 的 `org_budget` reservation 同时承担组织并发槽位：在 budget advisory lock 内统计该 org 的 held
org_budget rows，若已达到 `max_concurrent_runs`，无论 `budget_enforcement` 为何都返回
`CAPACITY_EXHAUSTED`。预算额度再按 observe/deny/require_approval 处理；observe 超额必须写 AuditLog 和
Reservation snapshot，不能静默放行。这样不增加第三种 reservation resource_kind。

## 14. 组织治理启动

复用现有 org membership/owner/admin，不创建新的企业或用户模型。平台先以 service identity seed
`kernel.governance` DomainPackageVersion 和 `kernel.governance_owner` RoleDefinitionVersion；它们仍属于
既有三个定义根，不新增顶级元模型。`org_kernel_setting.governance_state` 为 `uninitialized/active`：

- uninitialized 时只有现有 org owner/admin 能创建该 org 的 private definition、唯一 org_governance Scope、首批
  ScopeRoleAssignment；仍必须经过身份、硬策略、receipt、audit、event 和三类显式 Command；
- bootstrap 只把“已有 Scope 任职”检查替换为 org owner/admin 检查，不跳过 tenant、schema、policy、
  version 或幂等；
- org_governance Scope 使用 seed domain，parent/external_ref 为空，attributes 必须是 01 §5.1.1 的完整
  `OrgPolicyV1`；同 org 第二个治理 Scope 由数据库唯一约束拒绝，既有治理 Scope 不允许归档；
- 治理 owner assignment 激活事务同时验证 policy、写 `governance_scope_id` 并把 governance_state CAS 为
  active；指针、Scope 与 assignment 必须同 org；
- active 后 Definition/Scope 写必须经正常任职授权，不提供普通 API 回到 uninitialized；
- `update_scope` 是政策唯一修改入口，相同 checksum 是 no-op；成功修改后所有新决策使用新 revision，既有
  Plan/Run/Approval/Reservation 继续使用已固定 snapshot；
- platform public seed 使用受控 platform service identity，不借用任一 org 管理员。

## 15. Definition 写入口与 shadow

Definition 写入只允许 `DefinitionCommandService`。`DefinitionValidator` 和 `DefinitionCompiler` 是无 session、
不 commit、不持久化的纯内部组件；`KernelQueryService` 只读。不得提供能 publish/compile 并自行 commit 的
`DefinitionService` 旁路。

shadow 无条件非阻塞旧业务：legacy Task 先按原事务成功，并在同一事务 outbox 写
`kernel.shadow.materialize.v1` 及 pending link；worker 幂等创建 WorkItem/link。失败只重试/告警，不把已成功
legacy 请求改为失败。shadow WorkItem 不驱动 legacy 状态。kernel 模式才以 WorkItem Command 为真相并允许
原子失败。

## 16. 实现拒绝条件

出现以下任一情况，任务必须 `blocked-by-spec`：实现新增 JSONPath/表达式字符串、运行时 latest 定义解析、
通用 Approval/Schedule 信封、直接写 Approval/Schedule 状态、失败 Run 绕过 Evaluation、无幂等 provider 的
盲重试、Result 引用非 ready Artifact、Definition 写旁路、shadow 失败反向影响 legacy，或实现枚举与
`protocol-manifest.yaml` 不一致。
