# 03 责任、权限、策略与审批

## 1. 治理目标

角色不是 title，也不是用于挑选 Agent 的标签。角色是一个可审计责任席位，回答五个问题：

1. 谁对结果负责；
2. 在哪个 Scope 和 WorkItem 内负责；
3. 可以发出哪些命令、读取哪些数据、使用哪些工具；
4. 什么情况下必须交给其他角色或人；
5. 结果以什么标准被验收。

治理链必须先确定责任，再选择执行者；不能让能力路由反向决定责任。

## 2. Actor 与身份解析

首版支持：

| `actor_kind` | `actor_ref` 指向 | 状态来源 |
|---|---|---|
| `human` | `app_user.id` | 用户 active 且为 org member |
| `agent` | `agent.id` | Agent active，且固定有效 AgentVersion |
| `service` | `service_identity.id` UUID | 服务身份表与轮换凭证 |

ActorResolver 必须校验：

- actor 属于当前 org 或是允许的系统服务；
- actor 状态有效；
- Agent 使用的 version 已发布且与 RunManifest 一致；
- HTTP 请求中的 human actor 必须来自 JWT，不能信任 body；
- Temporal/Outbox 的 service actor 必须来自内部认证上下文，不能由公开 API 冒充。

不建立统一 Actor 超表，但 `service_identity(id, org_id, key, status, allowed_command_families)` 是明确运行
对象，保证 `actor_ref` 始终为 UUID。审计记录保留 `kind/ref/display_snapshot`，即使主体后续停用，历史
仍可解释。

## 3. RoleDefinition、RoleSlot、ScopeRoleAssignment 与 WorkRoleBinding

三者职责不同：

```text
RoleDefinitionVersion = 这个责任席位通常负责什么、最大能做什么
RoleSlot              = 这一种 WorkDefinition 需要哪些席位
ScopeRoleAssignment    = 具体谁在何范围、何时间承担该角色
WorkRoleBinding        = 本次工作哪个槽位由哪项任职负责、由谁执行
```

RoleSlot 可以进一步约束 RoleDefinition：

- `required`；
- `min_assignments/max_assignments`；
- `allowed_actor_kinds`；
- `separation_of_duties_from`；
- `required_for_commands`；
- `assignment_policy_keys`。

ScopeRoleAssignment/WorkRoleBinding 只能缩小 RoleDefinition 权限，不能扩大。扩大权限必须发布新 RoleDefinitionVersion 或
改变上层组织授权。

RoleSlot 的 `responsibility_kind` 是唯一责任类型；WorkRoleBinding 不接受调用方提交的
`binding_purpose`。Binding 中的 `responsibility_kind_snapshot` 只作不可变审计投影，必须与 bundle slot 相等。

### 3.1 任职与绑定解析顺序

对命令或 Plan 节点解析有效任职：

1. 读取 WorkItem 的 active WorkRoleBinding；
2. 对每项 binding 校验 responsible ScopeRoleAssignment 是否 active、在有效期内；
3. 精确 scope 优先；slot 为 `nearest` 时只取最近祖先层，为 `merge` 时合并各层，为 `none` 时禁止继承；
4. 校验 RoleDefinition 版本与 bundle slot 完全一致；
5. 校验职责分离、min/max 和 accountable 链；
6. `executor_ref` 存在时校验 slot execution_policy 与委派链，否则责任承担者也是执行候选；
7. required slot 无有效 binding 返回 `ASSIGNMENT_MISSING`，不得临时在全组织 Agent 中绕过。

定义可允许 `assignment_mode=elastic`。此时 Composer 可以提出候选 Agent，但必须先创建
`pending` ScopeRoleAssignment，经 Scope Command 激活并通过 Work Command 建立 binding 后才可执行。

## 4. 权限交集

一次实际动作的有效权限是以下集合的交集：

```text
平台硬上限
∩ 组织政策
∩ DomainPackage 默认政策
∩ RoleDefinition authority
∩ ScopeRoleAssignment authority_constraints
∩ WorkRoleBinding delegation constraints
∩ WorkDefinition/transition policy
∩ AgentVersion authority（若执行者为 Agent）
∩ Skill/Tool 自身声明
∩ 本次凭证 scope 与有效期
```

任意层缺省时采用最小权限，不采用“未配置即允许”。

其中 `ScopeRoleAssignment.authority_constraints` 的“缺省”只表示不在该层继续收窄，绝不产生角色定义
之外的权限。其精确结构与子集算法以 01 §5.3 `AuthorityConstraintsV1` 为准；有效权限按集合交集、
风险等级最小值和预算最小值计算，禁止 truthy/falsy 或字段存在性猜测。

三个 family 的任职入口固定为：Definition family 只解析治理 Scope 的 active assignment；Scope family
创建新 Scope 时解析治理 Scope assignment，修改既有 Scope 时允许“目标 Scope 有效 assignment”或“治理
Scope assignment”进入后续权限交集；Work family 只解析 WorkRoleBinding。治理 owner 不会因此自动获得
业务 Work 的执行/审批责任，除非另有有效工作 binding。该规则不使用 Scope 父子继承猜测控制面权限。

### 4.1 平台硬策略

不能被领域包覆盖：

- 跨 org 读写拒绝；
- 密钥、凭证、系统 prompt 不写入事件、记忆和日志；
- 删除、支付、外发、发布、权限提升等危险副作用至少需要显式政策；
- 未发布 Skill/AgentVersion 不得进入正式 Run；
- 定义中的任意代码与远程 Schema 引用拒绝；
- 已终结 WorkItem 不得重新执行，只能基于其创建新 WorkItem；
- Approval 不能批准自身、不能超出审批者权限；
- 评价模型不能成为审批者；
- object storage key 必须限制在 org/work 前缀。

### 4.2 领域策略

首版策略是声明式规则列表：

```json
{
  "key": "high_risk_start",
  "applies_to_commands": ["start_work"],
  "when": [
    {"op": "in", "path": "/work/inputs/risk_level", "value": ["high", "critical"]}
  ],
  "decision": "require_approval",
  "required_role_slots": ["owner"],
  "approval_ttl_seconds": 86400,
  "reason_code": "HIGH_RISK_START"
}
```

策略只能读取白名单上下文：

- actor 与 assignment 摘要；
- WorkItem 当前快照；
- Command 类型和 payload 的非敏感字段；
- bundle 中的定义；
- 当前预算/容量的数值快照；
- 已存在 Approval 摘要。

策略不得调用 LLM。需要模型判断的风险分类必须先产生一个有 provenance 的 Result，再由确定性策略
读取已确认的分类值。

## 5. PolicyDecision 合并

每条适用策略返回：

```json
{
  "decision": "require_approval",
  "policy_key": "high_risk_start",
  "reason_code": "HIGH_RISK_START",
  "required_role_slots": ["owner"],
  "constraints": {"expires_in_seconds": 86400}
}
```

合并规则：

1. 任一 `deny` → 最终 deny；
2. 否则任一 `require_approval` → require_approval，并合并所有 required roles；
3. 否则全部 allow → allow；
4. 无适用策略且命令不是明确安全的只读/内部记录命令 → deny；
5. 返回所有匹配策略和 reason codes，不能只保留最后一个。

每次决策写入 Transition 的 `policy_snapshot`，必须包含：组织 policy revision/checksum、DomainPackage policy
checksum、平台硬策略版本、policy interpreter version、kernel contract version 和匹配规则 key。高风险
拒绝和所有审批请求还要写 AuditLog。缺少任一 provenance 字段时不得得到 allow。

组织 policy 只能从 `org_kernel_setting.governance_scope_id` 指向的 active `org_governance` Scope 读取。
revision 是该 Scope version，checksum 是 `attributes.kernel_policy` 的 JCS checksum。读取不到唯一 Scope、
pointer 与 org 不一致、Scope 已归档或 checksum 不匹配时返回 `GOVERNANCE_SCOPE_MISSING`，不得回退到环境
变量、DomainPackage 默认值或空政策。

## 6. 精确审批

现有 Approval 仅用 `kind/ref_id`，无法证明“批准的究竟是哪一条命令”。第三轮审批必须绑定命令指纹。

### 6.1 Command fingerprint

计算内容：

```text
SHA-256(
  org_id
  + command_family
  + family_specific_target_id
  + family_specific_definition_id
  + command_type
  + canonical_json(payload 中的审批相关字段)
  + expected_target_state
  + expected_target_version
)
```

审批相关字段由 transition 声明；不能把易变时间戳等无关字段算入，也不能漏掉金额、收件人、目标
系统等危险字段。

Approval 至少保存：

| 字段 | 说明 |
|---|---|
| `command_family` | definition/scope/work |
| `domain_package_version_id/work_definition_version_id/role_definition_version_id/scope_id/work_item_id` | 按 family/kind 恰填一个业务目标；均为显式 FK |
| `command_type` | 请求动作 |
| `command_fingerprint` | 精确内容摘要 |
| `requested_by_kind/ref` | 请求者 |
| `required_role_slots` | 谁可审批 |
| `status` | `pending/approved/rejected/expired/revoked/consumed` |
| `approval_purpose` | `command_policy/execution_gate/quality_review` |
| `version` | Approval 乐观锁；每次实际状态变化 +1 |
| `expires_at` | 到期自动失效 |
| `decided_by/at` | 决策者 |
| `decision_reason` | 可选说明 |
| `consumed_by_command_id` | 一次性使用记录 |
| `command_receipt_id` | 指向可恢复的原命令 |
| `payload_snapshot` | 只保存非敏感审批摘要 |

### 6.2 审批规则

- 审批者必须有 required ScopeRoleAssignment/WorkRoleBinding；
- `requested_by` 默认不能等于 `decided_by`；领域明确允许自批的低风险动作除外；
- Approval 状态机固定为 `pending → approved → consumed`，或 `pending → rejected/expired/revoked`；
- approved Approval 只允许被一个指纹完全相同的命令消费；
- WorkItem version 或目标内容变化后，旧审批不能使用；
- approve 时已 stale 直接按 11 §8 撤销并终结原 receipt；reject 可关闭仍 pending 的旧 intent；
- 使用审批和状态转换必须在同一事务内；
- 到期任务必须构造对应 family 的 `expire_*_approval` 命令，不能直接 UPDATE；
- 拒绝后重新请求必须生成新 Approval，不复活旧记录。

### 6.3 原命令续接

需要审批时不要求客户端保存和重构命令。`CommandReceipt` 保存：family、command_type、target IDs、
expected version、definition/bundle ID、canonical command hash、command_payload、resume_mode 和 approval_id，状态
改为 `awaiting_approval`。

- `manual`：批准后 receipt 保持 awaiting_approval；调用方对 `/approvals/{id}/resume` 发 POST。服务读取
  receipt，以 CAS 将同一 receipt 改回 processing 并继续同一 command_id，不接收新的业务 payload；
- `automatic`：批准事务写 `kernel.approval.resume.v1` outbox，consumer 执行同一续接；
- 续接重新执行身份有效性、任职、硬策略、当前策略、guard、version 与 fingerprint 校验；Approval 不是
  绕过最新政策的凭证；
- 成功时目标变化、Approval `approved→consumed`、原 receipt `→succeeded` 同事务提交；
- version/input/危险字段已变化时 Approval `→revoked`、receipt `→rejected(APPROVAL_STALE)`，不得自动创建
  替代命令；
- 重复 resume 返回原 succeeded/rejected 结果，不产生第二次 version 增长。

Approval 决定、过期、撤销分别使用 `decide/expire/revoke_{definition|scope|work}_approval`，不得存在
`decide_approval` 通用命令。决定事务只锁 Approval；原命令续接锁 family 目标后再锁 Approval。每次决定
追加 `approval_decision`，事件 aggregate 为 Approval，完整状态所有权和锁顺序见 11 §8。

resume outbox 的 dedupe key 是 `approval-resume:{approval_id}:1`；它只唤醒原 receipt，不创建第二条业务
CommandReceipt。receipt 保存 `resume_attempt` 供恢复审计。

评价的 `human_review` 使用 02 §5 的固定派生规则：Approval 审批的是精确 `complete_work` intent，而不是
含糊的“同意评价”。拒绝不消费 complete intent，触发 bundle 固定的 rework/fail 动作。自由审批意见仅作
证据，不参与确定命令类型。

## 7. 执行路由

PlanNode 第三轮新增：

```json
{
  "id": "review",
  "type": "agent",
  "deps": ["draft"],
  "role_slot_key": "reviewer",
  "required_capabilities": ["review.quality_check"],
  "expected_output_schema_ref": "#/result_schema",
  "dangerous": false
}
```

路由顺序：

1. 获取 role slot 的有效 WorkRoleBindings 与 accountable 链；
2. 排除职责分离冲突、暂停、过期和没有可保留容量/预算的承担者；
3. 对 agent executor，解析固定 AgentVersion 的已发布 Skill 所派生能力；
4. 确认 required capabilities 全满足；
5. 再按健康度、历史质量、当前容量、成本和亲和度评分；
6. 生成责任 binding、执行者选择及委派理由，写入 RunManifest；
7. 无候选时返回 `ASSIGNMENT_MISSING` 或 `CAPABILITY_UNSATISFIED`，不得静默生成并直接上岗。

候选评分不等于占用容量。`start_work` 必须在同一事务创建 11 §13 的 held ExecutionReservation；并发请求
先锁 WorkItem，再按 11 §13 的稳定 advisory key 取 org budget、actor capacity 锁。只有保留成功才创建
queued Run；Run 终态/cancel/租约过期
通过命令或对账释放，保证 max_active_work_items 与组织预算在并发下不超卖。

Composer 可以组装候选角色/Agent，但其输出是 Proposal。只有经过任职命令后才进入候选集合。

## 8. 上下文与数据权限

ContextBuilder 必须以 WorkItem + WorkRoleBinding 为输入，不能只用 org_id + capability：

```text
可读 Artifact
∩ WorkItem/Scope 数据边界
∩ RoleDefinition data_scopes
∩ ScopeRoleAssignment constraints
∩ WorkRoleBinding delegation constraints
∩ AgentVersion/Skill 最小需要
```

Memory 读取规则：

- work/session memory：只允许当前 WorkItem/Run；
- role memory：命名空间必须与 role definition/assignment 对齐；
- org memory：仅在角色 data_scopes 允许时读取；
- 来自其他 WorkItem 的事实必须有 provenance 与置信度；
- WorkSnapshot 记录“读了什么和为什么”，不记录凭证明文。

## 9. 职责分离

至少支持：

- 产出者不能审批自己的危险动作；
- producer 与 reviewer 不得是同一 actor；
- 新 Skill 的作者不能作为唯一发布审批者；
- 高风险结果需要两个不同 responsible ScopeRoleAssignment 或 human actor；
- service actor 不得占据要求 human 的 slot。

编译 bundle 时检查静态冲突；激活 ScopeRoleAssignment/WorkRoleBinding 时检查实际 actor 冲突；执行命令时再次检查，以应对任职
状态变化。

## 10. 审计

以下行为必须写 AuditLog，并关联 correlation_id：

- 定义发布、废弃和 bundle 编译；
- ScopeRoleAssignment 激活、暂停、结束，以及 WorkRoleBinding 绑定、委派、撤销；
- Policy deny 与 require_approval；
- Approval 创建、决定、过期、消费；
- 高风险命令成功或失败；
- Run 启动、取消、强制恢复；
- 人工覆盖评价或触发器抑制；
- 管理员数据修复。

AuditLog 不代替 DomainEvent：前者服务安全审计，可跨聚合；后者服务业务闭环，只记录已发生领域事实。

## 11. 首次治理启动

组织首次进入内核时复用已有 org owner/admin。`governance_state=uninitialized` 期间只允许该身份通过三类
显式 Command 创建 private definition、唯一 `org_governance` Scope 和首批任职；除“已有任职”检查外，其余
身份、租户、硬策略、schema、receipt、审计和事件均不跳过。激活治理 Scope 的 owner assignment 时，事务
必须同时校验 `OrgPolicyV1`、写入 `governance_scope_id` 并 CAS `governance_state` 为 active；任一步失败全部
回滚。之后所有 Definition/Scope 写恢复正常任职授权，普通 API 不允许回退。平台 public seed 使用
platform service identity。完整规则见 11 §14。
