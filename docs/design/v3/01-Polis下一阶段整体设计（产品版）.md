# Polis 下一阶段整体设计（产品版）

> 状态：讨论稿 v0.1
> 日期：2026-07-14
> 目的：定义 Polis 从“多 Agent 任务执行与管理平台”走向“可运行虚拟企业”的产品模型、系统边界和实施路线。

## 1. 设计结论

### 1.1 当前判断

当前 Polis 已初步具备虚拟企业的**执行内核**：组织与 Agent、能力路由、计划 DAG、运行时、记忆、审批、护栏、成本、可观测、场景与资产沉淀。

但当前产品主循环仍然是：

```text
用户输入任务 → 系统拆解 → Agent 执行 → 返回结果 → 用户管理 Agent/Skill/模型
```

它更接近“受治理的多 Agent 企业任务平台”，还不是围绕经营目标持续运行的虚拟企业。

### 1.2 下一阶段产品总纲

**Polis 根据用户所处的经营阶段，以创生、共建或嵌入的方式，建立可由人和 Agent 共同运行的企业经营单元。**

产品从五个经营主体组织：

1. 企业；
2. 业务线；
3. 项目；
4. 任务；
5. 角色。

Agent 不是第六种组织主体，而是可以被角色任用、替换和评估的执行资源。行业经营蓝图不是企业实例，而是生成、识别和校验五个主体的元模型。

### 1.3 核心产品原则

1. **客户成熟度优先**：生成、导入、映射和自治方式由用户与企业成熟度决定。
2. **生成不等于事实**：系统生成的业务线、项目、角色和 Agent 必须标记依据、置信度、状态和验证方式。
3. **经营主线优先**：用户管理目标、客户、项目、交付与风险；Agent、模型、Skill 和 DAG 下沉为执行细节。
4. **角色承载责任**：角色必须包含目标、权限、协作、质量和升级规则，不是 title 或能力标签。
5. **Agent 先任职后上岗**：生成或组装只是候选，必须经过试岗、评测、限定任职和持续评价。
6. **人领导、Agent 运营**：人负责方向、关系、承诺和高风险决策；Agent 负责可授权的持续执行。
7. **不替代权威系统**：成熟企业的 CRM、ERP、财务、合同和项目系统仍是事实来源，Polis 是经营智能与行动层。

## 2. 客户分层与产品模式

### 2.1 三种模式

| 模式 | 目标用户 | 用户购买的产品 | 初始化方式 | 默认自治水平 |
|---|---|---|---|---|
| **创生 Genesis** | 独立运营者、专业创业者、AI 原生小团队 | 从专业能力和经营意图启动一家公司 | 系统生成候选结构，人确认并验证 | 中高，低风险工作可自动运行 |
| **共建 Augment** | 已经营的中小企业、专业服务机构 | 为现有业务线补充虚拟团队和经营能力 | 导入历史事实，系统识别与补强 | 中，逐项目/角色授权 |
| **嵌入 Embed** | 成熟企业、集团业务单元 | 在现有体系内建立受治理的虚拟职能团队 | 连接权威系统并映射现状 | 低起步，按风险逐步开放 |

三种模式不是三套底层系统，而是同一领域模型的三种对象来源、权威关系和自治策略。

### 2.2 首个市场假设

建议优先验证：

> 已有专业能力与潜在客户、但缺少完整团队的独立 B2B 专业服务创业者。

推荐先选择一个更窄的垂直场景，例如采购与供应链咨询、行业研究或 B2B 解决方案顾问，并限定一个主要项目类型和一种核心交付物。

购买者是创业者本人；价值是提高项目容量、缩短交付周期、降低过早招聘成本并沉淀可复用企业能力。

### 2.3 不同时期的商业定位

| 阶段 | 对外定位 |
|---|---|
| 首个产品 | 面向特定专业服务创业者的 AI 虚拟公司运营系统 |
| 扩展产品 | 面向中小专业服务机构的人机项目交付系统 |
| 企业产品 | 面向成熟企业的虚拟职能团队与 Agent 治理平台 |
| 长期愿景 | 人领导、Agent 运营的虚拟企业经营系统 |

## 3. 稳定领域模型

### 3.1 总体关系

```text
行业经营蓝图 IndustryBlueprint
  └─生成/识别规则→ 企业 Enterprise
                         ├─业务线 BusinessLine
                         │    ├─经营项目 / 客户项目 Project
                         │    │     └─任务 Task
                         │    └─经营例程 Routine ─→ Task / ProjectDraft
                         └─企业级角色 Role

角色 Role ──任职范围──→ 企业 / 业务线 / 项目
角色 Role ──由谁承担──→ 人 / Agent / 人机组合
任务 Task ──分配给──→ RoleAssignment ──选择执行资源──→ Agent
```

正常情况下项目属于一条主业务线；跨业务线项目通过参与关系关联其他业务线。任务通常属于项目，但业务线周期例程和经营事件也可以直接生成任务。

### 3.2 企业 Enterprise

企业是持续存在的经营与治理边界，不只是租户或工作区。

核心内容：

- 使命与阶段性战略；
- 目标客户和价值主张；
- 经营模式与收入/成果来源；
- 业务线组合；
- 公司级目标与指标；
- 资源、预算和风险偏好；
- 数据、工具和外部系统边界；
- 自治政策、审批规则和企业章程。

创生模式下由系统生成企业筹建草案；共建和嵌入模式下主要来自导入与映射。

### 3.3 业务线 BusinessLine

业务线是持续创造某类价值的经营单元，不是项目分类。

核心内容：

- 服务或产品；
- 目标客户与需求；
- 收费或成果模式；
- 负责人和核心角色；
- 目标、KPI 与验证状态；
- 客户机会和项目组合；
- 标准交付物和方法；
- 周期性经营例程；
- 预算、容量和风险。

来源可以是系统生成的候选假设、用户创建、历史事实识别或外部系统导入。

### 3.4 项目 Project

项目是有明确目标、范围、期限、资源和验收方式的阶段性经营活动。

项目至少分为：

- **经营验证项目**：验证客户、产品、渠道、定价和交付能力；
- **内部建设项目**：建立服务包、知识库、方法、工具或 Agent；
- **客户交付项目**：来自真实客户机会和商业承诺；
- **经营改进项目**：由指标异常、质量问题或风险触发；
- **战略项目**：由负责人直接发起。

项目状态不从“已创建”开始，而应经过：

```text
idea/candidate → discovery → scoped → approved → active
→ review/acceptance → completed/cancelled
```

客户交付项目只有在范围、交付物、时间和验收方式被人确认后才能进入 approved。

### 3.5 任务 Task

任务是可执行、可验收、可记录成本的最小工作单元。

任务来源：

- 项目计划分解；
- 业务线经营例程；
- 指标或外部事件触发；
- 人工指令；
- 执行质量门产生的返工和澄清。

任务必须包含目标、输入、期望产出、角色、依赖、期限、验收标准、权限与风险等级。现有 Plan/DAG 是任务或任务组的执行计划，不应继续作为用户经营层的核心对象。

### 3.6 角色 Role

角色是带责任和权限的组织席位，横跨企业、业务线和项目。

角色契约至少包括：

```text
mission            角色目标
accountabilities   结果责任
scope              作用范围
authority          决策/工具/数据/预算权限
inputs_outputs     输入与交付物
collaboration      上下游与协作协议
quality_bar        质量与验收标准
escalation         何时升级给人或上级角色
capacity           容量与成本
required_capability能力要求
```

建议区分：业务负责人、项目调度角色、专业执行角色、质量治理角色。项目调度角色是持续状态的协调者，不等同于一次性 Planner 调用。

### 3.7 Agent 与任职 AgentAssignment

Agent 是可版本化的执行资源；任职关系决定它在什么角色、范围和权限下工作。

同一个 Agent 可以在不同项目任职，但权限、上下文和评价分别隔离。一个角色也可以由人、Agent 或人机组合承担。

Agent 生命周期：

```text
岗位需求 → 检索/组装/生成候选 → 企业知识接地 → 试岗
→ 评测与人审 → 影子运行 → 限定任职 → 正式任职
→ 持续评价 → 晋级/返训/降级/停用
```

## 4. 行业经营蓝图

行业经营蓝图解决“不同行业下五个主体的内核不断变化”问题，同时避免为每个行业硬编码一套产品。

蓝图包含：

- 企业与业务线的字段和约束；
- 典型服务、客户和价值链；
- 业务线组合与候选生成规则；
- 目标和 KPI 树；
- 项目类型、生命周期和阶段门；
- 角色包、职责、协作关系和权限模板；
- 任务、交付物和验收模板；
- 周期性经营例程和事件触发规则；
- 所需技能、工具、模型和数据连接器；
- 风险、合规、审批和审计规则；
- Agent 试岗样本和评测集。

蓝图只提供起点和校验框架。企业使用后会形成自己的 `CompanyOperatingModel`，可覆盖蓝图并独立版本化；蓝图升级不能直接覆盖正在经营的企业。

## 5. 对象来源、权威与自治

为了同时支持创生、共建和嵌入，每个核心对象都需要以下元数据：

| 字段 | 含义 |
|---|---|
| `origin` | generated / user_created / imported / inferred |
| `authority_source` | Polis、用户或外部权威系统 |
| `evidence_refs` | 生成或识别依据 |
| `confidence` | 系统判断置信度 |
| `validation_status` | hypothesis / pending / validated / rejected |
| `autonomy_level` | advise / draft / act / govern |
| `approval_policy` | 哪些变化需要谁批准 |
| `version` | 经营模型与规则版本 |

自治等级统一定义：

1. **advise 建议**：只能提出建议；
2. **draft 草拟**：可以创建草案，需批准后生效；
3. **act 执行**：在预算与权限内自动执行；
4. **govern 治理**：可以监测结果、自动纠偏，并仅升级例外。

不同对象、角色和风险等级可以使用不同自治水平。成熟企业默认从 advise/draft 开始；创生模式的低风险内部工作可以从 act 开始。

## 6. 三条核心运行闭环

### 6.1 企业创生与验证闭环

```text
创始人能力/客户/约束
→ 生成企业筹建方案
→ 候选业务线和经营假设
→ 生成验证项目
→ 执行市场与交付验证
→ 收集客户/收入/质量反馈
→ 保留、修改或淘汰业务线
→ 更新企业经营模型
```

系统生成的是可执行假设，不是假装已经知道正确答案。

### 6.2 客户项目闭环

```text
客户信号/沟通/材料
→ 机会与需求发现
→ 澄清问题
→ 项目范围草案
→ 人与客户确认商业承诺
→ 正式立项
→ 角色组队与 Agent 任职
→ 计划、执行、协调、质检、返工
→ 人审核与客户验收
→ 交付、复盘与资产沉淀
```

### 6.3 持续经营闭环

```text
目标/KPI/经营例程/外部事件
→ 发现机会、异常或到期工作
→ 生成项目或任务草案
→ 按自治策略批准或自动启动
→ 执行与质量门
→ 更新项目、业务线和企业指标
→ 调整优先级、资源、角色和 Agent
```

## 7. 产品信息架构

### 7.1 一级导航

1. **经营驾驶舱**
   企业目标、业务线状态、机会与项目组合、预算、风险、待决策事项。

2. **业务线**
   服务与客户、经营假设、目标指标、机会、项目、例程、团队和资产。

3. **机会与项目**
   需求发现、范围确认、立项、里程碑、团队、交付、客户反馈和复盘。

4. **工作台**
   需要用户决定、沟通、审核或处理的事项，以及正在自动运行的工作。

5. **组织与角色**
   组织关系、角色契约、人/Agent 任职、能力缺口、容量和绩效。

6. **企业资产**
   方法、知识、案例、交付模板、场景、Skill、评测集和数据连接。

7. **治理**
   自治策略、审批、权限、预算、风险、审计、质量和运行观测。

### 7.2 三种模式的不同入口

- 创生模式首页首先是“创建与验证公司”；
- 共建模式首先是“导入现状并选择一条业务线”；
- 嵌入模式首先是“连接权威系统并定义虚拟职能团队边界”。

进入正常经营后，共享相同的信息架构，避免三套产品长期分叉。

### 7.3 产品呈现原则

- 默认展示经营对象和业务结果，不默认展示 DAG、prompt、模型或 token；
- 任务是执行细节，项目和交付物是用户主要管理对象；
- “需要你处理”以决策、沟通、审批和异常为单位，而不是技术事件；
- 所有生成对象必须可解释、可修改、可拒绝和可追踪；
- 低置信度和高风险工作不能用“已完成”掩盖不确定性。

## 8. 产品能力分层

### 8.1 企业经营层（下一阶段新增重点）

- 企业筹建与经营模型；

### 8.2 Agent 任职层（现有能力升级）

- Agent 检索、组装和生成；

- 试岗、影子运行和评测；

- 真实项目绩效和版本治理。

### 8.3 执行内核（现有能力复用）

- Planner 与 DAG；
- Capability Router；
- Agent Runtime 与 Skill/MCP；
- Temporal 协同、重试、返工和人审；
- Memory、Blackboard、Artifact；
- Guardrails、Credential、Budget；
- Observability、Eval、Run Manifest。

## 9. 现有 Polis 能力映射

| 现有对象/能力 | 下一阶段位置 | 处理方式 |
|---|---|---|
| `org` | 企业实例与租户边界 | 扩展经营属性，不改变隔离职责 |
| `role` | 角色基础定义 | 升级为角色契约和范围化任职 |
| `agent/agent_version` | 执行资源 | 增加试岗、任职、绩效和生命周期 |
| `task` | 最小工作单元 | 关联项目、例程、角色和经营来源 |
| `plan` | 单次执行计划/DAG | 下沉为任务执行快照 |
| `task_run` | 任务执行记录 | 汇总到项目和交付物 |
| `plan_template/scenario` | 项目/任务方法模板 | 并入企业资产与行业蓝图 |
| `skill/capability` | Agent 能力底座 | 由岗位需求驱动，不作为主导航 |
| `memory` | 企业/业务线/项目知识 | 扩作用域、出处和授权 |
| `artifact_descriptor` | 输入材料与交付物 | 增加项目、客户许可和验收语义 |
| `approval` | 人类治理入口 | 扩为决策、承诺、范围变更和高风险审批 |
| dashboard/observability | 执行看板 | 上卷为经营驾驶舱，保留技术下钻 |

现有执行内核不需要推倒重做；主要变化是增加经营控制面，并重新定义用户入口。

## 10. 需要新增或重构的核心对象

> **V3 内核边界说明（2026-07-21）**：本节是产品语义候选清单，不是物理表或顶级元模型清单，不能
> 直接作为内核建表依据。Enterprise/BusinessLine/Project 等优先由 DomainPackage 的 Scope 类型、
> ScopeRelation、WorkDefinition 和产品 BFF 映射；内核可独立发布的定义聚合根仍只有 ADR-0015 的三个。
> 任何对象需要成为独立内核表/定义根，必须先证明无法由现有协议表达并新增 ADR。权威开发规格见
> [稳定内核 README](kernel/README.md) 与 [无歧义自主开发验收](kernel/10-无歧义自主开发验收.md)。

### 10.1 新增

- `industry_blueprint`：行业经营蓝图；
- `company_operating_model`：企业自己的经营模型版本；
- `business_line`：业务线；
- `objective` / `metric`：目标与指标；
- `opportunity`：客户或经营机会；
- `project` / `project_stage`：项目及阶段；
- `project_business_line`：跨业务线参与；
- `operating_routine` / `business_signal`：经营例程与信号；
- `role_contract` / `role_assignment`：角色契约与任职；
- `agent_qualification`：试岗、评测、影子运行与任职证据；
- `decision` / `commitment`：经营决策和对外承诺；
- `deliverable` / `acceptance`：交付物与验收。

### 10.2 重构

- Org 从“公司工作区”升级为企业经营边界；
- Role 从 title/能力集合升级为责任与协作契约；
- Task 从用户主对象下沉为项目执行单元；
- Dashboard 从运行统计升级为经营状态与例外中心；
- Agent 管理从配置页升级为任职与绩效系统；
- 场景库从模板货架升级为行业蓝图和企业方法资产的一部分。

## 11. 建议实施路线

### 阶段 D0：客户与场景验证

目标：在继续开发前确定一个愿意付费的具体用户和一个可验收项目类型。

需要完成：

- 访谈 8～12 位候选用户；
- 收集真实项目生命周期和交付物；
- 选择一个垂直场景、一个主要买家、一个交付物；
- 明确用户愿意交给 Agent 的工作与必须保留给人的工作；
- 用现有内核手工服务 2～3 个真实项目，验证价值指标。

退出门：至少有真实用户愿意提供材料试用，并愿意为持续结果付费或签署明确试点意向。

### 阶段 D1：创生模式最小闭环

目标：让一个独立专业服务创业者从经营意图启动并完成一个真实客户项目。

范围：

- 企业筹建方案；
- 候选业务线与验证状态；
- 一个标准服务/项目类型；
- 机会与需求澄清；
- 项目草案、范围确认和正式立项；
- 四类基础角色；
- Agent 试岗和限定任职；
- 交付物审核、客户反馈和复盘。

明确不做：通用行业蓝图市场、完整 CRM、完整财务、自动签约和全自治经营。

### 阶段 D2：持续经营与共建模式

目标：从完成一次项目升级为持续经营一条业务线，并适配已有小企业。

范围：

- 业务线目标/KPI；
- 周期性经营例程；
- 项目组合和资源容量；
- 历史项目/SOP/组织导入；
- 项目调度角色持续运行；
- Agent 绩效和企业知识复用；
- 经营驾驶舱和异常驱动项目。

### 阶段 D3：成熟企业嵌入

目标：在一个成熟企业业务单元中建立受治理的虚拟职能团队。

范围：

- CRM/项目/文档/身份系统连接；
- 外部系统权威映射；
- 企业级权限、审计和部署；
- 人机组织关系与分级自治；
- 多业务线、跨项目资源和预算；
- 行业蓝图版本治理。

## 12. 产品成功指标

### 12.1 用户价值

- 真实客户项目完成数；
- 从需求确认到合格首稿的周期；
- 单项目人工投入；
- 单人可并行项目数；
- 首稿通过率和返工率；
- 客户验收率和复购率；
- 用户持续付费率。

### 12.2 虚拟企业运行质量

- 系统主动发起且被采纳的经营项目比例；
- 自动执行任务占比；
- 需要人介入的异常比例；
- 角色/Agent 任职成功率；
- Agent 因能力不足被降级或返训的比例；
- 企业资产在后续项目中的自然复用率；
- 经营模型假设得到验证或淘汰所需周期。

### 12.3 治理质量

- 高风险动作未经批准执行次数必须为 0；
- 对外承诺可追溯率 100%；
- 结论出处覆盖率；
- 权限越界和跨企业数据泄漏为 0；
- 实际成本与预算偏差；
- 外部系统事实冲突数。

## 13. 主要风险与约束

1. **用户过宽**：同时为创业者和大型企业设计，会导致入口、销售和能力边界全部失焦。先选一个首发用户。
2. **生成幻觉变成经营事实**：所有生成对象必须有来源、状态、证据和验证机制。
3. **项目被过度流程化**：ToB 项目中的沟通、承诺和范围变化必须进入一等模型。
4. **角色空心化**：不能用 title + prompt 冒充组织与协同能力。
5. **Agent 能力虚高**：上岗必须有试岗证据，评价必须来自真实结果。
6. **重造企业软件**：不重做 CRM、ERP、财务和合同系统；只补经营智能、协同和行动闭环。
7. **自治过快**：先建议和草拟，再逐步开放执行与治理权限。
8. **用户管理负担反增**：企业、业务线、项目、任务和角色不能变成五套需要手工维护的 CRUD。
9. **行业蓝图过早平台化**：先打穿一个行业和项目类型，再抽象蓝图协议。

## 14. 下一步必须回答的产品决策

1. 首个具体用户是谁，能否精确到行业、公司阶段和职位？
2. 首个真实项目类型和交付物是什么？
3. 首发采用创生模式，还是以一个已有小企业做共建模式？
4. 项目调度角色第一版由人主导、Agent 辅助，还是允许低风险项目 Agent 主导？
5. 哪些经营事实由用户确认，哪些可以通过外部系统成为权威？
6. Agent 试岗采用什么样本、指标和晋级门槛？
7. 首个闭环需要接入哪些沟通和文档来源？
8. 商业模式按订阅、项目容量、运行量还是交付结果计费？

在上述决策得到客户证据前，本篇作为产品方向与领域模型基线，不直接等同于研发排期。
