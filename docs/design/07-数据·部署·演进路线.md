# 07 数据 · 部署 · 演进路线 — 详细设计

> **本篇目标**：可指导开发的全局数据视图(ER 汇总)、部署拓扑与配置、CI/迁移、
> 以及把 02~06 的任务卷成里程碑计划。低运维、单体优先。

---

## 1. 全局数据视图（汇总 02~06 的表 → 一个 PostgreSQL）

```text
[组织] org, org_member, role
[资源] agent, agent_version, agent_capability, scenario_preset, capability
[技能] skill, skill_version, skill_invocation
[规划] plan, plan_template
[执行/记忆] memory(+vector), result_envelope, artifact_descriptor
[模型/凭证] model_catalog, credential(密文)
[治理/观测] run_manifest, approval, trace_ref, audit_log
```
关系要点：一切以 `org_id` 行级隔离；`agent_version.config`/`plan.dag`/`memory.embedding` 等用 JSONB/vector。
**向量不另设库**（pgvector）；产物字节进对象存储，库存 `artifact_descriptor`。

### 1.1 审计与隔离（横切）
```sql
CREATE TABLE audit_log (
  id BIGSERIAL PRIMARY KEY, org_id UUID, actor TEXT, action TEXT,
  target TEXT, detail JSONB, at TIMESTAMPTZ DEFAULT now());
-- 多租户: 所有业务表带 org_id; 应用层统一注入 org 过滤 + PostgreSQL RLS 兜底(V1 基线, 见 09/ADR-0005)
-- 平台级表(app_user/auth_session/org_invite)不带 org_id、不启用 RLS
```

## 2. 服务拓扑（单体优先，模块化）

```text
agentos-app (FastAPI modular monolith, 一个进程/镜像; Python 3.12)
 ├─ module: org/provision     (02)
 ├─ module: planner/router    (03)
 ├─ module: runtime/skill     (04)   ── Temporal Worker, 可水平多副本
 ├─ module: memory            (05)
 ├─ module: model/credential  (06)
 ├─ module: observability/eval(06)
 └─ module: api(FastAPI 路由层)

agentos-web (新建前端, React/Next.js)   ── 聊天+Org管理+计划审批+看板+审批收件箱
  └─ 经 REST/WS 调 agentos-app, 与后端语言无关

外部基础设施(MVP 仅 4 项)：
  PostgreSQL(+pgvector) · Temporal · LiteLLM · Langfuse
  对象存储: 本地卷(→MinIO)
```
> Temporal Worker 与 FastAPI API 可同进程(开发)或拆为 `api` 与 `worker` 两进程(同代码库, 生产)。
- **Worker 水平扩展**：runtime 模块以多副本作为 Temporal Worker 池(无状态)；其余模块单实例即可。
- **不拆微服务**(V1/V2)：少服务发现/网关/分布式调试成本。V3 再按压力拆(参考早期设想的 12 服务为终态)。

## 3. 部署与配置

### 3.1 docker-compose（MVP 单机）
```yaml
services:
  postgres:   { image: pgvector/pgvector:pg16, volumes: [pgdata:/var/lib/postgresql/data] }
  temporal:   { image: temporalio/auto-setup }
  litellm:    { image: ghcr.io/berriai/litellm, command: --config /litellm.yaml }
  langfuse:   { image: langfuse/langfuse }
  agentos-app:{ build: ./backend,  # FastAPI(uvicorn) + Temporal worker
                depends_on: [postgres,temporal,litellm,langfuse],
                environment: [DATABASE_URL, TEMPORAL_ADDR, LITELLM_BASE, LANGFUSE_KEYS, KMS_MASTER_KEY] }
  agentos-web:{ build: ./frontend, # React/Next.js
                environment: [API_BASE], depends_on: [agentos-app] }
volumes: { pgdata: {} }
```
> 代码库建议结构：`backend/`(Python/FastAPI) + `frontend/`(React) + `infra/`(compose/迁移)。
### 3.2 关键配置
- `KMS_MASTER_KEY`：信封加密主密钥(06)，MVP 注入应用，规模迁 Vault/云 KMS。
- `litellm.yaml`：声明支持的模型与默认；用户 Key 走 BYO(运行时注入)，不写配置。
- 迁移：用 Alembic/Flyway 管 DDL；`capability`/`scenario_preset` 等基础数据用 seed 脚本。

### 3.3 规模化(V3 才做)
- K8s：Temporal 集群、worker HPA 弹性、对象存储换 MinIO/S3、按需加 Redis 缓存。
- 明确**不在 V1/V2 引入** Kafka/ES/Milvus/OPA。

## 4. 里程碑计划（把 02~06 的 WBS 卷成版本）

### V1（0→1，目标：验证"大脑能不能转"）
```text
范围: 身份/账号(09) + 薄 Org + 1~2 Agent + MCP 工具 + 模板优先 Planner + Plan校验 + Temporal
      + 任务记忆(pgvector) + LiteLLM + Langfuse；单体 + compose；多租户 = org_id 行级 + RLS 兜底
里程碑:
  M1 基础设施起栈 + DDL 迁移(02/05/06/09 表) + 认证/RLS 基线  [T2.1,T5.1,T6.1,T9.1,T9.4]
  M2 Org/编配最小闭环(模板命中即建公司)            [T2.2~T2.8]
  M3 Planner(模板)+校验 + Temporal 串/并/人审       [T3.1~T3.5]
  M4 Runtime+MCP+Guardrails 单节点跑通             [T4.1~T4.7]
  M5 记忆写入/检索 + 出处 + 凭证短时注入            [T5.2~T5.7,T6.3]
  M6 Langfuse+Eval+审批收件箱                       [T6.4~T6.7]
  M7 采购场景端到端 + 量化指标(03 §9)              ← V1 验收门
  ── 前端轨(与后端并行) ──
  MW1 前端脚手架(React/Next) + 聊天界面(接 chat API)
  MW2 Org 管理 + 计划审批 + 公司看板 + 审批收件箱界面
```
**V1 验收门(决定是否进 V2)**：DAG 可用率≥70% / 路由命中≥75% / 人审通过≥50% / 成本时延在预算内。

### V2（1→10，目标：能协同）
能力路由完整(冷启动→历史) + 三层记忆 + 编配器(生成+验证) + 有界重规划 + guardrails 全量 + 多 Agent 共享记忆并发。对象存储换 MinIO。

### V3（10→100，目标：能治理）
Agent Registry/Marketplace + 治理/预算 + Policy Engine(OPA) + K8s 弹性 +（按需）schema/物理隔离、微服务拆分。
（多租户 RLS 已提前到 V1 基线，见 09/ADR-0005，不再列 V3。）
