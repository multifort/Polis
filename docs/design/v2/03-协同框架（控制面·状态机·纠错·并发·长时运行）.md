## 0. 一句话
协同框架是 V1 `TaskWorkflow` 的升级版**控制面**——只负责调度/状态/检测纠错决策/治理，**绝不亲自执行任务**；让任务**动态可调、跑不乱、错能修、能长跑、可并发**。

## 1. 背景与目标

### 1.1 V1 现状（`planner/workflow.py`）
- `TaskWorkflow`（Temporal workflow）：`run(plan, org_id, task_id)` 分层 ready-set 调度（`asyncio.gather` 跑就绪节点）+ `_node_status` 状态跟踪。
- `run_node`（@activity）：委派给执行面 `agent_runtime.execute_node`（**控制/执行已分离**）。
- 容错：activity `RetryPolicy(maximum_attempts=3)` + `_bounded_replan`（删失败节点+修依赖+重 `validate`，`MAX_REPLANS` 封顶）。
- 人审：human 节点 `wait_condition` 等 `approve` signal。
- 并发：每任务一个 workflow，Temporal 天然横向并发。

### 1.2 升级目标（用户 2026-06-22）
1. **任务动态调整**：失败/不达标 → LLM 局部重规划（回调内核），最小作用域优先。
2. **确保不乱**：每次调整后重 `validate()` + 结构不变量 + Temporal exactly-once/可恢复。
3. **出错能检测并修正**：异常**和质量不达标**都能检测（V1 只检测异常）；有界分级纠错。
4. **能长时间运行**：durable + workflow versioning + heartbeat + 长等待 human gate。
5. **多任务/计划并发**：org 级并发上限 + 公平 + 全局可见（预算只提示，见 §6）。

### 1.3 铁律：非任务执行器（控制面）
协同框架**只调用、不实现**：执行交执行面、重规划交内核、质量判定交 Evaluator、人审交审批收件箱。它本身**不调 LLM 干活、不跑 Skill**。

## 2. 设计总纲与边界：三面分工
```
内核(Planner/Composer) ──plan──▶  ┌──────────────────────────┐
        ▲ 局部 replan(子图)        │      协同框架(控制面)      │
        └──────────────────────────│  ①调度 ②状态 ③检测+纠错   │
                                    │  ④治理(不变量/并发/预算/  │
                                    │       人审/长跑)          │
                                    └───────────┬──────────────┘
   run_node(activity) │ quality check │ human gate │ 读写黑板 │ 可见
        ▼             ▼               ▼            ▼          ▼
   执行面          决策面          审批收件箱     记忆黑板    任务记录#6
 (Agent+Skill)   (Evaluator)
```
**控制面只做四件事**（其余全是"调用边界"）：调度、状态机、检测+纠错决策、治理。

## 3. 节点状态机（控制面维护，durable）
```
pending ─(deps 全 done)─▶ ready ─(派发)─▶ running ─┬─ done           产出 + 质量达标
                                                   ├─ failed         异常（重试耗尽）
                                                   ├─ needs_rework   质量不达标(judge<τ)   ← V2 新增
                                                   ├─ needs_human    危险/置信低/超纠错预算
                                                   └─ replanning      局部重规划中           ← V2 新增
任务态 = 由节点态派生（全 done→done；任一 failed 且不可纠正→failed；有 needs_human→waiting_human）
```
**合法转移**（其余转移非法，强校验防乱）：
| 从 | 到 | 触发 |
|---|---|---|
| running | done | 执行成功 + （关键节点）质量达标 |
| running | needs_rework | 质量不达标 |
| running | failed | 异常且 retry 耗尽 |
| needs_rework | running | rework / local replan 后重跑 |
| needs_rework | needs_human | 超纠错预算 → 升级 |
| replanning | ready/running | 子图拼接 + 重 validate 通过 |

## 4. 检测 + 纠错（有界分级）

### 4.1 检测
- **异常检测**：节点抛错 / 超时 / Guardrail block（V1 已有）。
- **质量检测（V2 新增，仅关键节点）**：节点产出过 Evaluator（断言 + LLM-judge），`judge < τ` = 软失败。**只评关键节点**（如 `report.generation`、危险节点；廉价/中间节点跳过，省 eval 成本）——节点配置 `evaluate: bool`。

### 4.2 分级纠错（逐级升级，每级有上限，最小作用域优先）
```
节点失败/不达标 → 按序尝试：
  ① retry         瞬时错误（Temporal RetryPolicy，已有）
  ② rework        把 judge/异常原因反馈喂回上下文，重跑【同节点】     上限 1
  ③ local replan  回调内核重生成【该节点 → 子图】→ validate → 拼接   上限 1（节点级）
  ④ escalate      升级人审（审批收件箱）
  ⑤ abort         超【全任务 replan ≤ 3】或纠错预算 → 任务 failed + 诊断
```

### 4.3 纠错预算（初值，按数据校准）
| 参数 | 初值 |
|---|---|
| 节点 rework | ≤ 1 |
| 节点 local replan | ≤ 1 |
| 全任务 replan（沿用 `MAX_REPLANS`） | ≤ 3 |
| 质量阈值 τ | ≈ 0.6 |
- 超预算 → 不再自动烧钱，转 `needs_human` 或 `abort`。这是"自动纠错"的安全边界（接信任分层：自动跑、有界、超界交人）。

---

## 5. 动态调整 / 局部重规划（确保不乱）
- **作用域最小优先**：单节点 rework → 子图 replan → 全图 replan。
- **流程**：协同框架圈定"失败子图"→ 回调**内核 Planner**（带 goal + 已完成上游产出摘要 + 失败原因）重生成 →**强制重 `validate()`**（无环/依赖闭合/能力可满足/预算内）→ 通过才**拼接进 DAG**（durable workflow state）→ 重新进 ready 调度。
- **不变量（每次调整后必校验，"确保不乱"的硬保证）**：
  1. DAG 始终无环；2. 无悬空依赖/孤儿节点；3. 已 done 节点不被重复执行（幂等 + 黑板缓存）；4. 能力闭合（只用 active 能力）；5. 预算/节点数上限。
- Temporal 提供 exactly-once 与可恢复；workflow state 持有"演进中的 DAG"。

## 6. 并发与预算（两者解耦）

### 6.1 并发（真实限制）
- 每任务一个 workflow，Temporal 横向并发；
- **org 级并发上限**：超限任务排队（admission），保资源公平；同 org 多任务 FIFO/优先级调度，防大任务饿死小任务；
- 实现复用优先：DB 计数 / Temporal 并发控制，**不引新组件**。

### 6.2 预算（**只提示、不当闸** — 用户决策）
- 预算**不作为并发/派发的阻断点**；
- 接近/超出 org 预算时 → **告警/提示 owner**（审批收件箱通知 / 任务记录标红），**不暂停、不降级、不拒绝**执行；
- 成本/token 数据照常采集（V1 成本门 + 记忆 §8 预算观测），供提示与事后分析。

> 区分清楚：**并发**是资源公平的真实闸；**预算**只是知情提示。二者互不耦合。

## 7. 长时间运行
- **持久**：Temporal 为长流程而生，崩溃可恢复、状态不丢；任务态全在 DB（记忆黑板）+ event history，天然 checkpoint。
- **长等待**：human gate = durable wait（可等天级）。
- **Workflow versioning（patching）**：让"改了编排代码"不破坏在飞任务——直击 V1 那个"改 worker 必重启、在飞任务会崩"的坑。**这是长时运行的硬要求**。
- **长 activity**：heartbeat + 合理 `start_to_close_timeout`；执行面节点支持中途心跳。

## 11. 关键决策（建议落 ADR-0012）
- 协同框架 = **纯控制面**（非任务执行器）。
- 质量门**只评关键节点**；纠错**有界分级、最小作用域优先**（初值见 §4.3）。
- **预算只提示、不阻断**；并发是独立的真实限制。
- 长时运行靠 Temporal 持久 + **workflow versioning**。

## 12. 未决问题

## 14. 核心交互时序流程

### 14.1 节点调度 + 质量门 + 分级纠错时序
**说明**：控制面跑一个节点的**主控环**——按依赖派发、委派执行面执行、对关键节点过质量门、不达标则有界分级纠错。它把"出错能检测并修正"落到每个节点的生命周期上，且控制面只调度/判定/决策，**绝不自己执行**。
```
TaskWorkflow(控制面)   执行面(run_node)   Evaluator        审批收件箱
  │ ready=依赖全done的节点                  │                │
  │ for n in ready:                        │                │
  │   状态[n]=running                       │                │
  │   execute_activity(run_node,n,goal) ──▶│ 跑 Agent+Skill  │   ① 委派执行(非内联)
  │                          产出/异常 ◀────│                │
  │  ┌ alt [n 是关键节点] score(产出) ──────────────▶│ judge  │   ② 仅关键节点质量门
  │  │                              judge ◀──────────│        │
  │  └ alt [非关键] judge=skip                       │        │
  │  ┌ alt [异常 或 judge<τ] → 分级纠错(见步骤序)              │
  │  │     retry → rework(≤1) → local_replan(≤1, 见14.2) → escalate → abort
  │  │     [escalate] 创建 needs_human ──────────────────────▶│ 人审
  │  └ alt [ok] 状态[n]=done; 写黑板(供下游)                   │
  │  循环直到无 ready；任务态由节点态派生                       │
```
规则：纠错每级有上限（rework≤1 / local replan≤1 / 全任务 replan≤3）；超界→needs_human 或 abort，不无限烧钱。

### 14.2 局部重规划时序（最小作用域 + 确保不乱）
**说明**：当某节点反复失败/不达标，**圈出最小失败子图、回调内核重生成、强制重校验后拼接回 DAG**。体现"任务动态调整"且"确保不乱"——每次改图都过 `validate` + 结构不变量，且作用域最小优先（单节点→子图→全图）。
```
TaskWorkflow        内核(Planner)        Validator
  │ 圈定失败子图 G'(失败节点+受影响下游)    │            │   ① 最小作用域
  │ replan(G', goal, 已完成上游摘要, 失败原因) │          │
  │──────────────────────▶│             │            │
  │            新子图 G'' ◀│ RAG 重生成   │            │
  │ validate(DAG.replace(G',G''), active) ───────────▶│   ② 拼接后强制重校验
  │            ok/errors  ◀───────────────────────────│
  │ [ok] 替换子图 + 重置受影响节点为 ready → 回 14.1 调度 │
  │ [errors 或 超 replan 上限] → abort + 诊断           │
```
不变量（每次改图后必校验）：无环 / 无悬空依赖 / 已 done 不重跑（幂等+黑板缓存）/ 能力闭合 / 预算与节点数上限。

### 14.3 并发准入 + 预算提示时序（两者解耦）
**说明**：多任务并发时**并发是真实闸（超限排队）、预算只提示（不阻断）**。准入控制保资源公平；预算接近/超出只告警 owner，绝不暂停/降级/拒绝执行。
```
新任务approve   并发Governor(org)        预算监视        owner
  │ admit(org)      │                     │             │
  │────────────────▶│ 在跑数 < cap ?       │             │   ① 并发真实闸
  │  ┌ alt [未超] 放行 → 启动 workflow      │             │
  │  └ alt [超限] 入队(FIFO/优先级)         │             │
  │ ── 运行中每派发贵节点 ───────────────▶│ 累计成本 vs 预算 │
  │                 │  [接近/超] 告警 ───────────────────▶│ 提示(不阻断)  ② 预算仅提示
```

## 15. 核心实现规则（状态机 · 算法 · 不变量 · 阈值）

### 15.1 关键数据结构（workflow 持久状态）
```python
WorkflowState = {
  node_status: {node_id: pending|ready|running|done|failed|needs_rework|needs_human|replanning},
  dag: PlanDag(可演进),               # 局部重规划会替换子图
  goal: str,                          # 贯通到执行(F3)
  correction: {node_id: {rework:int, replan:int}},  # 纠错预算计数
  replan_total: int,                  # 全任务重规划计数(≤MAX_REPLANS)
  task_id, org_id,
}
```

### 15.2 算法：就绪调度 `schedule_loop`
**说明**：控制面的主循环——找出依赖已满足的节点并发派发，直到无可调度。
```
while not abort:
  ready = [n for n in dag.nodes if status[n]∉{done,failed}
           ∧ all(status[d]==done for d in n.deps)]
  if not ready: break
  results = gather(exec_node(n) for n in ready)        # 并发派发(执行面)
  for n,r in results: handle_result(n, r)              # 见 15.3
派生任务态: 全 done→done; 不可纠正的 failed→failed; 有 needs_human→waiting_human
```

### 15.3 算法：检测 + 分级纠错 `handle_result`
**说明**：单节点返回后的判定与纠错决策树，是"出错能修"的核心规则。
```
输入: n, result
1. if result.error: 进纠错(异常分支)
2. elif n.evaluate(关键节点): judge ← Evaluator.score(result); if judge<τ: 进纠错(质量分支)
3. else: status[n]=done; write_blackboard(n); return
纠错(逐级, 每级查预算):
   ① 瞬时错误 → Temporal retry(已配 maxAttempts)
   ② correction[n].rework<1 → rework: 反馈(judge/异常因) 重跑同节点; rework++
   ③ correction[n].replan<1 ∧ replan_total<MAX_REPLANS → local_replan(n) (见15.4)
   ④ 否则 → status[n]=needs_human(escalate 审批收件箱)
   ⑤ 不可恢复/超总预算 → abort(任务 failed + 诊断)
阈值: τ≈0.6; rework≤1; replan≤1(节点); MAX_REPLANS=3(全任务)
```

### 15.4 算法：局部重规划 `local_replan`
**说明**：圈最小失败子图、回调内核重生成、校验后拼接，保证动态调整不破坏 DAG 合法性。
```
输入: failed_node
1. G' ← {failed_node} ∪ 受影响下游(数据依赖闭包)
2. G'' ← 内核.replan(G', goal, 已完成上游摘要, 失败原因)   # RAG 重生成子图
3. new_dag ← dag.replace(G', G'')
4. vr ← validate(new_dag, active_caps)                    # 强制重校验
   if not vr.ok: replan_total++; if 超限 → abort; else 退化到全图 replan 或 escalate
5. dag ← new_dag; 受影响节点状态重置为 pending; replan_total++
不变量校验: 无环/无悬空依赖/已done不重跑/能力闭合/预算&节点上限
```

### 15.5 并发准入 / 预算提示 / 长时运行规则
**说明**：并发与预算的实现规则（解耦）、以及长跑的关键保障。
```
并发(真实闸): admit(org): 计 running workflows(org); ≥cap → 入队; 否则放行。 cap=org 配置(默认 N)
预算(仅提示): 派发贵节点前 cost_so_far(org) vs budget; 接近(≥80%)/超出 → 通知 owner; 不暂停/降级/拒绝
长时运行: Temporal 持久(崩溃可恢复) + workflow versioning/patch(改编排码不破在飞任务)
          + activity heartbeat+超时 + human gate=durable wait(天级)
错误处理: 内核不可达→重试/退化; Evaluator 不可达→关键节点降级为"需人审"而非静默放行
```
