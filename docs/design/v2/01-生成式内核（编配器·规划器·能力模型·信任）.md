## 0. 一句话
给定目标，内核**先检索复用、不足则 LLM 生成**出"对的计划 + 对的团队"，**全部经确定性校验与分层信任背书**后运行，并把满意的产物**沉淀回仓库**——让虚拟公司越用越聪明。

## 1. 背景与目标

### 1.1 V1 现状（确定性薄脑）
- 规划器 `planner/service.plan`：**模板优先**——遍历 `plan_template`，取第一个"所有节点能力 ⊆ 公司可用能力"的模板，填 `goal`、过 `validate()`、确定性路由（`router.select_agent`）。无模板即 `NoTemplateMatch`(404)。
- 编配器 `org/provisioning.provision`：**仅从预设实例化**花名册（`scenario_preset.config.agentTemplates`），不会按需生成角色/技能。
- 能力词表 `capability` 表：受控、key 即主键、已带 `embedding`(1024)+hnsw 索引。
- 执行：`agent_runtime.execute` → lite-agent `_loop`（LLM 决定调用哪个 Skill/工具）。

### 1.2 V1 实证（M7 验收门 + F3）
- 验收门四项达标（DAG 可用率 100% / 路由命中 100% / 人审通过 66.7%→**91.7%** / 成本时延在预算）。
- **F3 关键发现**：人审通过率低的根因是"**所有目标被同一模板套死**，支出/质量类目标答非所问"。把用户 `goal` 贯通进 Agent 上下文后，同一确定性管线质量从 66.7%→91.7%。
- **结论**：V1 的天花板是"模板固定 → 结构不随目标变"。V2 必须让目标**驱动结构本身**（生成/挑选合适的计划与团队），而不只是改写模板内文字。

### 1.3 V2 目标
1. 规划器：**检索场景模板 → 不命中则 LLM 生成 DAG**（偿还 TD-020）。
2. 编配器：**检索角色/技能 → 不足则自动拼装 Agent；缺技能则生成（人审）**（落地"编配器(生成+验证)"）。
3. 能力词表：从"受控封闭"演进为"**开放但靠 Skill 背书**"。
4. 信任：建立**可自动跑完的分层信任机制**（人只卡在"新代码入库"）。

### 1.4 非目标（后置）
- 跨组织共享/Marketplace（V3；V2 只做组织内私有，schema 预留 `scope`）。
- 有状态/长生命周期 Agent、Agent 间直接通信（V2+，本块不做）。
- 多模型主模型/按 Agent 选型第二步（与本块解耦，见 TD-026）。

## 2. 设计总纲

### 2.1 统一范式：检索 → 生成 → 校验 → 门控 → 沉淀
规划器与编配器**共用同一条退化链**：

```
目标/需求
  │
  ├─①检索(便宜·确定·安全)  命中且满足 ─────────────┐
  │     语义匹配 模板/角色/技能（embedding）          │
  │                                                  ▼
  └─②生成(贵·LLM·容错)  未命中/不满足 ──▶ ③校验(确定性兜底) ──▶ ④门控 ──▶ 运行
        RAG 接地：照"最像的样例"改                     │            （审批/沙箱/eval）
                                                       │                 │
                                              失败→重试/重规划(#5)        ▼
                                                              ⑤沉淀(满意→模板 / 审核→Skill)
```

要点：
- **不是"模板 vs 生成"二选一**，是退化链：命中就用（省/稳），不命中才生成（通用），生成物验证通过后回沉淀，下次变成"可检索命中"。
- ⑤沉淀让仓库**自增长**（飞轮）：跑得越多 → 模板/角色/技能越厚 → 检索命中越高 → 生成越少 → 越快越省越准。

### 2.2 分工线：规划=LLM（容错），执行=确定
| 层 | 谁主导 | 容错策略 |
|---|---|---|
| 规划（目标→DAG） | **LLM** | 出错可重规划（#5 有界重规划） |
| 编配（需求→Agent 拼装） | **LLM**（拼装 Skill） | 拼装不满足→换 Skill/生成 |
| **执行（跑 Skill/工具）** | **确定性** | Skill 固定、沙箱、io_schema 约束 |
| 推理（Agent 选哪个 Skill/给什么参数） | LLM | 超步保护 + Guardrails |

**铁律**：LLM 生成的一切（DAG、Agent 配置、能力声明）**必须过确定性校验**才进入执行；真正"产生副作用"的只有确定性 Skill。

### 2.3 信任分层（人审收敛到 Skill）
关键洞察：**Skill 是唯一的"新代码"资产**；Agent/角色只是"在已审 Skill 之上的配置拼装"。因此：

| 路径 | 背书方式 | 频率 | 依据 |
|---|---|---|---|
| 任务热路径（生成计划/组队→跑→出结果） | **自动评测（Evaluator）** 无人卡 | 每任务 | 决策 #1 |
| 资产沉淀：**新 Skill 入库** | **人工审核**（代码/安全/沙箱） | 低频 | 决策 #2.4 |
| 资产沉淀：**场景模板入库** | **用户满意度**（用户主动保存） | 低频 | 决策 #2.4 |

→ "全自动"与"必须人审"不矛盾：**人离开每个任务的热路径（靠 eval），只在"往仓库沉淀新积木"时回来**。

## 3. 概念模型与数据模型

### 3.1 层级（确认）
```
Org（虚拟公司）
 └─ Agent（= 单一职责岗位/Role；按需组合；含多 Skill）
      └─ Skill（固定·确定执行·人审入库·provides 能力；自由组合）
```
- **Role = 岗位 ≈ Agent（1:1）**（用户 2026-06-22 决策）；"一角色多 Agent"滞后。"角色技能多"由"Agent 含多 Skill"承接。
- **Skill 是唯一固定/受审单元**；Agent、角色按需组合。

### 3.2 能力模型：开放词表 + Skill 背书
**能力归属在 Skill，不在 Agent**（用户决策）：
- `skill.capability`：该 Skill 实现的能力 key（V1 已存在该字段）。
- **Agent 的能力 = 其装备 Skills 的能力并集（派生）**，不独立声明。
- 计划节点 `required_capabilities` 声明所需；路由/编配匹配"Agent 派生能力 ⊇ 节点能力"。

**背书链**（一个能力变 `active/可信` 的条件）：
```
新能力提案 ──语义去重(capability.embedding 近邻)──▶ 复用已有？是→用旧key
                                                  否→需 Skill 背书
   ≥1 个 Skill 声明实现它  +  该 Skill 过 eval/审核(status=published, trust≥verified)
                                                  ──▶ 能力置 active
```
→ 能力的"信用"从"实现它的 Skill 过了审核+评测"流过来。规划器**只能用 `active` 能力出图**，保证 `validate()` 成立。

### 3.3 数据模型演进（全部向后兼容 · 先加后删 · Alembic）

| 表 | V1 现状 | V2 增量 | 迁移 |
|---|---|---|---|
| `capability` | key(PK)/domain/name/description/embedding | + `status TEXT default 'active'`（proposed/active/deprecated）<br>+ `scope TEXT default 'global'`（org/global）<br>+ `owner_org_id UUID null`（org 私有能力时） | 加列，旧行默认 active/global |
| `skill` | name/kind/trust/status/**capability**/owner/owner_org_id | （字段已够）规范化：`capability` 必填于 published；`trust` 晋升流转 | 不改结构，补约束于应用层 |
| `skill_version` | version/content/mcp_server/tool/io_schema/permissions | （已够）生成的 Skill 草稿写 content + io_schema | 无 |
| `agent` | role_id/name/source/status/current_version | `source` 增 `'generated'` 语义已在枚举内（preset/generated/custom） | 无 |
| `agent_capability` | agent_id/capability/level | **降级为"派生缓存"**：编配后由 Skill 并集回填；兼容旧预设直挂 | 保留，写入逻辑改 |
| `plan_template` | name/version/dag_skeleton/... | + `embedding`(1024)+hnsw（语义检索模板）<br>+ `scope`/`owner_org_id`（私有模板）<br>+ `source TEXT default 'curated'`（curated/generated） | 加列 + 手工建 hnsw 索引（见 0b 约定） |
| `agent_version` | version/config(JSONB)/status | config 增 `provenance`（生成出处）、`eval`（评分快照） | JSONB 加键，无 DDL |

> 说明：`capability.embedding`、`skill.capability`、`skill.trust/status`、`agent.source='generated'` **V1 已就位**——V2 主要是把这些字段的**流转激活**，结构改动很小。

## 4. 规划器（Planner）：检索 → 生成 → 校验

### 4.1 接口（演进 `planner/service.plan`）
```python
async def plan(session, org_id, goal) -> PlanResult:
    available = await available_capabilities(session)          # active 能力集
    tpl = await retrieve_template(session, goal, available)    # ① 语义检索（新）
    if tpl is not None:
        dag = fill_template(tpl, goal)                         # 模板优先（同 V1）
    else:
        dag = await generate_dag(gateway, goal, available,     # ② RAG 生成（新, TD-020）
                                 exemplars=await topk_templates(goal))
    vr = validate(dag, available)                              # ③ 确定性校验（同 V1）
    if not vr.ok:
        dag = await repair_or_raise(dag, vr, gateway)          # 一次自修复，仍失败→PlanInvalid
    routing = await route_or_compose(session, org_id, dag)     # ④ 路由/编配（见 §5）
    return persist(...)
```

### 4.2 ① 检索（V2-A.1，先做）
- 升级 V1 的"遍历+子集判断"为**语义检索**：`plan_template.embedding` 对 `goal` 取 top-k，过滤"能力 ⊆ available"，按余弦相似 + 历史成功率排序，取最佳。
- 复用：bge embedding（M6-A）+ pgvector hnsw。
- 退化：top-1 相似度 < 阈值 τ_tpl → 判定"未命中"→ 进②生成。

### 4.3 ② 生成（V2-A.2）：RAG 接地的 DAG 生成
- **接地**：把 top-k 最相似模板的 `dag_skeleton` 作为 **exemplars** 喂给 LLM（"照最像的改"），不凭空造。
- **输出契约**：LLM 必须产出**严格的 `PlanDag` JSON**（workflow_name/goal/budget_cents/nodes[]，每节点 id/type/deps/required_capabilities/...）。用 Pydantic `PlanDag.model_validate` 强约束；解析失败→重试（≤N）→失败兜底。
- **能力约束**：`required_capabilities` 只能取 `active` 能力 key（把词表注入 prompt）；若 LLM 需要不存在的能力→进编配器的 Skill 生成链（§5.4）。

### 4.3.1 生成 DAG 的伪码
```
exemplars = topk_templates(goal, k=3)
prompt = system(角色=规划师, 规则=只用active能力/必须无环/危险动作必须human节点)
       + user(goal + 可用能力词表 + exemplars 的 skeleton)
for attempt in 1..N:
    raw = gateway.chat(model, prompt)                 # LLM
    dag = try PlanDag.model_validate(json(raw))       # 结构校验
    if ok and validate(dag, active_caps).ok: return dag
    prompt += 错误反馈(校验失败原因)                    # 自修复反馈
raise PlanInvalid
```

### 4.4 ③ 校验（沿用强化 `validate()`）
现有规则全保留（无环 / 依赖存在 / 能力可满足 / 预算内 / 危险动作必 human gate）。V2 补：
- **预算闸**：`estimate_cost_cents(dag) ≤ budget`；生成的 DAG 节点数上限（防 LLM 膨胀）。
- **能力闭合**：所有 `required_capabilities ∈ active`，否则触发"缺能力"处理（§5.4）。

### 4.5 错误与重规划（衔接 #5）
- 生成/校验失败 → 一次自修复 → 仍失败 → `PlanInvalid`（前端提示，可人工改 goal）。
- 运行期节点失败 → 现有"有界重规划"（移除节点修复依赖）→ V2 增强为"**LLM 局部重规划**"（详见 #5 协同框架文档）。

### 4.6 验收指标（本子块）
| 指标 | 定义 | 门槛 |
|---|---|---|
| 无模板目标 DAG 可用率 | 模板未命中→生成→过 validate 的比例 | ≥ 70% |
| 生成 DAG 结构合法率 | LLM 输出一次过 Pydantic+validate | ≥ 80%（含 1 次自修复） |
| 检索命中率 | 目标命中已有模板（省去生成）的比例 | 观测项（飞轮指标，越高越好） |

## 5. 编配器（Composer）：角色驱动主干 + 任务驱动弹性

### 5.1 范式（ADR-0010）
- **角色驱动为底座**：公司有稳定花名册（贴"虚拟公司"隐喻；稳定组织才能积累 org 记忆/路由历史/亲和度）。沿用 V1 形态，不推翻。
- **任务驱动为弹性**：任务需要花名册覆盖不了的能力时，**按任务临时组队**；高频复用 → 晋升为常设角色。
- 同构于检索→生成链：常设花名册 = 检索命中；临时组队 = 生成兜底。

### 5.2 路由/编配主流程 `route_or_compose(dag)`
```
for node in dag.agent_nodes:
    agent = select_agent(node.required_capabilities)        # ① 检索：现有 Agent 能力覆盖？
    if agent is None:
        agent = await compose_agent(node.required_capabilities)  # ② 生成：拼 Skill 成 Agent
    routing[node.id] = agent.name
```

### 5.3 ② 生成 Agent = 拼装已有 Skill（自动，无人审）
```
needed = node.required_capabilities
skills = [retrieve_skill(cap) for cap in needed]          # 每能力找一个 published Skill
if all(skills):                                            # 全有 → 自动拼 Agent
    cfg = AgentConfig(prompt=生成的岗位说明, skills=[...], capabilities=needed, executor='lite-agent')
    agent = create_agent(source='generated', config=cfg)  # 即草稿→eval→激活（数据级，自动）
    return agent
else:
    return compose_with_new_skill(needed, missing=...)    # 缺 Skill → §5.4
```
- Agent 是**对已审 Skill 的配置拼装**，不产生新代码 → 可自动生成 + 自动 eval，**不卡人审**。
- `agent.source='generated'`、`agent_version.config.provenance` 记录生成出处。

### 5.4 缺 Skill → 生成链的"人审停点"
**自动化链跑多深 = 由模型本身决定**：
```
目标→DAG(需要能力X)→无 Agent 覆盖 X→无 Skill 提供 X
   →【生成新 Skill 草稿(skill.status=draft, trust=private)】
   →★人审墙（代码/安全/沙箱）★  ← 自动化在此停下
   →审核通过(status=published, trust=verified)→能力 X 置 active→回到 §5.3 自动拼装
```
- 即：**"拼装已有积木"全自动；"造新积木(Skill)"必停下等人审**——卡点是模型长出来的，非人为设。
- V2-A.1（检索期）可先不做 Skill 生成（缺 Skill 即"该目标暂不可办"，提示补 Skill）；V2-A.2 再开生成草稿能力。

### 5.5 验收指标（本子块）
| 指标 | 定义 | 门槛 |
|---|---|---|
| 编配覆盖率 | 节点能力被（检索或拼装）满足的比例 | ≥ 90% |
| 生成 Agent 采纳率 | 自动拼装的 Agent 产出经 eval 通过 | ≥ 75%（≈ V1 路由命中含义） |

## 6. 信任与评测（Evaluator 升级为核心）

### 6.1 Evaluator 从"验收工具"→"自动背书发动机"
现有 `observability/evaluator.score(gateway, model, output, expected_fields, acceptance_criteria) → {passed, assertions_ok, judge_score}`（断言 + LLM-judge）。V2 升级：
- **常态化三件套**：断言（结构/字段）+ LLM-judge（对齐验收标准）+ 回归集（防退化）。
- **热路径放行**：任务产出 `judge_score ≥ τ_pass` 才进入"沉淀/交付"；低于则触发重规划或标记需人审。
- **防糊弄**：judge prompt 固定、给评分维度与反例；必要时双 judge 取低分；回归集兜底。
- 阈值/容错策略**先用默认、有问题再迭代**（用户决策）。

### 6.2 沙箱（Skill 生成的安全闸）
- 生成的 Skill（`kind='manual'` 提示词 / `kind='tool'` MCP 工具）**入库前必在沙箱试跑**，`skill_version.permissions` 做最小权限；危险操作走 04 的 gate。
- 沙箱 + 人审是 Skill 进 `published/verified` 的唯一通道。

### 6.3 三条信任路径（汇总，见 §2.3 表）
热路径自动 eval / Skill 入库人审 / 模板用户满意度——本块只实现"热路径 eval + Skill 人审入口"，模板用户满意度入口在 #6 任务记录/#7 结果展示落 UI。

## 7. 端到端流程（序列）
```
用户 goal
  │ POST /api/plans
  ▼
规划器：①语义检索模板 ──命中──▶ 填模板
        └──未命中──▶ ②RAG 生成 DAG ──▶ ③validate(+自修复)
  ▼
编配器：逐节点 ①select_agent ──无──▶ ②拼 Skill 成 Agent ──缺Skill──▶ 生成草稿→★人审★
  ▼
PlanResult（DAG + routing）──▶ 用户审批（approve gate，已有）
  ▼
Temporal 编排执行（节点=确定性 Skill；记忆黑板见记忆文档）
  ▼
Evaluator 自动背书（judge≥τ 放行）
  ▼
沉淀：用户满意→存场景模板（embedding 回写）；新 Skill 审核→入库；fact→记忆晋升
```

## 9. 里程碑拆解
| 里程碑 | 内容 | 依赖 |

## 13. 核心交互时序流程

### 13.1 出图时序（检索命中 / 生成兜底 / 有界自修复）
**说明**：用户提交一个目标到系统给出可运行 DAG 的**主控流程**。它体现退化链——先语义检索可复用模板（命中即用，省且稳），未命中才让 LLM 生成，且**生成结果必过确定性校验、失败则有界自修复**。输入=目标文本，输出=经校验的 `PlanResult`（DAG + 路由）。触发时机：`POST /api/plans`。
```
User    PlanAPI        Planner          Repo(pgvector)   LLM(LiteLLM)   Validator
 │  POST   │   plan(goal)  │                  │              │            │
 │ /plans  │──────────────▶│                  │              │            │
 │         │               │ available_caps() │              │            │   ① 取 active 能力集
 │         │               │─────────────────▶│              │            │
 │         │               │ embed(goal)      │              │            │
 │         │               │────────────────────────────────▶│           │   ② goal 向量化
 │         │               │ topk_templates(vec, caps⊆, vis) │           │
 │         │               │─────────────────▶│              │            │   ③ 语义检索模板
 │         │               │   templates+sim  │              │            │
 │         │               │◀─────────────────│              │            │
 │         │      ┌── alt [sim ≥ τ_tpl 命中] ─────────────────────────────┐
 │         │      │        │ fill_template(goal)             │            │
 │         │      └── alt [未命中] ────────────────────────────────────────┤
 │         │      │        │ generate_dag(goal, exemplars=topk, caps)      │
 │         │      │        │────────────────────────────────▶│ DAG JSON   │   ④ RAG 接地生成
 │         │      │        │◀────────────────────────────────│            │
 │         │      │   loop [≤ N 次自修复]                                   │
 │         │      │        │ validate(dag, active_caps) ─────────────────▶│  ⑤ 结构+语义双校验
 │         │      │        │   ok / errors                  │             │
 │         │      │        │◀───────────────────────────────────────────│
 │         │      │        │ [errors] prompt += 反馈 → 重新 generate      │
 │         │      └────────────────────────────────────────────────────────┘
 │         │               │ route_or_compose(dag)  (见 13.2)             │
 │         │               │ persist(plan, routing)                       │
 │         │   PlanResult  │                                              │
 │◀────────│◀──────────────│                                              │
```
关键判定：`τ_tpl`（模板命中相似度阈值）；自修复上限 `N`；校验失败仍不过 → `PlanInvalid(422)`；无 active 能力可满足 → `NoTemplateMatch`/触发缺能力链(13.2)。

### 13.2 编配与"缺技能生成"时序（含人审停点）
**说明**：把 DAG 的每个节点**匹配/组建出能真正执行它的 Agent** 的过程。体现"先用现有 Agent → 不足则拼装现有 Skill（自动 + eval 背书）→ 缺 Skill 才生成草稿并**停下等人审**"。输入=DAG 各节点的能力需求，输出=`node→agent` 路由表（必要时新建 generated Agent / 新 Skill）。它决定了自动化链能跑多深、人审卡在哪。
```
Planner   Composer        SkillRepo        Evaluator     Sandbox    Reviewer(人)
  │ route_or_compose(dag) │                 │             │           │
  │ for each agent_node:  │                 │             │           │
  │   select_agent(caps)  │                 │             │           │
  │──────────────────────▶│ (现有 Agent 能力⊇caps?)       │           │
  │  ┌ alt [命中] route[node]=agent          │            │           │
  │  └ alt [未命中] compose_agent(caps)       │           │           │
  │     │ for cap in caps: retrieve_skill(cap, published, vis)        │
  │     │────────────────▶│                  │            │           │
  │     ├ alt [全部命中] 拼 AgentConfig(skills,caps) → create(source=generated)
  │     │     │ eval(agent 试产出) ──────────▶│ judge      │           │   配置类→自动背书
  │     │     │◀───────────────────────────── │            │           │
  │     │     │ [judge≥τ] activate; route[node]=agent       │           │
  │     └ alt [缺 cap 无 skill] generate_skill_draft(cap) ──▶│ 沙箱试跑 │   ★人审停点★
  │           │ skill(draft,private) → sandbox ───────────▶│            │
  │           │ ────────────────── 提交审核 ───────────────────────────▶│ 代码/安全审
  │           │◀────────── publish(verified) + capability=active ───────│
  │           │ 回到 compose_agent（此时已可拼装）          │            │
```
规则：**配置类（角色/Agent 拼装）走自动 eval 背书、无人卡；唯有"造新 Skill"撞人审墙**（停点由模型长出，非人为设）。

### 13.3 信任背书与沉淀时序（热路径自动 / 资产入库受控）
**说明**：产出**如何被信任、又如何沉淀成可复用资产**。运行时（热路径）靠自动评测放行/触发纠错，**无人卡**；而往仓库沉淀时分级把关——新 Skill 走人审、场景模板由用户满意度决定、记忆事实按阈值自动晋升。它落实"人离开每个任务热路径、只在积累新资产时回来"的信任分层。
```
协同框架     Evaluator       仓库(Repo)        用户
  │ node 产出 │                │               │
  │──────────▶│ score(断言+judge+回归集)        │
  │  judge    │                │               │
  │◀──────────│                │               │
  │ [≥τ] 放行 → 继续/完成       │               │
  │ [<τ] → 触发纠错(协同框架 §4) │               │
  │ ── 任务完成、用户满意 ───────────────────────▶│ "存为场景模板"
  │           │   save_template(plan→embedding 回填, domain/subcat)    │  用户满意度晋升
  │ ── 新 Skill 审核通过 ───────▶│ publish(verified) + cap active      │  人审入库
  │ ── fact 达阈值 ────────────▶│ 记忆晋升(见记忆文档)                  │  自动晋升
```

## 14. 核心实现规则（算法 · 数据结构 · 校验 · 阈值）

### 14.1 关键数据结构
```python
# 计划（沿用 V1 PlanDag/PlanNode）
PlanNode = { id, type, deps[], required_capabilities[], input_hint?, expected_output?, dangerous }
PlanDag  = { workflow_name, goal, acceptance_criteria?, budget_cents, nodes[] }

# 能力注册项（capability 表演进）
Capability = { key(PK), domain, name, description, embedding[1024],
               status: proposed|active|deprecated, visibility: public|private, owner_org_id? }

# 技能（skill/skill_version 演进）—— 能力的实现与背书源
Skill        = { id, name, kind: manual|tool, capability(key), trust, status, visibility, owner_org_id, embedding[1024] }
SkillVersion = { skill_id, version, content?, mcp_server?, tool?, io_schema, permissions }

# Agent 配置（能力派生：capabilities = ∪ skills.capability）
AgentConfig  = { prompt, skills[], capabilities[](派生), executor, model?, authority, provenance?, eval? }
```

### 14.2 算法：模板语义检索 `retrieve_template`
**说明**：给定目标，从场景模板库**语义检索出最匹配且能力可满足的模板**。命中则直接复用（免去 LLM 生成，省成本、结果稳定），是"检索优先"退化链的第一步实现。相似度低于阈值即判未命中、转生成。
```
输入: goal, available(active 能力集)
1. vec ← embed(goal)
2. cands ← plan_template WHERE visibility∈{public, current_org}
            ORDER BY cosine(embedding, vec) DESC LIMIT K          # pgvector hnsw
3. cands ← filter(cands, needs(tpl) ⊆ available)                 # 能力可满足
4. score(tpl) = w1·cosine + w2·success_rate(tpl)                 # 历史成功率加权
5. best ← argmax score; 若 cosine(best) ≥ τ_tpl → 命中 best
6. 否则 → 未命中（进生成）
规则: K=10; τ_tpl≈0.78(初值,按数据校准); success_rate 来自 run_manifest 统计, 冷启动取 0.5
```

### 14.3 算法：RAG 接地生成 + 双校验 + 有界自修复 `generate_dag`
**说明**：模板未命中时**让 LLM 按"最像的几个模板"为范例生成 DAG**（RAG 接地，比凭空造更稳更省），并施加**结构校验(Pydantic schema) + 语义校验(能力闭合/无环/预算)** 双重确定性把关；校验失败时把具体错误反馈回 LLM 驱动**有界自修复**。这是突破"固定模板表达局限"的核心——既要 LLM 的灵活，又要确定性的可执行保证。
```
输入: goal, available, exemplars=topk_templates(goal, k=3)
1. sys ← 规划师角色 + 硬规则{只用 active 能力 / 必须无环 / 危险动作必须 human 节点 / 节点数≤MAX_NODES}
   usr ← goal + 能力词表(active) + exemplars.skeleton(做范例)
2. for attempt in 1..N:                                          # N=2
     raw ← LLM.chat(model, [sys, usr])
     dag ← try PlanDag.model_validate(parse_json(raw))           # ① 结构校验(Pydantic)
     if parse/schema 失败: usr += 结构错误反馈; continue
     vr ← validate(dag, available)                               # ② 语义校验(复用 V1)
     if vr.ok and estimate_cost(dag) ≤ budget and len(nodes) ≤ MAX_NODES: return dag
     usr += vr.errors(具体到节点/能力)                            # ③ 错误反馈驱动自修复
3. raise PlanInvalid(最后一次 errors)
规则: MAX_NODES≈12; validate 规则=无环/依赖存在/能力⊆active/预算内/危险必 human(沿用并强化)
```

### 14.4 算法：能力背书与激活 `activate_capability`
**说明**：当生成需要"新能力"时，**先语义去重（避免 `report.gen`/`report.make` 同义爆炸），确实是新的才登记，且必须有"被审过的 Skill + 评测"背书才置为 `active`**。规划器只允许使用 active 能力——这样能力词表可开放扩展，但"可信能力"始终有被审技能撑着（背书链）。
```
输入: cap_proposal{key?, name, description}
1. vec ← embed(name + description)
2. near ← capability ORDER BY cosine(embedding,vec) DESC LIMIT 1
3. if cosine(near) ≥ τ_dedup: return near.key                   # 语义去重，复用已有
4. 否则 新建 capability(status='proposed', embedding=vec)
5. 要 active 必须: ∃ skill(capability=key, status='published', trust≥'verified')  # 技能背书
                   ∧ 该 skill 过 eval/沙箱 ∧ 过人审
6. 满足 → status='active'（规划器只用 active）
规则: τ_dedup≈0.86; 新代码(skill)的人审是能力可信的唯一来源(背书链)
```

### 14.5 算法：编配 `compose_agent`（拼装优先 / 生成停点）
**说明**：某节点没有现成 Agent 时，**如何组建一个能干这活的 Agent**：优先用已审过的 Skill 拼装（不产生新代码 → 自动 + eval 背书、免人审），只有"某能力连 Skill 都没有"时才生成新 Skill 草稿并**停下走人审**。它精确定义了"自动化跑到哪、人审卡在哪"——卡点是模型本身长出来的。
```
输入: caps(节点所需能力)
1. skills ← [retrieve_skill(c) for c in caps]                    # published ∧ visibility∈{public,org}
2. if all(skills):                                               # 全可拼装 → 自动
     cfg ← AgentConfig(prompt=岗位说明(caps), skills, capabilities=caps, executor='lite-agent', source='generated')
     agent ← create_agent(cfg); ev ← eval(agent 试产出)
     if ev.judge ≥ τ: activate(agent); return agent             # 配置类自动背书
     else: 调整 prompt 重试(≤1) / 标记 needs_human
3. else (有 cap 无 skill):                                       # 撞人审墙
     generate_skill_draft(missing_cap) → sandbox → ★人审★ → publish → activate_capability → 回到 1
规则: retrieve_skill 同能力多候选时按 trust 降序 + 历史成功率; 拼装不产生新代码故免人审
```

### 14.6 持久化 / 并发 / 错误处理规则
- **持久化**：plan/routing 落库（V1 已有）；生成的 agent/角色模板带 `provenance`(出处) + `eval`(评分快照)。
- **并发**：同 org 同时出图无副作用（纯读检索 + 写各自 plan 行，org_id+RLS 隔离）。
- **错误处理矩阵**：
  | 失败点 | 处理 |
  |---|---|
  | 无 active 能力满足任一模板 | `NoTemplateMatch` → 404（提示补 Skill/换目标） |
  | 生成 DAG 结构非法 | 有界自修复 ≤N → 仍败 `PlanInvalid` → 422 |
  | 节点缺 Skill | 生成草稿 → 人审（任务转 needs_human，不阻断其它节点） |
  | LLM 不可达/超时 | 重试 + 降级到"仅检索命中"路径（无生成） |
- **阈值汇总（初值，按数据校准）**：`τ_tpl≈0.78` · `τ_dedup≈0.86` · `judge τ≈0.6` · `N=2` · `K=10` · `MAX_NODES≈12`。
