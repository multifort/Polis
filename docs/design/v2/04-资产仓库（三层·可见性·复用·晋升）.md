## 0. 一句话
资产仓库是"虚拟公司越用越聪明"的底座——**Skill / 角色模板 / 场景模板**三层可复用资产，按租户分**公共/私有**（共享后置），是内核**检索的语料**与**沉淀的落点**。

## 1. 背景与目标

### 1.1 V1 现状
- `skill` + `skill_version`：底层已具 `trust`(official/verified/community/private) + `status`(draft/published/deprecated) + `capability` + `owner_org_id`。
- `capability`：受控词表 + embedding（横切）。
- `plan_template`：DAG 骨架（场景模板的前身）。
- `scenario_preset`：开办公司的预设（含 `config.agentTemplates` + `defaultPlanTemplates`）。

### 1.2 目标
1. 形式化**三层可复用资产** + 统一**可见性**（公共/私有，共享预留）。
2. 让资产**按引用复用、可版本钉选、可组合**。
3. 成为内核"检索→生成→沉淀"飞轮的**持久底座**（自增长）。

### 1.3 关键澄清（用户 2026-06-22）
- **场景模板 ≠ 公司蓝图**：场景模板是**可复用的业务场景任务模式**，**一个公司可有多个**（采购公司可同时有"交付分析/支出节降/合同审阅"多个场景模板）。
- **公司蓝图(`scenario_preset`)是正交概念**：开办公司的"启动包"，**打包引用**仓库里的角色模板 + 场景模板来实例化新公司——不是仓库顶层本身。

### 1.4 非目标（后置）
- **共享(shared / 点对点 / Marketplace)**：V2 不实现，仅留 schema 位（V3 治理）。
- 计费 / 评分 / 撤销等 Marketplace 治理（V3）。

## 2. 三层模型 + 与公司蓝图的关系

```
仓库三层（可复用资产）                       正交：公司蓝图（开办启动包）
┌───────────────────────────────┐
│ 上：场景模板（业务场景任务模式）│◀──┐       scenario_preset
│   plan_template 演进           │   │ 引用 ├─ 引用一组角色模板
│   一个公司可有多个             │   │      └─ 引用一组场景模板
├───────────────────────────────┤   │            │ provision 实例化
│ 中：角色模板（岗位定义）       │◀──┘            ▼
│   role_template（新增）        │           新公司（花名册 + 启动场景模板）
├───────────────────────────────┤
│ 底：Skill（原子·固定·受审）    │
│   skill + skill_version        │
└───────────────────────────────┘
        横切：capability（能力契约 + embedding）
```

| 层 | 定位 | 落表 | 复用方式 |
|---|---|---|---|
| 上 · 场景模板 | 业务场景任务模式（goal 类型→DAG + 所需角色），**一公司多个** | `plan_template`（演进） | 内核检索命中即用 / 沉淀回写 |
| 中 · 角色模板 | 岗位定义（人设 + 技能组合 + 派生能力） | **`role_template`（新增）** | 编配器选/拼 Agent 的来源 |
| 底 · Skill | 原子、确定执行、受审 | `skill`/`skill_version` | 角色模板/编配器引用 |
| 横切 · 能力 | 接口契约 | `capability` | 路由/校验/检索 |

## 4. 组合与复用

### 4.1 引用而非拷贝 + 版本钉选
```
场景模板.role_refs ─▶ role_template(id,version)
role_template.skill_refs ─▶ skill(id,version)
skill.capability ─▶ capability.key
```
- **按引用**：上层存下层的 `id + version`，下层升级可受控传播。
- **版本钉选**：org 引用某资产时钉版本；上游改版**不破坏在用方**（主动升级，不强推）。`skill_version` 已有；`role_template`/`plan_template` 加 `version`。

### 4.2 可组合性信任约束（防越权信任）
**上层可信度 ≤ 其依赖的下层最低可信度**：
- public 场景模板 **只能**由 public/verified 的角色模板与 Skill 组成；
- 不允许 public 资产引用某 org 的 private Skill（否则隔离被打穿）。
- 校验落在"发布/晋升"时（确定性检查依赖的 visibility/trust）。

## 5. 可见性模型（public / private；shared 预留）

| 可见性 | 含义 | 落点 | 谁可读 |
|---|---|---|---|
| **public** | 平台预置/官方，所有 org 可用 | `owner_org_id=NULL` + `visibility='public'` | 全部 org |
| **private** | 仅属主 org | `owner_org_id=X` + `visibility='private'` | 属主 org |
| **shared（预留，不实现）** | 属主显式授权他 org | 私有行 + `asset_grant`（V3） | 属主 ∪ 被授权 |

### 5.1 RLS 演进（动多租户根基，单列 ADR）
V1 RLS = "行带 org_id、按当前 org 过滤"（纯私有干净）。引入 public 后需演进：
- **读策略放宽**：`当前 org 私有 ∪ public`（shared 后续再加"∪ 授权给我的"）。
- **写策略不变（严格）**：只能写自己 org 的私有资产；public 资产由平台/seed 维护，org 不可改。
- **隔离不打穿**：放宽只在**读**、且只放行 public（无 org 归属）；private 仍严格按 org。
- 实现：调整相关表 RLS policy 为 `org_id = current_org OR visibility='public'`；写策略仍 `org_id = current_org`。

### 5.2 V2 范围
- **V2 做 public + private**；`visibility` 枚举含 `shared` 但不实现，`asset_grant` 表不建（留 V3）。

## 6. 晋升 / 信任（接前序决策）

| 层 | 入私有库 | 升 public（V3 平台 curation） |
|---|---|---|
| **Skill** | **必须人审**（代码/安全/沙箱）：draft→published、private→verified | 平台审 → official |
| **角色模板** | **纯自动 + eval**（用户决策；由已审 Skill 组合，数据级低风险，无人 curate） | 平台审 |
| **场景模板** | **用户满意度**（用户对结果满意 → 主动"存为模板"） | 平台审 |

- 三条与"信任分层"一致：唯一的"新代码"(Skill)必审；角色/模板是配置/结构，自动或用户驱动。
- 角色模板纯自动：编配器拼出的 Agent 经 eval 通过 → 抽象成 `role_template`（source='generated'）入私有库，下次可检索复用。

## 7. 与内核的接口（仓库 = 检索语料 + 沉淀落点）
| 内核步骤 | 对仓库的操作 |
|---|---|
| ①检索 | 语义检索 场景模板/角色模板/Skill/能力（embedding），**可见集 = 私有 ∪ public** |

## 10. 关键决策（建议落 ADR-0013）
- 三层资产：Skill / 角色模板 / 场景模板；**场景模板一公司可多个**；公司蓝图(preset)是正交的打包概念。
- 可见性：**V2 做 public+private，shared 预留不实现**。
- 晋升：Skill 必审 / 角色模板**纯自动+eval** / 场景模板用户满意度。
- **RLS 演进**：读放行 public，写仍严格按 org（隔离不打穿）。

## 11. 未决问题
1. `role_template` 与 V1 `role`/`agent` 的关系：role_template 是"定义"，provision/编配实例化为 org 内 `agent`——实例与模板的双向同步策略（模板升级是否提示实例升级）。
2. public 资产的属主/维护方（平台系统 org vs `owner_org_id=NULL`）与 seed 的衔接。
3. 能力 `visibility=org` 私有能力命名空间防冲突（与内核 §11.1 同源）。
4. 版本钉选的升级提示 UX（在用方如何感知上游有新版）。

## 12. 验收指标（本块）
| 指标 | 定义 | 门槛 |
|---|---|---|

## 13. 核心交互时序流程

### 13.1 资产检索时序（可见集 = 私有 ∪ public）
**说明**：内核"检索"步**如何在仓库里找可复用资产**（场景模板/角色模板/Skill）。关键是**可见性感知**——一个 org 只能看到"自己私有 + 平台 public"，由 RLS 读策略放行 public、绝不泄露他 org 私有。这是飞轮"检索优先"的数据来源，也是隔离不打穿的保证。
```
内核(检索)        仓库Repo            RLS(读策略)        pgvector
  │ retrieve_asset(kind, query)        │                │
  │────────────────▶│                  │                │
  │   embed(query)  │                  │                │
  │   SELECT ... WHERE 可见性过滤 ─────▶│ org_id=cur OR visibility='public'  ① 读放行public
  │                 │  ORDER BY cosine(embedding) ──────▶│   ② 语义召回
  │                 │◀──────────────────────────────────│
  │   按 trust·成功率·相似 重排 top-k   │                │
  │◀────────────────│                  │                │
```

### 13.2 资产沉淀时序（三路径分级把关）
**说明**：满意的产出/生成物**如何沉淀回仓库变成"下次可检索"**（自增长）。三条路径分级——场景模板由**用户满意度**触发、角色模板**自动+eval**入库、新 Skill **必经人审**。沉淀时回填 embedding 供后续语义检索。
```
用户/内核        仓库Repo          Evaluator      Reviewer(人)
  │ ── 用户满意"存为模板" ──▶│ save plan→plan_template(私有, domain/subcat, embedding回填)  用户满意度
  │ ── 生成 Agent 经 eval ──▶│ eval≥τ → role_template(source=generated, 私有, embedding)    自动
  │                          │◀── score ── Evaluator  
  │ ── 新 Skill 草稿 ────────│ sandbox → 提交审核 ───────────▶│ 代码/安全审  ★必人审★
  │                          │◀── publish(verified) + capability=active ──────────────────│
```

### 13.3 版本钉选 + 升级提示时序
**说明**：上层资产**按引用+钉版本**复用下层，下层升级**不强推、只提示**在用方。保证升级不破坏在用的任务/公司。
```
仓库          某 Skill            在用方(角色模板/场景模板)
  │ skill 发布新版 v2  │            │
  │───────────────────▶│           │
  │  在用方仍钉 v1(引用 id+version 不变, 不破坏)│   ① 钉版本=稳定
  │  生成"有新版"提示 ─────────────▶│ owner 决定是否升级    ② 主动升级,不强推
```

## 14. 核心实现规则（算法 · 数据结构 · 可见性 · 阈值）

### 14.1 关键数据结构
```python
RoleTemplate = { id, name, persona(prompt), skill_refs:[{skill_id,version}], capabilities(派生),
                 visibility: public|private, owner_org_id?, version, embedding[1024], status, source }
# Skill/SkillVersion/Capability/PlanTemplate 见内核文档 §14.1 + 本文 §3 增量
PlanTemplate(场景模板) += { embedding[1024], visibility, owner_org_id?, role_refs[], source, domain(大类), subcategory(小类) }
AssetGrant = { asset_kind, asset_id, grantee_org_id, ... }   # 预留, V2 不实现(shared)
```

### 14.2 算法：可见性感知检索 `retrieve_asset`
**说明**：在 RLS 读放行 public 的前提下做语义检索，是内核检索步的仓库侧实现。
```
输入: kind(skill|role_template|plan_template), query, k
1. vec ← embed(query)
2. rows ← kind WHERE (org_id=current_org OR visibility='public') ∧ status='published'   # RLS 读策略
            ORDER BY cosine(embedding, vec) DESC LIMIT k*α
3. rank = w1·cosine + w2·trust_weight + w3·success_rate
4. return top-k(钉可用 version)
规则: trust_weight: official>verified>community>private; 写仍严格 org_id=current_org(不放宽)
```

### 14.3 算法：资产晋升 `promote_asset`（三路径）
**说明**：三层资产各自的入库规则，统一"配置自动、代码人审、模板看满意度"。
```
场景模板: on 用户"存为模板": new plan_template(source='generated'|'curated', visibility='private',
          domain/subcategory 必填, embedding=embed(goal+skeleton))        # 用户满意度
角色模板: on 生成 Agent eval.judge≥τ: new role_template(source='generated', visibility='private',
          skill_refs=已审skills, embedding)                                # 自动+eval
Skill   : draft → sandbox → 人审 → publish(status='published', trust='verified') → activate_capability  # 必人审
规则: 私有入库无需人审(模板/角色); 仅 Skill(新代码) 必人审; 升 public 属 V3 平台 curation
```

### 14.4 算法：可组合信任校验 `composability_check`
**说明**：发布/晋升时确定性校验"上层不越权信任下层"，防隔离被打穿。
```
输入: asset(含 refs)
1. dep_trust = min(trust/visibility of all 直接依赖)        # skill_refs / role_refs
2. assert asset.visibility 可信度 ≤ dep_trust:
     public 资产 ⇒ 所有依赖必须 public ∧ trust≥verified
     private 资产 ⇒ 依赖可为 自身org private ∪ public
3. 违反 → 拒绝发布(明确报哪条依赖越权)
```

### 14.5 RLS 演进 / 版本钉选 / 阈值
**说明**：多租户根基的读放行规则、版本钉选规则与阈值汇总。
```
RLS: 读 policy = (org_id = current_org OR visibility='public'); 写 policy = org_id = current_org(不变)
     public 资产 owner_org_id=NULL, 由平台/seed 维护; 隔离回归接 M7-T8.3
版本钉选: 引用存 {id, version}; 上游发新版不改在用引用; 仅生成"可升级"提示
阈值: 检索 k=10, α=3; role 自动入库 judge τ≈0.6; 检索去重 τ_dedup≈0.86(与能力词表一致)
错误处理: 引用的 version 已 deprecated → 提示并建议升级, 不自动改; public 资产 org 侧只读(写报 403)
```
