# 09 身份 · 账号 · 多租户 — 详细设计

> **本篇目标**：补齐"共享后台 + 数据逻辑隔离 + 账号体系 + Web-only"的底座——
> app_user/认证、用户↔多 Org 成员与切换、权限矩阵、**应用层过滤 + PostgreSQL RLS 双保险**、API + WBS。
> 依赖：PostgreSQL、`02`(org/org_member)、`05`(org 记忆)、`06`(审批权限)。
> 定位：Polis 是**单一共享后台**服务所有城邦(Org)，数据**逻辑隔离**（同库 `org_id` 行级 + RLS 兜底）。
> 决策见 [ADR-0005](../decisions/0005-multi-tenancy-strategy.md)。

---

## 1. 使用模式（与产品形态对齐）

```text
浏览器(Web, 零安装) ──REST/WS──> 单一 Polis 后台(共享) ──> 一个 PostgreSQL
                                                          └─ 所有 Org 数据同库, org_id 行级隔离 + RLS
账号体系: 一个 app_user 可属于多个 Org(org_member); 登录后选择/切换"当前城邦"
每个 Org: 独立记忆命名空间(05)、独立预算(02 budget_cents)、独立花名册与计划
```
- **共享后台**：单进程 modular monolith（07 §2），无每租户独立进程。
- **零本地 client**：工具走服务端 MCP，用户只用浏览器（区别于 hermes/openclaw 的本地 client）。

## 2. 数据模型（补 app_user，对接既有 org/org_member）

```sql
CREATE TABLE app_user (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email         CITEXT NOT NULL UNIQUE,           -- 大小写不敏感唯一
  password_hash TEXT,                              -- argon2id; 第三方登录可空
  display_name  TEXT,
  status        TEXT NOT NULL DEFAULT 'active',    -- active|disabled
  last_login_at TIMESTAMPTZ,
  created_at    TIMESTAMPTZ DEFAULT now()
);

-- org_member 已在 02 定义：(org_id, user_id, role)，role ∈ owner|approver|member
-- 一个 user 多行 org_member = 属于多个城邦

CREATE TABLE auth_session (            -- refresh 令牌(可吊销), access 用短 JWT 无状态
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      UUID NOT NULL REFERENCES app_user(id) ON DELETE CASCADE,
  refresh_hash TEXT NOT NULL,          -- 存哈希, 不存明文
  user_agent   TEXT, ip INET,
  expires_at   TIMESTAMPTZ NOT NULL,
  revoked_at   TIMESTAMPTZ,
  created_at   TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE org_invite (              -- 成员邀请
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id    UUID NOT NULL REFERENCES org(id) ON DELETE CASCADE,
  email     CITEXT NOT NULL,
  role      TEXT NOT NULL,             -- approver|member
  token_hash TEXT NOT NULL,
  status    TEXT NOT NULL DEFAULT 'pending',  -- pending|accepted|revoked|expired
  invited_by UUID REFERENCES app_user(id),
  expires_at TIMESTAMPTZ, created_at TIMESTAMPTZ DEFAULT now()
);
```
> `app_user`、`auth_session`、`org_invite` 是**平台级表（无 org_id）**；其余一切业务表带 `org_id`（12 C11）。

## 3. 认证（JWT 短 access + 可吊销 refresh）

```text
注册/登录 → 颁发: access JWT(15min, 无状态) + refresh(7~30d, 存 auth_session 哈希, 可吊销)
access JWT claims: { sub=user_id, exp }   ← 不绑定 org, 不放权限(避免越权/换租户绕过)
刷新: refresh 有效且未吊销 → 换新 access; 登出/改密/异常 → 吊销 refresh
密码: argon2id 哈希; 找回走邮件一次性令牌; V1 邮箱+密码, OAuth/SSO 后置
```
- access JWT **不携带 org/权限**：当前城邦与权限**每请求服务端按成员关系实时判定**，
  防止"令牌里旧权限"导致的越权或切租户绕过。

## 4. ★ 当前城邦上下文与切换

```text
请求头: Authorization: Bearer <access>   +   X-Org-Id: <org_id>
中间件(每请求):
  1) 验 access JWT → user_id
  2) 查 org_member(org_id, user_id) → 不存在则 403（你不属于这个城邦）
  3) 取该 user 在该 org 的 role
  4) SET LOCAL app.current_org = org_id   ← 注入到本事务，供 RLS 使用(§5)
切换城邦 = 前端改 X-Org-Id（须是 GET /api/me.orgs 里的一项）；无状态、无需重登
```
- `GET /api/me` 返回 user + 其所属 Org 列表（含各自 role）→ 前端做城邦切换器。

## 5. ★ 双保险隔离：应用层过滤 + PostgreSQL RLS

**纵深防御**（来自 ADR-0005）：不只靠应用层，DB 层再兜一道，漏写一个 `WHERE org_id` 也不串数据。

```sql
-- 对每张带 org_id 的业务表启用 RLS（迁移里统一生成）
ALTER TABLE memory ENABLE ROW LEVEL SECURITY;
CREATE POLICY org_isolation ON memory
  USING (org_id = current_setting('app.current_org', true)::uuid);
-- result_envelope / plan / agent / approval / ... 同样套用
```
```python
# 应用层：repo 基类强制 org 过滤；同时每请求在事务里设 session 变量供 RLS
async with db.begin():
    await db.execute(text("SET LOCAL app.current_org = :org"), {"org": org_id})
    ...  # 此事务内所有查询自动受 RLS 约束
```
- 应用层（repo 统一注入 `org_id`）= 第一道；**RLS = 第二道兜底**。
- 平台级表(app_user/auth_session/org_invite)不启用 RLS（它们本就跨租户）。
- 后台运维/迁移用单独的 `BYPASSRLS` 角色，业务连接用普通角色。

## 6. 权限矩阵（owner / approver / member）

| 动作 | owner | approver | member |
|---|:--:|:--:|:--:|
| 对话 / 建任务 / 看看板 | ✓ | ✓ | ✓ |
| 审批：计划/危险动作/编配/返工(06) | ✓ | ✓ | ✗ |
| 管理 Agent/技能/预算 | ✓ | ✗ | ✗ |
| 邀请/移除成员、改成员角色 | ✓ | ✗ | ✗ |
| 连接模型凭证(BYO Key, 06) | ✓ | ✗ | ✗ |
| 归档/删除城邦 | ✓ | ✗ | ✗ |
- 服务端用 FastAPI 依赖注入做路由守卫：`require_role("approver")` 等；危险动作 gate(04/06) 与此一致。

## 7. API

```http
POST /api/auth/register            # 邮箱+密码 → 建 app_user
POST /api/auth/login               # → access + refresh
POST /api/auth/refresh             # refresh → 新 access
POST /api/auth/logout              # 吊销 refresh
POST /api/auth/password/reset      # 邮件一次性令牌
GET  /api/me                       # user + 所属 Org 列表(含 role)
POST /api/orgs                     # 建城邦(创建者=owner)  ── 对接 02 provision
GET  /api/orgs/{id}/members        # 成员列表(owner/approver)
POST /api/orgs/{id}/invites        # 邀请(owner)
POST /api/invites/{token}/accept   # 接受邀请 → 建 org_member
PATCH /api/orgs/{id}/members/{user_id}   # 改成员角色(owner)
DELETE /api/orgs/{id}/members/{user_id}  # 移除成员(owner)
```
- 所有 `/api/orgs/{id}/*` 与业务接口走 §4 中间件（验成员 + 设 RLS 上下文）。

## 8. 边界与错误
- 不属于该 Org → 403，不泄露资源是否存在。
- refresh 被吊销/过期 → 401，前端引导重登。
- 最后一个 owner 不能被移除/降级（防止城邦无主）。
- 邀请邮箱已是成员 → 幂等返回，不重复建。

## 9. 与既有设计的衔接
- `02` org/org_member/provision：建 Org 时创建者落 `org_member(owner)`；本篇补 app_user 与成员/邀请。
- `05` 记忆：检索/写入的 `org_id` 即来自 §4 的当前城邦上下文；RLS 再兜底。
- `06` 审批：approver/owner 才能 decide；与权限矩阵一致。
- `07` 部署：仍单库单后台；RLS 时机从 V3 提前（见 §10 与路线图更新）。

## 10. 任务分解(WBS)
| 任务 | 交付 | 验收 | 里程碑 |
|---|---|---|---|
| T9.1 app_user/auth_session DDL + 注册/登录/JWT(access+refresh)+argon2 | 认证模块 | 登录拿到 token，刷新/登出/改密吊销生效 | M1→M2 |
| T9.2 当前城邦中间件 + 多 Org 切换(X-Org-Id 校验成员) | 中间件 + /api/me | 非成员 403；切 Org 无需重登 | M2 |
| T9.3 权限矩阵 + 路由守卫(require_role) | 依赖注入守卫 | member 无法审批/管成员，owner 可 | M2 |
| T9.4 PostgreSQL RLS 策略 + 每请求 set current_org | 迁移 + 事务注入 | 漏写 org 过滤仍被 RLS 拦；隔离回归(T8.3)全绿 | M1(基线)→贯穿 |
| T9.5 成员邀请流程(org_invite + 接受) | 邀请 API | 邀请→接受→成员生效；幂等/过期正确 | V2(或 M2 精简版) |
