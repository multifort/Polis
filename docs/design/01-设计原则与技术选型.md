# 01 设计原则与技术选型（开源复用优先 · 低运维）

> **解决的问题**：用哪些组件搭这套系统，才能"运维轻、上手快、初期不自研"。
> 原则：**能复用成熟开源就直接用，只自研差异化粘合逻辑。**

---

## 0. 后端技术栈（已定：Python / FastAPI）

| 层面 | 选型 | 说明 |
|---|---|---|

| 数据访问/迁移 | **SQLAlchemy 2.0 + Alembic** | ORM + 迁移；pgvector 经 `pgvector-python`/SQLAlchemy 类型 |

| 记忆/向量 | **PostgreSQL + pgvector** | (05) |
| 可观测/评估 | **Langfuse Python SDK** | (06) |

## 1. 选型总表（能力 → 复用组件 → 为什么 → 是否自研）

| 能力 | 复用的开源组件 | 为什么（降运维/上手） | 自研? |
|---|---|---|---|
| 工作流编排 | **Temporal** | 持久化/重试/人审暂停/断点续跑现成，自己写极易出 bug | ❌ 复用 |
| 模型网关/路由/BYO-Key/成本 | **LiteLLM** | 一个网关接 OpenAI/Claude/Gemini/DeepSeek/Qwen，自带 BYO-Key、用量与成本统计、统一 OpenAI 协议 | ❌ 复用 |
| 工具/技能执行 | **MCP + 现成 MCP Server** | 工具的事实标准；浏览器/文件/DB/搜索等直接接现成 server（含 NoDeskAI `browser-pilot`） | ❌ 复用 |
| 向量检索/记忆 | **PostgreSQL + pgvector** | **一个库搞定元数据+向量**，免去独立向量库(Milvus)的部署运维 | ❌ 复用 |
| 可观测/追踪/评估/成本 | **Langfuse** | Trace 全链路 + LLM-as-judge eval + 成本，开源自托管，省自建 | ❌ 复用 |
| 防护栏/注入检测 | **Guardrails-AI / NeMo Guardrails** | 输入输出校验、注入/PII 过滤现成 | ❌ 复用 |
| 对象存储(产物) | 本地卷 → **MinIO** | MVP 用文件系统，规模再上 MinIO(S3 兼容) | ❌ 复用 |
| 秘密管理 | 应用层信封加密 → **Vault** | MVP 用 KMS/库内信封加密(轻)，规模再上 Vault | 阶段复用 |
| Agent 框架(可选) | **LangGraph** 或极简循环 | 若需图式 agent 循环可复用 LangGraph；否则一个最小 tool-calling 循环即可 | 倾向复用 |
| 部署 | **docker-compose → K8s** | MVP 单机 compose，规模再 K8s | ❌ 复用 |

## 2. 我们**只**自研这几块（差异化粘合，不碰基础设施）

1. **角色编配器**（意图 → 组装预设/缺口生成 → 建 Org）——见 `02`。
2. **能力路由**（节点能力需求 → 选 Agent，含记忆亲和+冷启动）——见 `03`。
3. **组织/治理粘合**（Org 容器、最小权限、审批收件箱）——见 `02`/`06`。
4. **出处与结果信封纪律**（数据准确性）——见 `05`。
5. **计划校验 + 有界重规划**逻辑——见 `03`。

> 这些是别人没有、且构成差异化与潜在知识产权的部分（见 `08`）；其余全部站在开源之上。

## 3. MVP 的最小基础设施足迹（低运维的硬指标）

```text
只跑 4 样：
  PostgreSQL(+pgvector)   元数据 + 记忆向量 + 台账
  Temporal                编排
  LiteLLM                 模型网关
  Langfuse                可观测/评估
对象存储：先用本地卷
部署：docker-compose 单机
```
**明确不在 MVP 引入**：Kafka、Elasticsearch、Milvus、Redis(非必需)、12 微服务、K8s、OPA。
（这些都在 `07` 标为"规模到了再上"。）

## 4. 上手成本的控制

- **单体优先**：V1 一个服务（modular monolith），内部模块化，不拆微服务——少了服务发现/网关/分布式调试的全部成本。
- **统一协议**：模型走 LiteLLM 的 OpenAI 协议、工具走 MCP——开发者只学两套标准接口。
- **声明式资源**：Agent/Skill/Preset 都是声明式配置（YAML/JSON 入库），改行为不改代码。

## 5. 取舍说明（诚实）

- pgvector vs Milvus：规模极大(亿级向量)时 Milvus 更强；但 MVP/中等规模 pgvector 足够且**少一个组件**。等真撞到瓶颈再迁，接口不变。
- LiteLLM vs 自研 Model Router：LiteLLM 覆盖 80%（同步+SSE+成本+BYO）；异步视频/签名鉴权等长尾仍需薄适配（见 `06`）。
- Agent 框架：若团队已熟 LangGraph 直接用；否则"最小 tool-calling 循环 + MCP"上手更快、依赖更少——**二选一，别两个都引**。
